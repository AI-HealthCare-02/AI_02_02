## 0. 혼자 해도 되는 것 vs 팀과 맞춰야 하는 것

### 핵심 원칙
> 코드는 혼자 짜도 되지만, **"설계(구조)"는 팀과 맞춰야** 합니다.

### 혼자 해도 되는 것 ✅
| 작업 | 이유 |
|------|------|
| FINDRISC 점수 계산 코드 작성 | 공식이 정해져 있으므로 |
| AI Worker 내부 로직 구현 | ai_worker/ 폴더는 LLM 파트 전용 |
| LLM API 연결 코드 (Claude/GPT) | AI 파트만 사용 |
| 테스트 코드 작성 | 자기 코드 검증 |

### 팀과 맞춰야 하는 것 🤝
| 작업 | 이유 | 누구와? |
|------|------|---------|
| **DB 모델 구조** (테이블 열 이름, 데이터 타입) | 백엔드가 API 만들 때, 프론트가 화면 만들 때 다 이걸 기준으로 함 | 백엔드 + 프론트 |
| **API 주소와 요청/응답 형태** | 프론트엔드가 이 주소로 데이터를 보내고 받아야 함 | 프론트엔드 |
| **환경변수** (API 키, DB 비밀번호 등) | .env 파일을 모두가 공유함 | 전체 팀 |
| **Docker 서비스 설정** | ai-worker가 다른 서비스와 통신해야 함 | DevOps/백엔드 |

### 공유 자원 6가지 (모두가 함께 쓰는 것)

#### 1. User 모델 = 모든 것의 기반 (이미 완성됨 ✅)
```
비유: 도서관 회원카드
모든 서비스가 "이 사용자가 누구인지" 알려면 User를 참조해야 함.
건강기록도 User 소유, 채팅도 User 소유, 챌린지도 User 소유.
```
- 위치: `app/models/users.py`
- 이미 구현됨: id, email, name, gender, birthday 등
- **우리가 할 일**: 새 모델(daily_health_logs 등)에서 User를 "참조"만 하면 됨

#### 2. 인증 시스템 = 신분증 확인 (이미 완성됨 ✅)
```
비유: "로그인한 사람만 이용 가능"한 시스템
모든 건강 API에 "이 사람이 누군지" 확인하는 코드가 필요.
이미 만들어져 있으니 가져다 쓰면 됨.
```
- 사용법: API에 `Depends(get_request_user)` 한 줄 추가하면 자동으로 로그인 확인
- JWT 토큰: 로그인하면 "신분증"(토큰)을 발급, 매 요청마다 이걸 보여줌

#### 3. 데이터베이스(PostgreSQL) = 모든 데이터가 저장되는 곳
```
비유: 공용 냉장고
백엔드, AI Worker 모두 같은 냉장고(PostgreSQL)를 사용.
새 칸(테이블)을 만들면 모두가 접근 가능.
```
- 설정: `app/db/databases.py`의 TORTOISE_ORM
- **중요**: 새 모델 만들면 여기 `TORTOISE_APP_MODELS` 리스트에 등록해야 함

#### 4. Redis = 메시지 전달 시스템
```
비유: 주방과 홀 사이의 호출벨
FastAPI(주방)가 "위험도 계산해줘" 버튼을 누르면
AI Worker(전문셰프)가 벨 소리 듣고 처리.
```
- 현재: 설정만 되어있고 실제로 안 씀
- 우리가 할 일: FastAPI ↔ AI Worker 통신에 사용

#### 5. 환경변수(.env) = 공용 비밀번호 보관함
```
비유: 직원들만 아는 금고 비밀번호
DB 비밀번호, API 키 등이 여기에 저장됨.
코드에 직접 쓰면 안 되고, 여기서 읽어와야 함.
```
- 위치: 프로젝트 루트 `.env` (현재 26줄, DB/Redis/Docker 설정이 들어있음)
- 설정 읽는 코드: `app/core/config.py` (여기서 .env를 자동으로 읽어옴)
- **우리가 추가할 것**: LLM API 키. `.env` 파일 맨 아래에 추가하면 됨:
  ```
  # LLM (AI 파트에서 사용)
  OPENAI_API_KEY=sk-여기에-발급받은-키를-넣으세요
  # 또는 Claude를 쓴다면:
  # ANTHROPIC_API_KEY=sk-ant-여기에-발급받은-키를-넣으세요
  ```
- **추가 후 해야 할 일**: `app/core/config.py`에도 해당 변수를 등록해야 코드에서 쓸 수 있음 (Claude에게 "config.py에 API 키 변수 추가해줘" 라고 하면 됨)

#### 6. Docker 네트워크 = 건물 내부 통로
```
비유: 같은 건물 안에서 각 방(서비스)이 서로 통신하는 복도
fastapi, postgres, redis, ai-worker, nginx가 같은 네트워크에 있어서
서로 이름으로 접근 가능 (예: "postgres:5432"로 DB 접속)
```

| 파트 | 역할 | 작업 폴더 | 비유 |
|------|------|----------|------|
| **프론트엔드 (FE)** | 사용자가 보는 화면 만들기 | `apps/web/` | 식당 홀 인테리어 |
| **백엔드 (BE)** | 데이터 저장/처리/API | `app/` | 주방 운영 |

| **DevOps** | 배포 + 테스트 + 환경 관리 | Docker, scripts | 건물 관리 |

### 전체 프로젝트 진행 순서 (Phase 0~5)

```
Phase 0: 환경 설정 ✅ (이미 완료)
├── BE: User 모델, 인증 API ✅
├── FE: Next.js 셋업, CSS ✅
└── ALL: Docker, Git ✅

├── FE: 설문 화면 UI 만들기
├── BE: 설문 데이터 저장 API
├── AI(나): FINDRISC 점수 계산 로직 ⭐
└── 이 3개는 동시에 진행 가능! (DB 모델만 먼저 합의)

Phase 2: 건강 데이터 수집 + 대시보드
├── BE: 건강기록 저장/조회 API
├── FE: 대시보드 차트, 채팅 화면
├── AI(나): AI 채팅 + 건강질문 삽입 ⭐

Phase 3: 위험도 예측 모델
├── AI(나): ML 모델 학습 + 추론 ⭐⭐⭐
├── BE: 예측 결과 저장 API
└── Phase 2에서 건강데이터가 쌓여야 학습 가능

Phase 4: 생활습관 챌린지
├── BE: 챌린지 API

└── Phase 2-3 데이터 기반

Phase 5: 통합 테스트 + 배포
└── ALL: 모든 기능이 합쳐져서 돌아가는지 확인

```
⚠️ 이 순서를 어기면 만들어도 연결이 안 됩니다:

## 1. 먼저 알아야 할 개념들 (비유로 설명)

### 우리 프로젝트 = 식당 운영

프로젝트를 **식당**에 비유하면 이해가 쉽습니다:

| 개발 용어 | 식당 비유 | 우리 프로젝트에서는 |
|----------|----------|------------------|
| **프론트엔드** (`apps/web/`) | 식당 홀 (손님이 보는 메뉴판, 인테리어) | 사용자가 보는 화면 (채팅창, 대시보드) |
| **백엔드** (`app/`) | 주방 (요리사가 실제 음식을 만드는 곳) | 데이터 저장, 계산, 처리하는 곳 |
| **데이터베이스(DB)** | 냉장고 + 재료 창고 | 사용자 정보, 건강 기록 보관소 |
| **API** | 주문서 (홀→주방 전달 양식) | "이 데이터 저장해줘", "리포트 보여줘" 요청서 |
| **AI Worker** (`ai_worker/`) | 전문 셰프 (특수 요리만 담당) | 위험도 예측, AI 채팅 처리 전담 |
| **Docker** | 이동식 주방 (어디서든 같은 환경) | 누구 컴퓨터에서든 동일하게 실행 |
| **모델(Model)** | 주문표 양식 (이름/수량/가격 칸) | 데이터의 형태 정의 (이름/나이/혈당 칸) |

### "DB 모델을 만든다" = "엑셀 시트의 열 제목을 정하는 것"

예를 들어, `daily_health_logs` 모델을 만든다는 것은:

```
엑셀 시트 "건강기록"을 새로 만들고 열 제목을 정하는 것:
| 날짜 | 사용자 | 수면시간 | 운동여부 | 운동종류 | 식사 | 채소 | 단음료 | 기분 |
```

이 "틀"을 코드로 정의하는 게 "모델 만들기"입니다.
실제로 우리가 할 일은 Claude에게 "이런 엑셀 시트 만들어줘"라고 말하는 것입니다.

### "API를 만든다" = "주방에 주문할 수 있는 메뉴를 정하는 것"

```
메뉴판(API 목록):
- "오늘 건강 데이터 저장해줘"  → POST /api/v1/health/daily
- "이번 주 리포트 보여줘"      → GET /api/v1/health/report
- "내 위험도 점수 알려줘"       → GET /api/v1/predict/score
- "AI한테 메시지 보내줘"        → POST /api/v1/chat/message
```

### "서비스(Service)" = "실제 요리하는 레시피"

API(주문서)가 들어오면, 서비스가 실제 일을 합니다:
1. 주문서(API)가 "위험도 계산해줘" 요청
2. 레시피(서비스)가 냉장고(DB)에서 재료(데이터) 꺼냄
3. 요리(계산) 후 결과를 접시(응답)에 담아 전달

## 3. 구현 순서 — "그래서 뭐부터 하라는 거야?"

### ⚠️ Step 0: 팀과 먼저 합의할 것 (코드 작성 전!)

코드를 쓰기 전에 팀원들과 이것만 먼저 맞추세요:

```
1. "건강기록 테이블에 어떤 칸(열)을 넣을지"
   → 수면시간, 운동여부, 식사, 채소, 단음료, 기분, 복약...
   → BE(백엔드)와 합의: "이 구조로 API 만들어줄 수 있어?"
   → FE(프론트)와 합의: "이 데이터를 화면에서 이렇게 보여줄 거야?"

2. "API 주소를 뭘로 할지"
   → 예: POST /api/v1/health/daily (건강데이터 저장)
   → 예: POST /api/v1/chat/messages (AI 채팅)
   → FE와 합의: "이 주소로 데이터 보내면 돼?"

3. "LLM API를 뭘 쓸지"
   → OpenAI(GPT) vs Anthropic(Claude) vs 무료(Ollama)
   → 비용/성능/팀 예산 고려

이건 Slack이나 회의에서 10분이면 됩니다!
```

### 큰 그림 (5단계)

```
Step 1: 냉장고에 칸 만들기 (DB 모델) ← 팀 합의 후 시작!
   ↓ "데이터를 저장할 곳이 있어야 다른 걸 만들 수 있으니까"
Step 2: 주문서 양식 만들기 (건강 데이터 API)
   ↓ "저장소가 생겼으니 데이터를 넣는 입구를 만들자"
Step 3: 위험도 점수 계산기 만들기 (예측 모델) ← 혼자 해도 OK!
   ↓ "데이터가 쌓이면 이걸로 위험도를 계산"
Step 4: AI 채팅방 만들기 (LLM 통합)
   ↓ "사용자랑 대화하면서 자연스럽게 건강 데이터를 모으는 채널"
Step 5: 결과 보여주기 (대시보드 + 챌린지)
   ↓ "모인 데이터와 예측 결과를 예쁘게 보여주기"
```

## Step 1: 냉장고에 칸 만들기 (건강 데이터 DB 모델)

### 이게 대체 뭔데?

지금 우리 냉장고(데이터베이스)에는 "회원 정보" 칸(users)만 있어요.
건강 데이터를 저장하려면 **새로운 칸 12개**를 더 만들어야 합니다. (총 13개 테이블)

### 만들 엑셀 시트 12개

#### 시트 1: 동의 정보 (user_consents)
> 약관·개인정보·건강정보 동의 상태

왜 필요해? → 가입 후 동의 없이는 건강 데이터를 수집할 수 없어요.

#### 시트 2: 건강 프로필 (health_profiles)
> 온보딩 설문 결과 + BMI·FINDRISC 계산값

왜 필요해? → 사용자의 기본 건강 정보와 그룹(A/B/C)을 저장합니다.

#### 시트 3: 매일 건강기록 (daily_health_logs)
> 매일 기록하는 건강 일기장

```
| 날짜 | 누구 | 수면질 | 수면시간 | 아침 | 점심 | 저녁 | 채소 | 식단구성 | 단음료 | 운동여부 | 운동종류 | 운동시간 | 산책 | 물 | 야식 | 복약 | 기분 | 음주 | 음주량 |
```

왜 필요해? → AI 채팅이나 직접입력으로 받는 건강 데이터를 여기에 저장해요. 각 필드마다 `_source`(출처 추적) 필드가 14개 있어요.

#### 시트 4: 주기적 측정값 (periodic_measurements)
> 체중·허리둘레·혈압·HbA1c·공복혈당

왜 필요해? → 매일 묻지 않고, 사용자가 직접 입력하는 수치형 데이터예요.

#### 시트 5: 위험도 결과 (risk_assessments)
> FINDRISC 점수 계산 결과 저장

왜 필요해? → 온보딩 때 한 번, 이후 데이터가 쌓이면 다시 계산해서 기록해요.

#### 시트 6: 참여 상태 (user_engagements)
> 사용자 응답 참여도 관리 (쿨다운·묶음 횟수)

왜 필요해? → 건강질문 90분 쿨다운, 하루 묶음 횟수 등 운영 로직에 필요해요.

#### 시트 7: 챌린지 템플릿 (challenge_templates)
> 챌린지 종류와 목표 정의

왜 필요해? → "주 3회 운동" 같은 챌린지의 마스터 데이터예요.

#### 시트 8: 사용자 챌린지 (user_challenges)
> 사용자가 참여 중인 챌린지 상태

왜 필요해? → 각 사용자가 어떤 챌린지에 참여 중인지 추적해요.

#### 시트 9: 챌린지 체크인 (challenge_checkins)
> 날짜별 챌린지 수행 기록

왜 필요해? → 매일 달성 여부를 기록하는 스탬프 카드예요.

#### 시트 10: 사용자 설정 (user_settings)
> 알림, 닉네임, 선호 시간

왜 필요해? → 개인 설정값을 저장해요.

#### 시트 11: 채팅방 목록 (chat_sessions)
> AI와의 대화방 관리

왜 필요해? → 한 사용자가 여러 번 대화할 수 있으니까 대화방 단위로 관리해요.

#### 시트 12: 대화 내용 (chat_messages)
> 실제 대화 기록

왜 필요해? → 대화 기록을 저장해서 AI가 문맥을 이해하고, 건강질문 응답도 추적해요.

엑셀에서 "열 추가"를 눌러야 실제로 열이 생기는 것처럼, 코드만 짜면 안 되고 이 명령을 실행해야 DB에 반영됩니다.

```bash
# Docker 컨테이너 안에서 실행해야 합니다
# 1단계: 마이그레이션 파일 생성 (변경 내역서 만들기)
docker compose exec fastapi aerich migrate --name add_health_models

# 2단계: 실제 DB에 적용 (변경 내역서를 실행)
docker compose exec fastapi aerich upgrade
```

> **주의**: 마이그레이션 설정은 `pyproject.toml`의 `[tool.aerich]` 섹션에 있고,
> 마이그레이션 파일은 `app/db/migrations/` 폴더에 자동 생성됩니다.

## Step 2: 주문서 양식 만들기 (건강 데이터 수집 API)

### 이게 대체 뭔데?

Step 1에서 냉장고(DB)에 칸을 만들었죠?
이제 그 칸에 **데이터를 넣는 방법**을 만들어야 합니다.

식당 비유: 냉장고 칸은 만들었는데, 아직 "재료 주문서" 양식이 없어서
재료(건강 데이터)를 넣을 수가 없는 상태입니다.

### 만들 주문서(API) 목록

```
1. "오늘 건강 데이터 저장해줘"
   POST /api/v1/health/daily
   → 수면, 운동, 식사, 채소, 단음료, 기분 등을 한번에 저장

2. "오늘 내 건강 데이터 보여줘"
   GET /api/v1/health/daily
   → 저장된 오늘 데이터를 조회

3. "이번 주/달 건강 리포트 보여줘"
   GET /api/v1/health/report?period=7d
   → 7일/30일/90일 단위 요약 리포트

4. "아직 안 답한 질문이 뭐야?"
   GET /api/v1/health/pending
   → 오늘 아직 응답하지 않은 건강 질문 목록
```

### 90분 쿨다운은 여기서 어떻게 작동해?

```
상황: 사용자가 아침 8시에 AI 채팅에서 "수면 질문"에 답함

→ 백엔드가 "마지막 건강질문 응답: 08:00" 기록
→ 08:00 + 90분 = 09:30까지는 새 건강질문 안 넣음
→ 09:30 이후에 다음 질문 가능

코드로는: user_engagements 시트에서 "가장 최근 답변 시간"을 확인하고,
         현재 시간과 비교해서 90분 지났는지 체크
```

### 작업 위치
- API 라우터: `app/apis/v1/` (새 파일: `health_routers.py`)
- 서비스 로직: `app/services/` (새 파일: `health.py`)
- 저장소: `app/repositories/` (새 파일: `health_repository.py`)
- 데이터 형태: `app/dtos/` (새 파일: `health.py`)

## Step 3: 위험도 점수 계산기 만들기 (예측 모델)

### 이게 대체 뭔데?

사용자가 온보딩 설문(13개 질문)에 답하면,
그 답변으로 **"당뇨에 걸릴 확률이 얼마나 높은지"** 점수를 매기는 계산기입니다.

식당 비유: 손님의 건강 상태를 보고 "이 손님은 저염식 메뉴가 필요해요" 라고
판단해주는 **영양사**를 고용하는 것과 같습니다.

### 어떻게 계산하는 건데?

이미 의학적으로 검증된 **FINDRISC 공식**이 있습니다:

```
점수 계산 예시:

질문: "나이가 어떻게 되세요?"
→ 45세 미만: 0점
→ 45-54세: 2점
→ 55-64세: 3점
→ 65세 이상: 4점

질문: "규칙적으로 운동하세요?"
→ 매일: 0점
→ 주 3-4회: 0점
→ 주 1-2회: 1점
→ 거의 안 함: 2점

... (8개 항목을 이렇게 점수화)

전부 더하면 → 총점 0~22점
0-7점: 낮음 (초록) 😊
8-14점: 보통 (노랑) 🤔
15-18점: 높음 (주황) ⚠️
19-22점: 매우 높음 (빨강) 🔴
```

### 2단계로 나눠서 만들어요

**Phase 1 (먼저)**: 공식 기반 점수 계산
- FINDRISC 공식 그대로 코드로 옮기기
- 온보딩 설문 답변 → 점수 계산 → 결과 저장
- 이것만으로 AUC 0.80 (의학적으로 인정받는 수준)

**Phase 2 (나중)**: AI 학습 모델
- 매일 쌓이는 생활습관 데이터로 점수를 더 정확하게
- scikit-learn(파이썬 AI 도구)으로 학습
- AUC 0.85~0.90 목표

### 작업 위치
- 점수 계산 로직: `app/services/` (새 파일: `prediction.py`)
- 예측 API: `app/apis/v1/` (새 파일: `predict_routers.py`)
- AI 모델 학습: `ai_worker/` (나중에)

### Claude에게 이렇게 말하세요

```
"Step 3 시작할게. 당뇨 위험도 예측 기능을 만들어야 해.
먼저 Phase 1으로, FINDRISC 공식 기반 점수 계산을 구현해줘.
- 온보딩 설문 답변(diabetes.js의 질문들)을 입력으로 받아서
- 0~22점 총점 + 4단계 등급을 계산하고
- risk_assessments 테이블에 저장하는 API
의학적 근거는 medical-domain.md 참고해줘.
/plan"
```

### 완료 확인 방법

- 같은 입력에 항상 같은 점수 (일관성)

## Step 4: AI 채팅방 만들기 (LLM 통합)

### 이게 대체 뭔데?

사용자가 AI와 일상 대화를 하는 **채팅방**입니다.
특별한 점: 대화 중에 **건강 질문을 자연스럽게 끼워 넣는** 것.

식당 비유: 단순히 주문만 받는 게 아니라,
"오늘 많이 피곤해 보이시네요~ 어젯밤 잘 주무셨어요?" 라고
자연스럽게 물어보는 **친절한 웨이터**를 만드는 것입니다.

### 채팅 흐름 (실제 사용자 경험)

```
👤 사용자: "오늘 날씨 좋다~"

🤖 AI: "네, 정말 맑은 날이에요! 이런 날 밖에서 활동하기 좋겠어요 😊"

      ┌─────────────────────────────┐
      │ 💤 어젯밤 수면은 어떠셨어요? │  ← 건강질문 카드 (자동 삽입!)
      │ ○ 6시간 이하, 뒤척임       │
      │ ○ 6-7시간, 보통            │
      │ ● 7시간 이상, 숙면         │  ← 사용자가 터치
      │ ○ 모르겠어요               │
      └─────────────────────────────┘

👤 사용자: (7시간 이상 숙면 선택)

🤖 AI: "잘 주무셨군요! 충분한 수면은 혈당 관리에 정말 도움이 됩니다.
        오늘 하루도 활기차게 보내세요! 😄"
```

### 핵심 규칙 (이건 설계팀이 확정한 거라 바꾸면 안 됨!)

| 규칙 | 설명 | 왜? |
|------|------|-----|
| **90분 쿨다운** | 건강질문 간 최소 90분 간격 | 사용자가 귀찮아하지 않게 |
| **최대 2묶음** | AI 응답 1회에 질문 2개까지만 | 부담 줄이기 |
| **3 터치포인트** | 아침(7-9시), 점심(11:30-13:30), 저녁(17-20시)만 | 적절한 타이밍 |
| **야간 금지** | 22시~07시 건강질문 안 함 | 수면 방해 금지 |
| **First Answer Wins** | AI 채팅이든 직접입력이든 먼저 답한 게 최종 | 중복 방지 |

### 이걸 만들려면 필요한 것

1. **LLM API 연결**: Claude/GPT 같은 AI에게 대화를 보내고 답을 받는 것
2. **건강질문 삽입 로직**: "지금 질문을 끼워넣어도 되는지" 조건 체크
3. **채팅 API**: 프론트엔드에서 메시지를 보내고 받는 통로
4. **면책조항**: AI 답변에 항상 "교육 목적, 전문가 상담 대체 아님" 포함

### 작업 위치
- LLM 처리: `ai_worker/` (AI API 호출 + 프롬프트)
- 채팅 API: `app/apis/v1/` (새 파일: `chat_routers.py`)
- 삽입 로직: `app/services/` (새 파일: `chat.py`)

### Claude에게 이렇게 말하세요

```
"Step 4 시작할게. AI 채팅 시스템을 만들어야 해.
1. 사용자 메시지를 받아서 LLM API(일단 OpenAI)로 보내고 답 받기
2. 90분 쿨다운 + 시간윈도우 + 미답변 체크해서 건강질문 삽입 판단
3. 조건 맞으면 AI 응답 + 건강질문 카드 함께 반환
4. 모든 AI 응답에 면책조항 포함

/plan"
```

- 90분 안에 또 질문하면 건강질문이 안 나옴

## Step 5: 결과 보여주기 (대시보드 + 챌린지)

### 이게 대체 뭔데?

지금까지 모은 건강 데이터와 위험도 점수를
**예쁜 그래프와 카드**로 보여주는 화면입니다.

식당 비유: 매달 "이번 달 영양 리포트"를 예쁜 표로 정리해서
손님에게 보여주는 것과 같습니다.

### 대시보드에 뭐가 보이는데?

```
┌─────────────────────────────────────┐
│  🎯 당뇨 위험도: 11점 (보통)         │ ← 위험도 점수
│  ███████████░░░░░░░░░░ 50%          │ ← 색상 막대
│                                      │
│  📊 이번 주 추이                     │ ← 7일 그래프
│  운동: ■■■□□□□  3/7일               │
│  채소: ■■■■■□□  5/7일               │
│  수면: ■■■■■■□  6/7일               │
│                                      │
│  🏆 진행 중 챌린지                    │ ← 챌린지 카드
│  "주 3회 30분 걷기" - 2/3 완료! 🔥   │
└─────────────────────────────────────┘
```

### 챌린지는 뭔데?

**스탬프 카드**와 같습니다:
- AI가 사용자 데이터를 분석해서 맞춤 챌린지를 추천
- 예: 운동을 안 하는 사용자 → "이번 주 3번 산책하기" 추천

## 7. 용어 사전 (계속 추가)

| 용어 | 쉬운 설명 |
|------|----------|
| **API** | 프로그램끼리 대화하는 창구. "이 데이터 줘" "이거 저장해" 같은 요청서 |
| **DB (데이터베이스)** | 엑셀 파일 같은 데이터 저장소. 여러 시트(테이블)로 구성 |
| **모델 (Model)** | 엑셀 시트의 열 제목(구조)을 코드로 정의한 것 |
| **서비스 (Service)** | 실제 일을 하는 코드. "점수를 계산해라", "데이터를 저장해라" |
| **라우터 (Router)** | API 요청이 들어올 때 "이 요청은 여기로 보내" 하는 길안내 |
| **DTO / Schema** | 데이터의 포장 양식. "이름은 문자, 나이는 숫자" 같은 규칙 |
| **마이그레이션** | 엑셀 시트의 열 구조를 실제 DB에 적용하는 것 |
| **LLM** | Large Language Model. ChatGPT, Claude 같은 AI 대화 모델 |
| **AUC** | 모델 정확도 지표. 1.0이면 완벽, 0.80이면 "꽤 정확" |
| **FINDRISC** | 핀란드에서 만든 당뇨 위험도 계산 공식. 전 세계에서 인정 |
| **Tortoise ORM** | 파이썬에서 DB를 쉽게 다루게 해주는 도구. SQL 안 써도 됨 |
| **FastAPI** | 파이썬으로 API를 빠르게 만드는 도구 |
| **Next.js** | 화면(프론트엔드)을 만드는 자바스크립트 도구 |
| **Docker** | "내 컴퓨터에서는 되는데?" 문제를 해결하는 가상 환경 도구 |
| **Git** | 코드 변경 기록을 관리하는 타임머신 (되돌리기 가능) |
| **PR (Pull Request)** | "내가 이렇게 바꿨는데, 확인해줄래?" 요청서 |
| **JWT** | 로그인 후 받는 신분증. 매 요청마다 "나 로그인한 사람이야" 증명 |
| **쿨다운** | 건강질문을 너무 자주 안 하도록 최소 90분 대기하는 규칙 |
| **듀얼채널** | AI 채팅 + 직접입력(사이드패널) 두 가지 경로 동시 운영 |
| **First Answer Wins** | 두 채널 중 먼저 답한 것이 최종 답변이 되는 규칙 |

## 8. Docker 실행 시 주의 (Windows 필수!)

프로젝트 경로에 **한글**이 포함되어 있어서, Docker가 에러를 냅니다.
Docker 실행 전에 **반드시** 아래를 먼저 실행하세요:

```powershell
# PowerShell에서 실행 (한글 경로를 X: 드라이브로 우회)
subst X: "C:\Users\mal03\Desktop\레퍼런스\마지막 웹프로젝트"
Set-Location X:\

# 그 다음 Docker 실행
docker compose -p ai-health-local up -d postgres redis fastapi nginx
```

왜? → Windows에서 한글 경로를 Docker BuildKit이 처리 못 해서 임시 드라이브(X:)로 우회하는 것.

**확인 방법**: 브라우저에서 `http://localhost/api/docs` 접속 → Swagger UI가 보이면 성공!
