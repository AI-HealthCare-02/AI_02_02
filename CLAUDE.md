# CLAUDE.md — 다나아 프로젝트 실행 가이드

## 1. 프로젝트 개요

- **다나아 (DA-NA-A)**: 만성질환(당뇨·고혈압) 생활습관 AI 건강관리 웹 서비스
- 필수 3기능: `당뇨 위험도 예측 모델` / `건강 추적 대시보드` / `생활습관 챌린지`
- 스택: **FastAPI 백엔드** + **Next.js 14 프론트엔드** + Docker

---

## 1-1. 사용자 소통 규칙 (반드시 지킬 것)

IMPORTANT: 이 프로젝트의 사용자는 코딩 경험이 없는 바이브코더(vibe coder)이다. 모든 소통에서 아래 규칙을 반드시 지킨다:

- **쉬운 말 사용**: 모든 설명, 답변, 플랜에서 전문용어 대신 일상어로 설명한다
- **전문용어 해설 필수**: 개발 용어 사용 시 반드시 괄호 안에 쉬운 해설을 추가한다 (예: "API(프로그램끼리 대화하는 창구)", "DB 모델(엑셀 표의 열 제목을 정의하는 것)")
- **비유와 예시 활용**: 추상적 설명 대신 일상 비유로 설명한다 (예: "서버는 식당 주방, API는 주문서, DB는 냉장고")
- **구체적 행동 지시**: "뭘 해야 하는지" 단계별로 명확하게 안내한다. "이 파일을 수정합니다" 대신 "이 파일에서 3번째 줄을 이렇게 바꿉니다" 수준으로
- **학습 지원**: 단순히 대신 해주는 것이 아니라, 바이브코딩하면서 개발 개념을 자연스럽게 학습할 수 있도록 배경 설명을 곁들인다
- **플랜 모드에서도 동일 적용**: `/plan` 출력에서도 기술 용어에 해설을 붙이고, "왜 이 순서인지" 이유를 쉽게 설명한다

## 4. 작업 실행 규칙 (반드시 지킬 것)

IMPORTANT: 사용자가 "실행해줘", "해줘", "적용해줘" 등 명시적으로 실행을 지시하기 전까지는 **질문에 대한 답변만** 한다. 코드 수정, 파일 생성, 스크립트 실행 등 실제 작업을 먼저 하지 않는다.

- **질문/논의 모드**: 설명, 제안, 플랜 제시까지만. 실행은 하지 않음.
- **실행 모드**: 사용자가 실행을 지시하면 추가 확인 없이 바로 실행한다.

IMPORTANT: 구현 전 반드시 자가 점검 — 하나라도 NO면 먼저 조사한 후 진행:
1. 이 기능과 관련된 기존 파일을 Glob/Grep으로 검색했는가? → YES/NO
2. 기존 네이밍 컨벤션(변수명, 파일명, 함수명)을 확인했는가? → YES/NO
3. 수정할 파일을 import하는 다른 파일이 있는지 확인했는가? → YES/NO
4. 이미 존재하는 유틸/헬퍼/컴포넌트를 재사용할 수 있는지 확인했는가? → YES/NO

IMPORTANT: 큰 파일(50줄 이상)은 반드시 Edit(부분 수정)을 사용. Write(전체 교체) 금지.

## 5. 구현 후 자기 검증 (반드시 수행)

IMPORTANT: 코드 구현 완료 후, 사용자에게 결과를 보고하기 전에 아래 5단계를 반드시 수행한다. 건너뛰지 않는다.

1. **변경 리뷰**: `git diff`로 자신이 변경한 내용을 다시 읽고, 의도와 다른 부분이 없는지 확인
2. **의존성 확인**: 수정한 파일을 import/참조하는 다른 파일을 Grep으로 검색하여 깨진 곳이 없는지 확인
3. **패턴 일관성**: 기존 코드와 네이밍·에러처리·import 구조가 일관적인지 확인
4. **도메인 규칙**: 의료적 표현 금지 위반, 민감정보 노출, 면책조항 누락이 없는지 확인
5. **실행 가능 검증 + 보안**: 가능한 경우 `ruff check`(Python) 또는 `npm run build`(Next.js) 실행하여 구문 오류 없음 확인 + 하드코딩 비밀번호·SQL injection·XSS·로그 민감정보 없는지 확인

결과를 PASS/WARN/FAIL로 간결히 보고한다. FAIL이 있으면 먼저 수정 후 보고.

IMPORTANT: Build 모드에서 §5 완료 후 `/evaluate`는 **필수**다 (선택 아님). "실행해줘"를 받으면 → 구현 → §5 자기검증 → `/evaluate` → 결과 보고까지 한 번에 실행할 것.

## 6. 플랜 품질 기준

IMPORTANT: `/plan` 또는 구현 계획 수립 시, 아래 항목이 포함되어야 한다:

1. **변경 파일 목록**: 수정/생성/삭제할 파일 경로 명시
2. **기존 코드 조사 결과**: 관련 기존 함수/패턴을 확인한 결과 (재사용 가능 여부)
3. **영향 범위**: 변경이 다른 파일/기능에 미치는 영향
4. **위험 요소**: 깨질 수 있는 부분, 주의할 점
5. **검증 방법**: 구현 후 어떻게 테스트/확인할 것인지 (Grep 전체 검색 + 직접 카운트 + 실행 결과 확인 포함)
6. **정량 근거**: 기술 선택에 수치·논문·데이터 근거 (해당 시)

### 플랜 모드 작업 방식 (반드시 지킬 것)

IMPORTANT: 플랜 모드(`/plan` 명령어 또는 Shift+Tab으로 진입) 시 아래 방식으로 작업한다. 사용자가 매번 지시하지 않아도 자동 적용:

- **에이전트 팀 병렬 처리**: 독립적인 조사/작업은 에이전트 팀을 만들어 병렬 실행. 순차 처리 금지
- **교차 검증 필수**: 수정 완료 후 **별도 에이전트**(다른 시각)로 검증. 자기가 만든 걸 자기가 검증하지 않음
- **숫자/명칭 Grep 검증**: 변경된 모든 숫자·이름을 Grep으로 전체 프로젝트 검색하여 누락 확인
- **최신 데모 기준**: 구현은 반드시 V8-8(`docs/prototypes/다나아_웹서비스_데모V8-8.html`)만 참조. 구버전 패턴 사용 금지

## 10. 스킬/명령어 사용 규칙

IMPORTANT: 90개 이상의 스킬 중 실제로 쓸 것만 아래에 정리. 이 규칙을 벗어나지 말 것.

### 5가지 기본 모드
| 모드 | 상황 | 명령어 순서 |
|------|------|------------|
| **Start** | 세션 시작 | `"progress.md 읽어줘"` → `/next-task` |
| **Build** | 새 기능 만들기 | `/plan` → 확인 → `"실행해줘"` → **자기검증(§5)** → `/evaluate` → `/commit-push-pr` |
| **Fix** | 에러/버그 수정 | `/build-fix` → **자기검증(§5)** → `/learn` |
| **Review** | 코드 정리/리팩토링 | `/explore` → `/plan` → `/code-review` |
| **End** | 세션 종료 | `/checkpoint` → `/commit-push-pr` |

### 추가 명령어 (필요 시)
- `/tdd` — 테스트 작성 (RED-GREEN-REFACTOR)
- `/handoff-verify` — PR 머지 직전 fresh-context 최종 검증
- `/code-review` 또는 `/security-review` — 인증/건강 데이터 API 수정 시

### UI 스킬 규칙
- Tailwind 설치 전: `ui-ux-pro-max`만 사용
- Tailwind 설치 후: `baseline-ui`로 컴포넌트 스타일링
- `frontend-design`: **사용 금지** (baseline-ui와 직접 충돌, 의료 앱 부적합)
- `cache-components`: **사용 금지** (Next.js 14.2에서 'use cache' 미지원)

### 보안 필수 검증 시점
- `app/apis/`, `app/services/auth.py` 수정 시 `/code-review` 필수
- PR 머지 전 `/handoff-verify` 실행 필수

### 사용 금지 스킬 (충돌·레거시·미설치 의존성)
`pi-planning-with-files`, `verify-loop`, `frontend-design`, `cache-components`, `executing-plans`
