# AGENTS.md

## 목적

이 파일은 이 저장소에서 작업하는 AI 어시스턴트용 공통 작업 지침이다.
`CLAUDE.md`, `docs/TEAM_AI_PROMPT.md`, 현재 팀 작업 흐름을 합쳐서 만든 기준 문서다.

프로젝트 질문에 답하거나 코드를 수정할 때 먼저 이 파일을 참고한다.

## 현재 팀 기준 구조

- 현재 작업 디렉토리는 팀원이 만든 뼈대 구조와 맞춰진 상태로 본다.
- 백엔드 주요 구조는 아래 기준을 따른다.
  - `backend/apis/v1/`
  - `backend/models/`
  - `backend/services/`
  - `backend/dtos/`
  - `backend/core/`
  - `backend/middleware/`
- 사용자가 명확하게 구조 변경을 요청하지 않는 한, 이 구조를 현재 프로젝트 기본 뼈대로 본다.

## 명세 기준

- 현재 프로젝트는 `최종확정안 V2` 방향을 기준으로 작업 중인 것으로 본다.
- 저장소 안에 `docs/collaboration/`의 구버전 문서가 남아 있어도, 그것이 자동으로 V2보다 우선한다고 가정하지 않는다.
- 코드, `docs/collaboration/*`, handoff 문서, 사용자가 말한 최신 V2 기준이 서로 다르면:
  - 무엇이 서로 다른지 먼저 분명하게 설명하고
  - 어떤 쪽이 더 오래된 기준인지 말하고
  - 큰 방향을 바꾸기 전에는 사용자에게 확인한다.

## 반드시 먼저 알려야 하는 변경

- 작업 중 아래 항목이 바뀌게 되면, 진행 전에 반드시 사용자에게 먼저 알린다.
  - API 경로
  - 요청/응답 형식
  - enum 값
  - DB 스키마
  - migration 상태
  - seed 데이터 전제
- DB나 API가 바뀌는 작업을 단순 리팩터링처럼 조용히 처리하면 안 된다.
- 구현하다가 예상치 못하게 DB/API 변경이 필요해진 경우도 바로 작업 업데이트에서 알린다.

## 소통 원칙

- 사용자는 초보자일 수 있다고 가정하고 설명한다.
- 기술 용어는 필요하면 짧게 풀어서 설명한다.
- 어려운 말보다 명확한 말로 설명한다.
- 단계 안내는 순서대로, 실제로 따라 하기 쉽게 적는다.
- 프로젝트 기준이 여러 개로 보이면 지금 무엇을 기준으로 말하는지 먼저 밝힌다.

## 파일 수정 원칙

- 사용자가 명시적으로 요청하지 않으면 `archive/`, `docs/planning/`, `docs/prototypes/`는 수정하지 않는다.
- `docs/collaboration/`은 통제되는 명세 문서로 취급한다.
- 이 문서들을 수정해야 하는 경우, 먼저 사용자에게 왜 수정이 필요한지 설명한다.
- 넓게 갈아엎는 방식보다 필요한 부분만 정확하게 수정하는 방식을 우선한다.
- 사용자가 요청하지 않은 대규모 폴더 이동이나 이름 변경은 하지 않는다.

## 백엔드 구조 규칙

- API 라우터는 `backend/apis/v1/`에 둔다.
- Pydantic 요청/응답 모델은 `backend/dtos/`에 둔다.
- Tortoise ORM 모델은 `backend/models/`에 둔다.
- 비즈니스 로직은 `backend/services/`에 둔다.
- 설정, 로깅, Redis, JWT, 캐시, Sentry는 `backend/core/`에 둔다.
- 미들웨어는 `backend/middleware/`에 둔다.
- 새 모델을 추가할 때는 모델 등록과 migration 처리까지 같이 확인한다.

## 건강 데이터와 의료 안전

- 이 프로젝트는 건강 관련 민감 정보를 다룬다.
- 로그, 예시, 설명에 민감한 값을 불필요하게 그대로 노출하지 않는다.
- 의료 진단처럼 단정적으로 말하지 않는다.
- 표현은 권고와 안내 중심으로 유지한다.

## 검증 원칙

- 변경 완료라고 말하기 전에 가능한 범위에서 확인한다.
  - 수정된 파일
  - import 영향
  - 테스트 또는 최소 동작 확인
  - 모델 변경 시 migration 영향
  - 라우터/DTO 변경 시 실제 API 영향
- 검증을 못 했으면 했다고 하지 말고, 못 했다고 분명하게 말한다.

## 참고 우선순위

특별한 지시가 없으면 아래 순서로 기준을 본다.

1. 현재 대화에서 사용자가 직접 한 지시
2. 이 `AGENTS.md`
3. 현재 저장소의 실제 코드와 폴더 구조
4. `CLAUDE.md`
5. `docs/TEAM_AI_PROMPT.md`
6. 현재 공식 명세 문서 또는 사용자가 직접 전달한 최신 확정안
7. 예전 handoff 문서나 레거시 메모

## 애매할 때 원칙

- 조용히 추측하지 않는다.
- 무엇이 애매한지 드러내서 말한다.
- 어떤 파일이나 어떤 기준이 충돌하는지 구체적으로 말한다.
- 필요하면 가정을 먼저 밝히고 진행하거나 사용자에게 확인한다.
