# DANAA DB 명세 최종확정안

- 작성일: 2026-04-02
- 문서 성격: 팀 공유용 대표 DB 문서

## 1. 이 문서는 무엇인가

이 문서는 DANAA 서비스에서 어떤 데이터를 어디에 저장하는지 쉽게 설명한 최종 DB 문서입니다.

쉽게 말하면 아래 3가지를 빠르게 이해하기 위한 문서입니다.

- 사용자가 처음 가입할 때 입력한 정보는 어디에 저장되는가
- 매일 답하는 건강 질문은 어떤 구조로 쌓이는가
- 리포트, 챌린지, 설정 데이터는 어떤 테이블로 나뉘는가

이 문서는 대표 문서이면서, 동시에 구현에 필요한 핵심 규칙까지 포함한 최종 기준 문서입니다.

## 2. 먼저 알아두면 좋은 말

- `table`: 같은 종류의 데이터를 모아두는 저장 공간
- `column`: 테이블 안의 한 항목
- `PK`: 한 행을 구분하는 고유 키
- `FK`: 다른 테이블과 연결하는 키
- `정본(source of truth)`: 실제 원본 데이터
- `cache`: 화면을 빠르게 보여주기 위해 저장한 요약값
- `null`: 아직 답하지 않았거나 해당 없음

## 3. 전체 테이블 한눈에 보기

| 테이블명 | 역할 | 핵심 설명 |
|---|---|---|
| `user_consents` | 동의 정보 | 약관, 개인정보, 건강정보, 마케팅 동의의 최신 상태 |
| `health_profiles` | 온보딩 프로필 | 가입 직후 받는 기본 건강 정보 |
| `daily_health_logs` | 매일 건강 기록 | 수면, 식사, 운동, 물, 기분, 음주 등의 일일 응답 |
| `periodic_measurements` | 측정값 기록 | 체중, 허리둘레, 혈압, HbA1c, 공복혈당 |
| `risk_assessments` | 위험도 계산 결과 | 주간/월간 리포트 계산 결과 |
| `user_engagements` | 참여 상태 | 사용자가 얼마나 꾸준히 응답하는지 관리 |
| `challenge_templates` | 챌린지 마스터 | 챌린지 종류와 목표 정의 |
| `user_challenges` | 사용자 챌린지 | 사용자가 현재 참여 중인 챌린지 상태 |
| `challenge_checkins` | 챌린지 수행 기록 | 날짜별 체크인 결과 |
| `user_settings` | 사용자 설정 | 알림, 닉네임, 선호 시간, export 이력 |

## 4. 테이블별 설명

## 4-1. `user_consents`

회원의 최신 동의 상태를 저장합니다.

주요 컬럼:

- `user_id`
- `terms_of_service`
- `privacy_policy`
- `health_data_consent`
- `disclaimer_consent`
- `marketing_consent`
- `consented_at`

핵심 규칙:

- MVP에서는 최신 상태만 저장합니다.
- 별도의 동의 이력 테이블은 두지 않습니다.
- 사용자 1명당 1행입니다.

## 4-2. `health_profiles`

온보딩 설문 결과를 저장합니다.

여기에 들어가는 정보:

- 사용자 그룹 `A/B/C`
- 성별, 연령대, 키, 몸무게
- 가족력, 동반 상태, 치료 방식
- 운동 습관, 식습관, 수면 습관
- 목표
- 초기 FINDRISC 점수와 초기 위험도

주요 컬럼:

- `user_group`
- `relation`
- `gender`
- `age_range`
- `height_cm`
- `weight_kg`
- `bmi`
- `family_history`
- `conditions`
- `treatments`
- `hba1c_range`
- `fasting_glucose_range`
- `exercise_frequency`
- `diet_habits`
- `sleep_duration_bucket`
- `alcohol_frequency`
- `smoking_status`
- `goals`
- `ai_consent`
- `initial_findrisc_score`
- `initial_risk_level`

파생값:

- `bmi`는 키와 몸무게로 계산합니다.
- `has_hypertension`은 `conditions`에 `hypertension`이 있으면 `true`입니다.
- `has_high_glucose_history`도 같은 방식입니다.

중요:

- `relation`은 사용자가 직접 고른 원본 응답입니다.
- `user_group`은 `relation`을 바탕으로 서버가 판단하는 그룹값입니다.
- 즉, 원본은 `relation`, 분류 결과는 `user_group`이라고 이해하면 됩니다.
- `ai_consent` 허용값은 `agreed`, `declined`입니다.

## 4-3. `daily_health_logs`

가장 중요한 테이블입니다.  
사용자가 매일 답한 건강 질문이 날짜별로 저장됩니다.

예를 들면:

- 오늘 잠은 어땠는지
- 아침/점심/저녁을 먹었는지
- 운동했는지
- 물을 몇 컵 마셨는지
- 야식, 기분, 복약, 음주 여부

주요 컬럼:

- `log_date`
- `sleep_quality`
- `sleep_duration_bucket`
- `breakfast_status`
- `lunch_status`
- `dinner_status`
- `vegetable_intake_level`
- `meal_balance_level`
- `sweetdrink_level`
- `exercise_done`
- `exercise_type`
- `exercise_minutes`
- `walk_done`
- `water_cups`
- `nightsnack_level`
- `took_medication`
- `mood_level`
- `alcohol_today`
- `alcohol_amount_level`

source 컬럼:

같은 답변이라도 어디서 들어왔는지 추적해야 해서 source 컬럼을 둡니다.

예:

- `sleep_quality_source`
- `breakfast_status_source`
- `exercise_done_source`

허용값:

- `chat`
- `direct`
- `backfill`

중요 규칙:

- 사용자 1명당 하루 1행만 저장합니다.
- `unique(user_id, log_date)`를 가집니다.
- `exercise_done=false`면 `exercise_type`, `exercise_minutes`는 `null`이어야 합니다.
- `alcohol_today=false`면 `alcohol_amount_level`은 `null`이어야 합니다.
- 답하지 않은 항목은 억지로 `0`이나 `false`로 바꾸지 않고 `null`로 둡니다.
- `exercise_type`과 `exercise_minutes`는 `exercise_done_source`를 공유합니다.

대표 enum:

- `sleep_quality`: `very_good`, `good`, `normal`, `bad`, `very_bad`
- `sleep_duration_bucket`: `under_5`, `between_5_6`, `between_6_7`, `between_7_8`, `over_8`
- `*_status`: `hearty`, `simple`, `skipped`
- `vegetable_intake_level`: `enough`, `little`, `none`
- `meal_balance_level`: `balanced`, `carb_heavy`, `protein_veg_heavy`
- `sweetdrink_level`: `none`, `one`, `two_plus`
- `exercise_type`: `walking`, `running`, `cycling`, `swimming`, `gym`, `home_workout`, `other`
- `nightsnack_level`: `none`, `light`, `heavy`
- `mood_level`: `very_good`, `good`, `normal`, `stressed`, `very_stressed`
- `alcohol_amount_level`: `light`, `moderate`, `heavy`

비저장 계산값:

- `completion_count`
- `total_question_count`
- `completion_rate`

## 4-4. `periodic_measurements`

매일 질문이 아니라, 별도로 입력하는 수치형 건강 데이터 테이블입니다.

예:

- 체중
- 허리둘레
- 혈압
- HbA1c
- 공복혈당

주요 컬럼:

- `measurement_type`
- `measured_at`
- `numeric_value`
- `numeric_value_2`
- `unit`
- `source`

중요 규칙:

- `blood_pressure`는 숫자 2개가 필요하므로
  - 수축기 -> `numeric_value`
  - 이완기 -> `numeric_value_2`
- `source`는 `manual`, `import`, `medical_checkup` 중 하나입니다.

MVP 측정 타입:

- `weight`
- `waist`
- `blood_pressure`
- `hba1c`
- `fasting_glucose`

확장 후보:

- `cholesterol_total`
- `cholesterol_ldl`
- `cholesterol_hdl`

## 4-5. `risk_assessments`

위험도 계산 결과와 리포트용 핵심 점수를 저장합니다.

여기에 저장되는 것:

- FINDRISC 총점
- 위험도 단계
- 수면/식단/운동 점수
- 생활습관 종합 점수
- 상위 긍정 요인
- 상위 위험 요인

주요 컬럼:

- `period_type`
- `period_start`
- `period_end`
- `findrisc_score`
- `risk_level`
- `sleep_score`
- `diet_score`
- `exercise_score`
- `lifestyle_score`
- `data_days`
- `total_days`
- `confidence_note`
- `top_positive_factors`
- `top_risk_factors`

DB에 저장하지 않는 것:

- `risk_badge`
- `overall_grade`
- `coach_summary`

이 값들은 화면 표현이나 API 계산 결과로 다루는 것이 더 안전합니다.

## 4-6. `user_engagements`

사용자의 응답 참여 상태를 운영용으로 관리합니다.

예:

- 최근 7일 응답률
- 연속 미응답 일수
- 현재 상태가 얼마나 유지됐는지
- 오늘 질문 묶음을 몇 번 보여줬는지
- 다음 질문 가능 시점

주요 컬럼:

- `state`
- `seven_day_response_rate`
- `state_changed_at`
- `consecutive_no_response_days`
- `days_at_current_state`
- `upgrade_streak_days`
- `last_response_at`
- `last_session_at`
- `last_question_bundle_at`
- `bundles_today`
- `cooldown_until`
- `last_bundle_key`

이 테이블은 사용자에게 직접 크게 보이는 데이터라기보다, 질문 운영 로직에 중요합니다.

## 4-7. `challenge_templates`

챌린지의 원본 정의 테이블입니다.

예:

- 주 150분 운동
- 11시 이후 야식 금지
- 하루 물 6컵 이상

주요 컬럼:

- `code`
- `name`
- `emoji`
- `category`
- `description`
- `goal_criteria`
- `default_duration_days`
- `evidence_summary`
- `for_groups`
- `phase`
- `sort_order`
- `is_active`

## 4-8. `user_challenges`

사용자가 실제 참여한 챌린지 상태를 저장합니다.

주요 컬럼:

- `template_id`
- `selection_source`
- `status`
- `started_at`
- `ends_at`
- `completed_at`
- `current_streak`
- `best_streak`
- `progress_pct`
- `target_days`
- `days_completed`
- `today_checked`
- `daily_log`

운영 규칙:

- 동시에 활성 챌린지는 최대 2개입니다.
- 같은 템플릿에 대한 중복 active 레코드는 허용하지 않습니다.
- 완료 후 재참여는 가능합니다.
- cache 컬럼 갱신 책임은 DB trigger가 아니라 서비스 레이어에 둡니다.

## 4-9. `challenge_checkins`

날짜별 챌린지 수행 결과를 저장합니다.

주요 컬럼:

- `user_challenge_id`
- `checkin_date`
- `status`
- `judged_by`
- `source_field_keys`
- `source_period`
- `note`

중요 규칙:

- 같은 챌린지에 대해 같은 날짜 체크인은 1개만 저장합니다.
- `unique(user_challenge_id, checkin_date)`를 가집니다.

## 4-10. `user_settings`

사용자 설정의 최신 상태를 저장합니다.

예:

- 챌린지 리마인더
- 주간 리포트 알림
- 선호 시간대
- 최근 export 시점

주요 컬럼:

- `nickname`
- `morning_reminder`
- `evening_reminder`
- `challenge_reminder`
- `weekly_report`
- `reminder_time_morning`
- `reminder_time_evening`

- `risk_change_alert`

## 5. JSON 컬럼 규칙

JSON 컬럼은 "여러 값을 한 컬럼에 묶어서 저장하는 구조"입니다.

주요 JSON 컬럼:

- `conditions`
- `treatments`
- `diet_habits`
- `goals`
- `top_positive_factors`
- `top_risk_factors`
- `goal_criteria`
- `for_groups`
- `preferred_times`
- `source_field_keys`

상세 규칙:

- `conditions`
  - string 배열
  - 허용값: `hypertension`, `high_glucose_history`, `heart_disease`, `none`
- `treatments`
  - string 배열
  - 허용값: `none`, `oral_medication`, `insulin`, `diet_exercise_only`
- `diet_habits`
  - string 배열
  - 허용값: `irregular_meals`, `frequent_snacks`, `late_night_eating`, `low_vegetable_intake`
- `goals`
  - string 배열
  - 허용값: `weight_management`, `blood_sugar_control`, `improve_sleep`, `increase_activity`, `regular_meals`
- `top_positive_factors`
  - string 배열
  - 예시값: `good_sleep`, `consistent_breakfast`, `regular_walk`, `good_hydration`
- `top_risk_factors`
  - string 배열
  - 예시값: `low_activity`, `carb_heavy_meals`, `frequent_nightsnack`, `poor_sleep`
- `for_groups`
  - string 배열
  - 허용값: `A`, `B`, `C`
- `preferred_times`
  - string 배열
  - 허용값: `morning`, `lunch`, `evening`
- `source_field_keys`
  - string 배열
  - 예시: `["exercise_done", "exercise_minutes"]`

`goal_criteria` 예시:

```json
{
  "metric": "exercise_minutes",
  "operator": "gte",
  "target": 30,
  "period": "daily"
}
```

## 6. 제약조건과 인덱스

Unique:

1. `user_consents.user_id`
2. `health_profiles.user_id`
3. `daily_health_logs(user_id, log_date)`
4. `challenge_checkins(user_challenge_id, checkin_date)`
5. `user_settings.user_id`
6. `user_engagements.user_id`
7. `challenge_templates.code`

Partial unique index 후보:

1. `user_challenges(user_id, template_id)` where `status='active'`

Indexes:

1. `daily_health_logs(user_id, log_date desc)`
2. `periodic_measurements(user_id, measurement_type, measured_at desc)`
3. `risk_assessments(user_id, period_type, period_end desc)`
4. `user_challenges(user_id, status, started_at desc)`
5. `challenge_checkins(user_challenge_id, checkin_date desc)`

FK 정책:

1. 사용자 삭제 시 도메인 row는 cascade 처리
2. `challenge_templates`는 hard delete 대신 `is_active=false`
3. `user_challenges` 삭제 시 `challenge_checkins`는 cascade 처리

구현 메모:

- 위 목록은 실제 migration에 반영해야 하는 핵심 제약입니다.
- `user_challenges(user_id, template_id)`의 active 중복 방지는 partial unique index가 가장 적합합니다.
- `daily_health_logs`, `periodic_measurements`, `risk_assessments`는 사용자 기준 조회가 많으므로 복합 인덱스가 중요합니다.

## 7. API 정렬 메모

신규 API 필드명은 DB 의미와 맞춘 이름을 사용합니다.

대표 필드:

- `sleep_duration_bucket`
- `vegetable_intake_level`
- `meal_balance_level`
- `exercise_done`
- `walk_done`
- `mood_level`
- `alcohol_amount_level`

과도기 alias:

- `sleep` -> `sleep_quality`
- `veggie` -> `vegetable_intake_level`
- `foodcomp` -> `meal_balance_level`
- `exercise` -> `exercise_done`
- `walk` -> `walk_done`
- `mood` -> `mood_level`
- `alcohol_amount` -> `alcohol_amount_level`

더 이상 쓰지 않는 이름:

- `sleep_hours`

## 8. 구현 부록

실제 구현 시 자주 참고할 항목만 다시 묶어두면 아래와 같습니다.

partial unique:

- `user_challenges(user_id, template_id)` where `status='active'`

필수 JSON 허용값 묶음:

- `conditions`: `hypertension`, `high_glucose_history`, `heart_disease`, `none`
- `treatments`: `none`, `oral_medication`, `insulin`, `diet_exercise_only`
- `diet_habits`: `irregular_meals`, `frequent_snacks`, `late_night_eating`, `low_vegetable_intake`
- `goals`: `weight_management`, `blood_sugar_control`, `improve_sleep`, `increase_activity`, `regular_meals`
- `for_groups`: `A`, `B`, `C`
- `preferred_times`: `morning`, `lunch`, `evening`

필수 인덱스 묶음:

- `daily_health_logs(user_id, log_date desc)`
- `periodic_measurements(user_id, measurement_type, measured_at desc)`
- `risk_assessments(user_id, period_type, period_end desc)`
- `user_challenges(user_id, status, started_at desc)`
- `challenge_checkins(user_challenge_id, checkin_date desc)`

## 9. 최종 정리

DANAA의 DB는 크게 5덩어리로 이해하면 됩니다.

1. 가입과 동의: `user_consents`, `health_profiles`
2. 매일 기록: `daily_health_logs`
3. 수치 입력과 리포트: `periodic_measurements`, `risk_assessments`
4. 챌린지: `challenge_templates`, `user_challenges`, `challenge_checkins`
5. 운영과 설정: `user_engagements`, `user_settings`

이 구조를 기준으로 엑셀 설계서, ORM 모델, migration, API가 이어집니다.
