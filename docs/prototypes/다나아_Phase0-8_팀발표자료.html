Phase별 구현 내용

8개 Phase에 걸쳐 백엔드 전체를 순차적으로 구현했습니다

Phase 0-1 인증 + JWT + DB 기초

회원가입(이메일 중복 체크, bcrypt 비밀번호 암호화), 로그인(JWT(디지털 신분증) 발급), 토큰 갱신(리프레시 쿠키), 사용자 정보 관리

  • API 6개: 회원가입, 로그인, 토큰갱신, 동의, 내정보 조회, 내정보 수정
  • 핵심: Tortoise ORM(코드로 DB 조작), HTTPBearer 인증, Aerich DB 버전관리

Phase 2 온보딩 + FINDRISC 위험도

14개 설문 항목으로 사용자 건강 프로필 수집, 자동 BMI(체질량지수) 계산, FINDRISC(당뇨 위험도를 측정하는 국제 의학 공식) 초기 점수 산출

  • API 3개: 동의, 설문제출(+새 신분증 발급), 진행상태 확인
  • 핵심: 당뇨 관계에 따라 그룹 자동 분류 (진단받음→A, 위험군→B, 일반→C)
  • FINDRISC 8변수: 나이+BMI+허리둘레+운동+채소+고혈압+고혈당+가족력 = 0~26점

Phase 3-4 건강 데이터 수집 (듀얼 채널)

매일 18개 건강 항목을 AI 채팅 + 직접입력 두 가지 경로로 수집. "먼저 답한 쪽이 이기는(First Answer Wins)" 규칙 적용

  • API 6개: 오늘기록 조회/수정, 일괄입력, 미입력항목, 측정값 등록/조회
  • 핵심: 14개 출처 추적 필드(AI 채팅인지 직접입력인지 기록), 소급입력 3일 제한
  • 측정값: 체중, 허리둘레, 혈압, HbA1c, 공복혈당

Phase 5 AI 채팅 (LLM + SSE 스트리밍)

OpenAI GPT-4o-mini로 실시간 스트리밍 응답. 대화 중 건강질문을 자연스럽게 삽입하는 AI 코칭 채팅

  • API 3개: 메시지 전송(SSE 실시간 전송), 대화기록, 건강답변 저장
  • 핵심: 9단계 처리 과정(→ LLM 파이프라인 탭에서 상세 설명), 7개 질문 묶음, 90분 간격 제한
  • 안전장치: 의료 진단/처방 표현 금지, "전문가 상담 추천" 면책조항 자동 삽입

Phase 6 챌린지 + 대시보드 + 위험도

생활습관 챌린지(6카테고리, 스트릭), 앱 첫 화면 대시보드, 생활습관 점수 기반 위험도 분석

  • API 8개: overview, join, checkin, calendar, dashboard/init, risk/latest, recalculate, analysis/summary
  • 핵심: 최대 2개 동시 챌린지, 생활습관 점수(수면25%+식단30%+운동30%+음주15%)

Phase 7 스케줄러 + 크론 작업

매일 자정에 자동으로 FINDRISC 위험도 재계산 + 사용자 참여 상태 갱신. 여러 서버가 동시 실행되지 않도록 잠금 장치 적용

  • 자동 작업 2개: 매일 자정(00:00) + 매주 월요일(01:00)
  • 핵심: Redis 분산 잠금으로 중복 실행 방지, 5단계 참여 상태(활발→보통→낮음→휴면→동면)
  • 안전장치: 자동 작업이 실패해도 API는 정상 동작 (독립적)

Phase 8 테스트 + 버그 수정 + 보안 강화

코드 전체 점검으로 버그 7개 수정, 보안 강화 6건, 자동 테스트 총 46개 작성

  • 보안: 쿠키 위조방지(SameSite), HTTPS 강제(HSTS), 운영환경 API문서 숨김, 비활성 계정 차단
  • 테스트: FINDRISC 점수 경계값, 실시간 전송(SSE) 포맷, 온보딩 전체 흐름, 참여 상태 전환

LLM(AI 대화 엔진) 채팅 파이프라인

사용자가 메시지를 보내면 AI가 실시간으로 답하기까지의 9단계 처리 과정

한 줄 요약

사용자가 "안녕" → 서버가 검증 → 대화 이력 불러오기 → 건강질문 넣을지 판단 → AI한테 보내기 → 실시간으로 글자 전송 → 저장

파이프라인 흐름도 (체인 구조 = 순차 실행)

각 단계가 완료된 후 다음 단계로 넘어가는 체인(Chain) 패턴입니다. 공장 컨베이어 벨트처럼 1번부터 9번까지 순서대로 처리됩니다. 모든 함수는 app/services/chat.py에 정의되어 있습니다.

1
사전 검증 — "보내도 되는지 확인"
_validate_request(user_id, session_id)

받는 것: 사용자 번호, 대화방 번호 → 돌려주는 것: 에러 메시지 or 통과
AI 서비스 연결 확인 + 오늘 50회 넘었는지 체크

↓
2
대화방 준비 — "어느 대화방에서 할지 정하기"
_prepare_session(user_id, message, session_id)

받는 것: 사용자 번호, 메시지 내용, 대화방 번호 → 돌려주는 것: 대화방 정보
기존 대화방 불러오기 or 새로 만들기 + 사용자 메시지 저장

↓
3
이전 대화 불러오기 — "뭘 얘기했었는지 기억하기"
ChatMessage.filter(session=session).limit(10)

받는 것: 대화방 정보 → 돌려주는 것: 최근 메시지 10개 목록
AI가 맥락을 이해하도록 최근 대화를 함께 전달

↓
4
건강질문 넣을지 판단 — "지금 물어봐도 될까?" (라우터 = 갈림길)

받는 것: 사용자 번호 → 돌려주는 것: 질문 묶음 이름들 (예: "수면", "운동")
4가지 조건(90분 경과? 시간대 맞나? 아직 안 물었나? 밤이 아닌가?) 모두 만족하면 질문 삽입

↓
5
AI 역할 지시서 작성 — "이렇게 대답해!" 지시문 만들기
_build_system_prompt(user_id, eligible_bundles)

받는 것: 사용자 번호, 질문 묶음 → 돌려주는 것: 지시문 텍스트
사용자 그룹(A/B/C)에 맞게 + 건강질문 삽입 지시를 동적으로 조합

↓
6
AI한테 보낼 대화 묶기 — "전체 맥락을 한 봉투에 담기"
_build_openai_messages(user_id, history, bundles)

받는 것: 사용자 번호, 대화이력, 질문묶음 → 돌려주는 것: 대화 목록
역할 지시서 + 이전 대화 + 현재 메시지를 하나로 조립

↓
7
AI에게 질문하기 — "글자가 하나씩 실시간으로 돌아옴"
_stream_openai(messages)

받는 것: 대화 목록 → 돌려주는 것: 글자 조각들 (실시간 스트리밍)
GPT-4o-mini 모델, 최대 1024 토큰(약 500자), 창의성 0.7

↓
8
AI 답변 저장 — "나중에 다시 볼 수 있도록 기록"
_save_response(session, full_response, bundles)

받는 것: 대화방, 전체답변, 질문묶음 → 돌려주는 것: 없음 (저장만)
AI 응답 전체를 DB에 저장 + 어떤 건강질문을 물었는지 기록

↓
9
"다 끝났어!" 신호 보내기 — 프론트에 결과 전달
_build_done_data(session_id, bundles)

받는 것: 대화방 번호, 질문묶음 → 돌려주는 것: 완료 데이터
화면에 건강질문 UI를 띄울 수 있도록 데이터를 함께 전달

SSE 이벤트 포맷 (국제 표준)

SSE(Server-Sent Events) = 서버가 브라우저에 실시간으로 글자를 하나씩 보내주는 기술. 카카오톡처럼 메시지가 뜨는 것과 비슷!

// 1. 토큰 이벤트 (실시간 글자 전송) event: token data: {"content": "안녕"} // 2. 완료 이벤트 (건강질문 데이터 포함) event: done data: {"session_id": 42, "health_questions": [...]} // 3. 에러 이벤트 event: error data: {"message": "대화를 찾을 수 없어요."}

핵심 기술 결정

왜 이 기술을 선택했는지, 어떤 대안이 있었는지

DB 도구: Tortoise ORM vs SQLAlchemy

선택: Tortoise ORM (코드로 DB를 다루는 도구)

  • FastAPI와 같은 비동기(동시 처리) 방식으로 빠름
  • 데이터 검증 도구(Pydantic)와 자연스럽게 연동
  • DB 구조 변경 기록(마이그레이션) 도구 내장

실시간 전송: SSE vs WebSocket

선택: SSE (서버→브라우저 단방향 실시간 전송)

  • AI 채팅은 서버가 보내기만 하면 되니 SSE가 딱 맞음
  • 일반 HTTP 기반이라 Nginx 설정이 간단
  • 연결 끊기면 자동 재연결, 브라우저가 기본 지원

중복 실행 방지: Redis 잠금 vs DB 잠금

선택: Redis 분산 잠금 (임시 메모판으로 "내가 하고 있어!" 표시)

  • DB에 부담 주지 않고 빠르게 잠금/해제
  • 서버가 갑자기 꺼져도 시간 지나면 자동으로 잠금 해제
  • Lua(Redis 내장 언어) 스크립트로 "확인+잠금"을 한 번에 처리 (중간에 끼어들기 불가)

위험도 계산: 의학 공식 vs AI 학습 모델

선택: FINDRISC 규칙 기반 (논문으로 검증된 국제 공식)

  • 의학 논문으로 검증된 공식 (정확도 AUC ~0.80, 즉 100명 중 80명 정확 판별)
  • 학습 데이터 없어도 바로 쓸 수 있음
  • 다음 단계에서 AI 학습 모델(정확도 0.82~0.85, 더 정밀) 추가 계획
