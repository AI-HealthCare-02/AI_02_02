LLM이란? (Large Language Model = 대규모 언어 모델)

쉬운 비유: 수백만 권의 요리책을 읽은 매우 똑똑한 요리사라고 생각하세요.

  • 이 요리사는 수백만 권의 책(인터넷의 글, 논문, 대화 등)을 읽고 "다음에 어떤 말이 올까?"를 예측하는 훈련을 받았어요.
  • "안녕하"까지 입력하면 "세요"가 올 확률이 높다는 걸 알고 있는 거예요.
  • ChatGPT, Claude, Gemini 같은 서비스가 모두 LLM을 기반으로 만들어졌어요.
  • 우리 다나아 프로젝트에서는 OpenAI의 GPT-4o-mini라는 LLM을 사용할 거예요.
좀 더 자세히 알고 싶다면 (선택)

LLM은 "대규모 언어 모델"의 약자예요. "대규모"란 모델의 크기(파라미터 수)가 수십억~수천억 개라는 뜻이에요.

GPT-4o-mini도 수백억 개의 파라미터를 가지고 있어요. 파라미터란 "AI의 뇌세포" 같은 거예요. 뇌세포가 많을수록 더 복잡한 생각을 할 수 있죠.

중요한 건, LLM은 "진짜로 이해하는 게 아니라 통계적으로 예측"한다는 거예요. "사과는 (빈칸)다"에서 "맛있"이 올 확률이 높다는 걸 학습한 거지, 사과가 뭔지 진짜 아는 건 아니에요.

토큰이란? (AI가 말을 세는 단위)

사람은 글자 수나 단어 수로 글의 길이를 재지만, AI는 토큰(token)이라는 단위로 셉니다.

한글의 경우

한글 1글자 = 약 2~3 토큰
"안녕하세요" = 약 7~8 토큰
"오늘 날씨가 좋네요" = 약 12~14 토큰

영어의 경우

영어 1단어 = 약 1~2 토큰
"Hello" = 1 토큰
"Good morning" = 2 토큰

왜 토큰을 알아야 해요? 두 가지 이유가 있어요:
1. 비용: 토큰 수에 비례해서 돈이 나가요. 토큰을 아끼면 비용도 아껴요.
2. 기억력: "최대 토큰 수"를 넘기면 AI가 이전 대화를 잊어버려요. 한 번에 보낼 수 있는 양에 한계가 있는 거예요.
토큰 계산 예시 보기
텍스트예상 토큰 수이유
"오늘 아침에 운동했어요"약 15~18 토큰한글 9글자 + 조사/공백
시스템 프롬프트 (200자)약 400~600 토큰한글은 영어보다 토큰 효율이 낮음
대화 5턴 (질문+답변)약 2,000~3,000 토큰이전 대화도 매번 같이 보냄

GPT-4o-mini의 입력 비용이 $0.15/100만 토큰이니까, 3,000토큰을 보내면 약 $0.00045(= 0.6원) 정도예요.

컨텍스트 윈도우 (AI의 작업 기억력)

쉬운 비유: 요리사의 작업대 크기예요. 작업대가 크면 더 많은 재료(대화 내용)를 한꺼번에 올려놓고 요리할 수 있어요.

다나아에서 사용자가 AI랑 대화할 때, 이전 대화 내용도 함께 보내야 해요. AI한테는 "기억"이라는 게 없어서, 매번 전체 대화 기록을 다시 보내는 거예요. 컨텍스트 윈도우가 크면 더 오래된 대화도 포함할 수 있어요.

실전 팁: 128K 토큰이 넉넉해 보이지만, 한글은 토큰 효율이 낮아서 생각보다 빨리 차요. 대화가 길어지면 오래된 메시지부터 잘라내는 로직이 필요해요. 이걸 "대화 이력 관리"라고 해요.

  • 절대 공개하면 안 돼요! 누가 가져가면 내 돈으로 AI를 써버릴 수 있어요

비동기(async/await)란? (한 번에 여러 일 처리하기)

쉬운 비유: 카페에서 커피 주문을 생각해 보세요. 동기(sync)는 1번 손님 커피 완성될 때까지 2번 손님이 주문조차 못 하는 거예요. 비동기(async)는 1번 주문받고, 커피 만드는 동안 2번 주문도 받는 거예요.

동기 (Sync) 방식

1. OpenAI에 질문 보냄
2. 3초간 아무것도 못 함 (대기)
3. 답변 받음
4. 다음 사용자 처리

문제: 100명이 동시에 접속하면 99명이 기다려야 함!

비동기 (Async) 방식 ← 우리가 쓸 방식!

1. OpenAI에 질문 보냄
2. 기다리는 동안 다른 사용자 처리
3. 답변 오면 그때 이어서 처리
4. 동시에 여러 명 처리!

100명이 동시에 접속해도 거의 동시에 처리!

Python 코드로 보면?
# 동기 방식 (안 좋은 방식) def ask_ai_sync(question): response = openai.chat.completions.create(...) # 3초 대기 (블로킹!) return response # 비동기 방식 (좋은 방식) ← 우리가 쓸 것! async def ask_ai_async(question): response = await openai.chat.completions.create(...) # 3초 대기하지만 # await 덕분에 기다리는 동안 다른 요청 처리! return response

async def는 "이 함수는 비동기예요"라는 선언이고, await는 "여기서 기다리는데, 기다리는 동안 다른 일 해도 돼"라는 표시예요.

FastAPI는 비동기를 기본 지원하는 프레임워크예요. 그래서 async def를 쓰면 자연스럽게 비동기로 동작해요.

파이프라인이란?

완료율(completion_rate) — 오늘 데이터를 얼마나 모았나?

대시보드에서 "오늘 60% 완료"처럼 보여주는 그 숫자의 계산 방식이에요.

# 완료율 계산 의사코드 # 그룹별 필수 필드가 다름! if user_group == "A": required = ["sleep", "breakfast", "took_medication", "lunch", "exercise", "veggie", "walk", "sugar", "water"] # 9개 else: # B, C 그룹 required = ["sleep", "breakfast", "lunch", "exercise", "veggie", "walk", "sugar", "water"] # 8개 # null이 아닌 필드 수 / 전체 필드 수 filled = sum(1 for f in required if getattr(daily_log, f) is not None) completion_rate = filled / len(required) # 0.0 ~ 1.0 # 예: A그룹이 sleep, breakfast, exercise 3개 답함 # → 3/9 = 0.33 (33%)

FINDRISC란?

Finnish Diabetes Risk Score의 약자예요. 핀란드에서 만든 당뇨 위험도 점수인데, 전 세계적으로 검증됐어요 (Lindstrom & Tuomilehto 2003).

혈액검사 없이도 나이, BMI, 허리둘레, 운동, 식습관 같은 생활습관 정보만으로 당뇨 위험도를 예측할 수 있어서 우리 프로젝트에 딱 맞아요.

왜 FINDRISC를 선택했나요? 혈액검사 같은 의료 데이터 없이도 쓸 수 있고, 논문으로 검증됐고, 계산이 단순해서(if-else만으로 가능) MVP에 최적이에요. 나중에 KDRS(한국형)나 ML 모델을 추가하면 정확도가 더 올라가요.

스시 컨베이어벨트처럼 하나씩 나오는 대로 바로 먹음!

AI가 글자를 생성하는 즉시 화면에 표시. ChatGPT처럼 타닥타닥! 사용자 경험이 훨씬 좋아요.

스트리밍 코드 보기 (Python)
# stream=True를 추가하면 스트리밍! stream = await client.chat.completions.create( model="gpt-4o-mini", messages=[...], stream=True # ← 이 한 줄이 핵심! ) # 토큰이 하나씩 흘러옴 (비동기 for 루프) async for chunk in stream: token = chunk.choices[0].delta.content if token: print(token, end="") # 출력: "잘" → "하셨" → "어요" → "!" # 한 글자(토큰)씩 실시간으로 나옴!

SSE (Server-Sent Events) — 서버가 클라이언트에게 실시간 전송

비유: 라디오 방송이에요. 서버(방송국)가 계속 데이터를 내보내고, 클라이언트(라디오)가 계속 들어요. 한 번 연결하면 서버가 원하는 만큼 계속 보낼 수 있어요.

FastAPI에서는 StreamingResponse라는 걸 써서 SSE를 구현해요. 프론트엔드(Next.js)에서는 EventSource 또는 fetch + ReadableStream으로 받아요.

SSE vs WebSocket: 둘 다 실시간 통신이지만, SSE는 "서버 → 클라이언트" 일방통행이에요. AI 채팅에서는 AI가 한쪽으로 계속 보내기만 하면 되니까 SSE가 딱 맞아요. WebSocket보다 구현이 훨씬 간단해요.

우리의 4가지 SSE 이벤트

AI가 응답할 때 4가지 종류의 "소포"가 순서대로 날아와요. 프론트엔드는 이 소포의 종류(event)를 보고 화면을 업데이트해요.

1단계: token 이벤트 (AI 답변 글자)
AI가 생성하는 글자가 하나씩 날아와요. 여러 번 반복됩니다. 프론트엔드는 받는 즉시 화면에 이어붙여요.
event: token
data: {"token": "잘"}

event: token
data: {"token": "하셨"}

event: token
data: {"token": "어요!"}
2단계: health_questions 이벤트 (건강질문 카드) — 4조건 충족 시에만!
4가지 조건이 모두 충족되면 건강질문 카드 데이터가 날아와요. 프론트엔드는 이걸 받으면 질문 카드 UI를 띄워요.
event: health_questions
data: {"bundle_key": "sleep", "questions": [{"field": "sleep", "question": "어젯밤 수면은 어떠셨나요?", "options": ["매우 나빴어요", "나빴어요", "보통이에요", "좋았어요", "아주 잘 잤어요"]}]}
3단계: disclaimer 이벤트 (면책조항)
모든 AI 응답에는 반드시 면책조항이 포함돼요. 이건 빠지면 안 돼요!
event: disclaimer
data: {"text": "이 정보는 교육 목적이며, 전문 의료 상담을 대체하지 않습니다."}
4단계: done 이벤트 (스트림 종료)
"전송 끝!"이라는 신호예요. 프론트엔드는 이걸 받으면 로딩 표시를 끄고, 메시지를 DB에 저장할 수 있어요.
event: done
data: {"message_id": "msg_123", "session_id": "uuid-here"}

90분 쿨다운 구현

Redis(빠른 임시 저장소)에서 마지막으로 건강질문을 보낸 시각을 확인해서, 90분이 지났는지 체크해요.

Redis란? 데이터를 메모리(RAM)에 저장하는 초고속 저장소예요. DB(PostgreSQL)보다 훨씬 빨라서, 자주 확인해야 하는 쿨다운 시간 같은 걸 여기에 넣어요.

# 의사코드 (pseudocode) # Redis에서 마지막 건강질문 시각 조회 last_q_time = redis.get(f"user:{user_id}:last_health_q") if last_q_time is None: # 한 번도 건강질문 안 했음 → 삽입 가능 can_insert = True elif (현재시각 - last_q_time) >= 90분: # 90분 지남 → 삽입 가능 can_insert = True else: # 아직 90분 안 됨 → 삽입 불가 can_insert = False # 삽입 성공하면 시각 갱신 if can_insert: redis.set(f"user:{user_id}:last_health_q", 현재시각)

Nginx 설정 — SSE가 끊기지 않게!

SSE는 오래 연결을 유지해야 하는데, Nginx(웹서버 앞단의 교통정리 서버)가 중간에 끊어버릴 수 있어요. 아래 설정이 꼭 필요해요.

# nginx/conf.d/default.conf에 추가 location /api/v1/chat/ { proxy_read_timeout 300s; # 5분까지 기다림 (기본 60초면 끊김!) proxy_buffering off; # 버퍼링 끄기 (쌓아두지 말고 즉시 전달) chunked_transfer_encoding on; # 조각 전송 허용 proxy_set_header Connection ''; # 연결 유지 proxy_http_version 1.1; # HTTP 1.1 필수 (SSE 지원) }
이거 빠뜨리면: SSE 연결이 60초 후에 끊겨요. AI가 긴 답변을 생성 중이면 중간에 잘려요. 반드시 설정하세요!

프론트엔드(Next.js)에서 SSE 받기

백엔드가 SSE로 보내면, 프론트엔드에서 받아서 화면에 표시해야 해요. JavaScript의 fetch + ReadableStream을 사용해요.

프론트엔드 SSE 수신 코드 보기
// Next.js 컴포넌트에서 SSE 수신하는 방법 const sendMessage = async (content) => { // 1. 서버에 메시지 보내기 (POST 요청) const response = await fetch('/api/v1/chat/messages', { method: 'POST', headers: { 'Content-Type': 'application/json', 'Authorization': `Bearer ${token}` // JWT 토큰 }, body: JSON.stringify({ session_id: sessionId, content: content }) }); // 2. SSE 스트림 읽기 const reader = response.body.getReader(); const decoder = new TextDecoder(); while (true) { const { done, value } = await reader.read(); if (done) break; const text = decoder.decode(value); // text = "event: token\ndata: {\"token\": \"안\"}\n\n" // 3. SSE 이벤트 파싱 const lines = text.split('\n'); for (const line of lines) { if (line.startsWith('event: ')) { eventType = line.slice(7); // "token", "health_questions", etc. } if (line.startsWith('data: ')) { const data = JSON.parse(line.slice(6)); switch (eventType) { case 'token': // 글자를 화면에 이어붙이기 setMessage(prev => prev + data.token); break; case 'health_questions': // 건강질문 카드 UI 표시 setHealthQuestions(data.questions); break; case 'disclaimer': // 면책조항 표시 setDisclaimer(data.text); break; case 'done': // 로딩 끝, 메시지 ID 저장 setLoading(false); break; } } } } };

이 코드를 외울 필요는 없어요! Claude Code한테 "SSE 받는 프론트엔드 코드 만들어줘" 하면 만들어줘요. 구조만 이해하면 돼요.

LLM은 기억이 없어요. 매번 전체 대화를 다시 보내야 해요. 이걸 어떻게 관리하는지 알아봅시다.

# 매번 보내는 messages 배열의 구조 messages = [ # 1. 시스템 프롬프트 (항상 첫 번째) {"role": "system", "content": "당신은 다나아 건강관리 AI..."}, # 2. 이전 대화 이력 (오래된 것부터) {"role": "user", "content": "오늘 날씨 좋다~"}, {"role": "assistant", "content": "정말 좋은 날씨네요!"}, {"role": "user", "content": "산책 갔다 왔어"}, {"role": "assistant", "content": "산책 좋아요!"}, # 3. 지금 사용자가 보낸 메시지 (항상 마지막) {"role": "user", "content": "운동도 해야 하나?"} ]
주의! 대화가 길어지면 토큰이 빠르게 소모돼요. 전략:
- 최근 10턴(사용자+AI 각각 = 20개 메시지)까지만 포함
- 오래된 대화는 요약해서 시스템 프롬프트에 포함하는 방법도 있음 (MVP 이후)
- 시스템 프롬프트(~500토큰) + 이력(~2000토큰) + 새 메시지(~20토큰) = 약 2,500토큰

FastAPI에서 SSE 보내는 구조

백엔드에서 SSE를 보내려면 StreamingResponse라는 특별한 응답 방식을 써요.

FastAPI SSE 엔드포인트 구조 보기
from fastapi.responses import StreamingResponse @router.post("/chat/messages") async def send_message(req: ChatRequest, user = Depends(get_current_user)): async def event_stream(): # 1단계: OpenAI 스트리밍 호출 stream = await openai_client.chat.completions.create( model="gpt-4o-mini", messages=build_messages(req, user), stream=True ) full_response = "" async for chunk in stream: token = chunk.choices[0].delta.content if token: full_response += token # SSE 형식으로 토큰 전송 yield f"event: token\ndata: {json.dumps({'token': token})}\n\n" # 2단계: 건강질문 삽입 판단 if await should_insert_health_question(user.id): questions = await get_next_questions(user.id) yield f"event: health_questions\ndata: {json.dumps(questions)}\n\n" # 3단계: 면책조항 disclaimer = "이 정보는 교육 목적이며, 전문 의료 상담을 대체하지 않습니다." yield f"event: disclaimer\ndata: {json.dumps({'text': disclaimer})}\n\n" # 4단계: 완료 yield f"event: done\ndata: {json.dumps({'message_id': msg_id})}\n\n" return StreamingResponse( event_stream(), media_type="text/event-stream", headers={"Cache-Control": "no-cache"} )

yield는 "여기까지 먼저 보내고, 나머지는 나중에 보낼게"라는 뜻이에요. 이게 스트리밍의 핵심!

가드레일 구현 — 3단계 안전 체계

AI가 부적절한 응답을 하는 것을 막기 위해 3단계로 방어해요.

1단계: 입력 필터 (Pre-filter)
사용자 메시지가 들어오자마자 검사해요.
- 욕설/혐오 키워드 매칭 (빠른 정규식 검사)
- 개인정보 패턴 감지 (주민번호, 전화번호 등)
- 감지 시: AI 호출 없이 바로 안내 메시지 반환 → 비용 절약!
2단계: 프롬프트 제어 (In-prompt)
시스템 프롬프트에 금지 규칙을 명시해요.
- "의료적 진단 표현 절대 금지"
- "혈당 수치를 묻지 마세요"
- "~해야 합니다 대신 ~을 권장합니다 사용"
대부분의 경우 이 단계에서 걸려요. 프롬프트가 잘 만들어져 있으면 AI가 알아서 따라요.
3단계: 출력 검사 (Post-filter)
AI 응답이 생성된 후, 전송 전에 최종 검사해요.
- 금지 패턴 정규식: "~해야 합니다", "당뇨입니다", "약을 드세요" 등
- 감지 시: 해당 표현을 안전한 표현으로 자동 교체
- 교체 로그를 남겨서 나중에 프롬프트 개선에 활용
왜 3단계나 필요해요? AI는 가끔 예측 불가능한 답변을 해요. 프롬프트로 99% 막아도, 나머지 1%가 의료 표현일 수 있어요. 의료 도메인에서는 그 1%가 큰 문제가 될 수 있으니 다중 방어선이 필수예요!

JWT 페이로드 상세 — 입장권 안에 뭐가 적혀 있나?

JWT(입장권) 안에 들어있는 정보를 페이로드(payload)라고 해요. 우리 JWT에는 이런 정보가 들어 있어요.

JWT 페이로드 예시 보기
// 온보딩 완료 후 JWT 페이로드 { "sub": "user_12345", // 사용자 고유 ID "nickname": "건강이", // 닉네임 "user_group": "B", // A/B/C 그룹 ← 온보딩 후 추가됨! "onboarding_done": true, // 온보딩 완료 여부 "iat": 1711929600, // 발급 시각 (issued at) "exp": 1711933200 // 만료 시각 (1시간 후) }

중요 포인트:

  • 회원가입 직후: user_group이 없음 (null)
  • 온보딩 완료 후: user_group이 "A", "B", "C" 중 하나로 설정됨
  • 온보딩 완료 API 응답에 새 JWT가 포함되어 있고, 프론트엔드는 이 새 JWT로 교체해야 함
흔한 버그: 온보딩 완료 후 프론트엔드에서 JWT를 새 것으로 교체하지 않으면, user_group이 null인 상태로 API를 호출하게 됩니다. 그러면 A그룹 전용 기능(복약확인)이 B/C그룹에게도 보이거나, 아예 작동하지 않아요.

HTTP 메서드란? (API의 "동사")

API 요청에는 "무엇을 할 건지"를 알려주는 동사가 있어요. 식당 비유로 설명하면:

메서드뜻식당 비유다나아 예시
GET 데이터 조회 (가져오기) "메뉴판 보여주세요" GET /health/daily/2026-04-01 → 4월 1일 건강 기록 조회
POST 데이터 생성 (새로 만들기) "주문하겠습니다" (새 주문 생성) POST /chat/sessions → 새 대화방 생성
PATCH 데이터 부분 수정 "음료만 바꿔주세요" (일부만 변경) PATCH /health/daily/2026-04-01 → 수면 데이터만 추가
DELETE 데이터 삭제 "주문 취소해주세요" (다나아에서는 거의 안 씀. 건강데이터 삭제 불가)
PATCH vs PUT: PUT은 전체를 통째로 교체하고, PATCH는 바뀐 부분만 보내요. 우리는 건강 데이터를 한 번에 다 안 채우고 조금씩 추가하니까 PATCH가 딱 맞아요!

HTTP 상태 코드 — 서버의 "답변 신호"

서버가 요청을 받으면 숫자로 된 상태 코드를 돌려줘요. 이걸 보고 성공인지 실패인지 알 수 있어요.

코드의미색상다나아에서
200 성공! 초록 정상 조회/수정 완료
201 생성 성공! 초록 새 대화방/챌린지 생성 완료
400 잘못된 요청 노랑 필수 필드 누락, 형식 오류
401 인증 실패 빨강 JWT 만료 → 로그인 다시
403 권한 없음 빨강 야간 차단(NIGHT_BLOCKED), 그룹 제한
409 충돌 노랑 First Answer Wins에 의해 이미 답한 필드
422 처리 불가 노랑 소급입력 3일 초과, 미래 날짜
429 너무 많은 요청 빨강 API 속도 제한 초과
500 서버 에러 빨강 버그! 즉시 수정 필요

인증 흐름 — JWT가 API에 어떻게 쓰이나?

모든 API 요청에는 JWT(입장권)가 필요해요 (로그인/가입 제외). 이걸 Bearer 토큰이라고 해요.

// API 요청 시 JWT를 이렇게 보내요 fetch('/api/v1/chat/messages', { method: 'POST', headers: { 'Authorization': 'Bearer eyJhbGciOiJIUzI1...', // ← JWT 여기! 'Content-Type': 'application/json' }, body: JSON.stringify({ session_id: "uuid-here", content: "오늘 날씨 좋다~" }) })
서버에서 JWT를 검증하는 과정:
1. Authorization 헤더에서 JWT 꺼내기
2. JWT 서명 검증 (위조 여부 확인)
3. 만료 시각(exp) 확인 (만료됐으면 401 에러)
4. 페이로드에서 user_id, user_group 꺼내기
5. 이 정보로 해당 사용자에 맞는 처리 진행

에러 응답 형식 (모든 API 공통)

에러가 발생하면 항상 같은 형식으로 응답해요. 프론트엔드가 에러를 처리하기 쉽게요.

// 에러 응답 공통 형식 { "error": { "code": "NIGHT_BLOCKED", // 에러 코드 (영어 대문자) "message": "야간 시간에는 건강질문을 받을 수 없습니다", // 사람이 읽는 메시지 "detail": { // 추가 정보 (있을 때만) "blocked_until": "2026-04-02T07:00:00" } } }

주요 에러 코드: NIGHT_BLOCKED, COOLDOWN_ACTIVE, ALREADY_ANSWERED, BACKFILL_TOO_OLD, FUTURE_DATE, GROUP_RESTRICTED, ONBOARDING_REQUIRED
