Project Overview

이 프로젝트가 뭔가요?

코딩을 처음 하는 분도 이해할 수 있도록 쉽게 설명합니다

💡
다나아(DA-NA-A)란?
만성질환(당뇨, 고혈압) 환자와 예방이 필요한 분들을 위한 AI 생활습관 건강관리 웹 서비스입니다. 매일 짧은 질문(하루 ~26초)으로 건강 데이터를 수집하고, AI가 당뇨 위험도를 예측해서 맞춤 챌린지를 제시합니다.
🎯
필수 3기능
📊 당뇨 위험도 예측 모델
설문 데이터 → AI가 FINDRISC 기반으로 위험도 계산
📈 건강 추적 대시보드
운동·수면·식단 등 7묶음 10문항으로 생활습관 추적
🏆 생활습관 챌린지
P1/P2/P3 우선순위에 따라 맞춤 챌린지 제공
🤖
AI 채팅 수집 방식
AI가 대화하듯 건강 질문을 삽입합니다.

90분 쿨다운 규칙:
건강 질문 간격은 최소 90분. 사용자 피로도 최소화.

3 터치포인트:
🌅 아침 7-9시 | ☀️ 점심 11:30-13:30 | 🌆 저녁 17-20시

하루 총 응답 시간:
10문항 × ~2.6초 = 약 26초
📊
의학적 근거
모든 질문은 국제 논문으로 검증된 근거가 있습니다:
🏃 운동: 주 150분 → 당뇨 위험 58% 감소 (DPP 2002)
😴 수면: 6시간 이하 → 위험 28% 증가 (Cappuccio 2010)
🍚 백미: 과다섭취 → 위험 55% 증가 (Hu 2012)
😔 우울감 → 당뇨 위험 60% 증가 (Mezuk 2008)

Hooks System

.claude/hooks/ — 자동 방어 메커니즘

Claude가 실수로 코드를 망가뜨리는 것을 자동으로 방지합니다

🛡️ write-guard.sh 작동 원리

① 트리거
Claude가 Write 도구를 사용하려 할 때마다 자동 실행
↓
② 파일 확인
대상 파일이 존재하는지, 몇 줄인지 확인
↓
③-A: 50줄 미만
✅ 그냥 진행 (새 파일이거나 작은 파일)
③-B: 50줄 이상
⚠️ "정말 전체 교체가 필요한가요?" 확인 요청
사용자가 승인해야만 진행
✅
왜 50줄 기준인가?
50줄 이상 파일 = 중요한 기존 코드가 많이 있는 파일. 이걸 Write로 전체 교체하면 기존 코드를 날릴 위험이 있습니다. Edit(부분 수정)을 쓰는 게 훨씬 안전합니다.

🔧 write-guard.sh 코드 분석

#!/bin/bash # Write 도구 사용 전 자동 실행 # jq 없이 순수 bash로 작성 (Windows 호환) # stdin에서 JSON 읽기 INPUT="" while IFS= read -r line || [ -n "$line" ]; do INPUT="${INPUT}${line}" done # 파일 경로 추출 (jq 없이 grep으로) FILE_PATH=$(echo "$INPUT" | grep -oP '"file_path"\s*:\s*"[^"]*"' | ...) # 파일 존재 + 줄 수 확인 if [ -f "$FILE_PATH" ]; then LINE_COUNT=$(wc -l < "$FILE_PATH") if [ "$LINE_COUNT" -gt 50 ]; then # ask = 사용자에게 확인 요청 echo '{"hookSpecificOutput":{ "permissionDecision":"ask", "permissionDecisionReason":"..."}}' fi fi
🪟
Windows 호환성 최적화
기존 설계의 Hook은 jq(JSON 파서)를 사용했는데, Windows Git Bash에는 jq가 없어서 실패했습니다. 이 버전은 순수 bash grep으로 대체했습니다.

⚙️ settings.json — Hook 등록

{ "hooks": { "PreToolUse": [ ← 도구 사용 직전 { "matcher": "Write", ← Write만 감시 "hooks": [{ "type": "command", "command": "bash .claude/hooks/write-guard.sh", "timeout": 5000 ← 5초 제한 }] } ] } }
🔬
/evaluate 스킬 — 코드 품질 자동 검증
구현이 완료되면 /evaluate 명령어를 입력하세요. 격리된 에이전트가 최근 변경 파일을 자동으로 검증합니다.
① 기존 패턴 준수
네이밍 일관성, 에러 핸들링 패턴, import 구조, 중복 코드 여부
② 의존성 안전
수정 파일을 import하는 곳 영향, 삭제된 export 사용 여부
③ 도메인 규칙
의료 표현, 면책 문구, 건강 데이터 최소 수집 원칙
④ 보안
하드코딩 비밀번호, SQL 인젝션, XSS 가능성, API 키 노출
📊 결과 형식: ✅ PASS ⚠️ WARN (파일:줄번호) ❌ FAIL (수정 방향)

기술 스택 완전 해설

각 기술이 무엇이고 왜 사용하는지 초보자 눈높이로 설명합니다

🌐
프론트엔드: Next.js 14
Next.js란?
React를 기반으로 만든 웹 프레임워크. 쉽게 말하면 "웹사이트의 화면을 만드는 도구"입니다.

주요 파일 구조:
apps/web/app/ ├── page.js ← 홈/랜딩 페이지 ├── layout.js ← 공통 레이아웃 └── onboarding/ ← 설문 페이지 └── diabetes/ ← 당뇨 온보딩
React 18.3.0 Next.js 14.2.0
⚡
백엔드: FastAPI
FastAPI란?
Python으로 만든 API 서버 프레임워크. "프론트엔드의 요청에 응답하는 서버"입니다.

레이어드 아키텍처:
🌐 apis/ — API 진입점 (라우터)
⚙️ services/ — 비즈니스 로직
🗄️ repositories/ — DB 쿼리
📊 models/ — DB 테이블 정의
🧠
AI Worker: PyTorch
AI Worker란?
무거운 AI 계산을 FastAPI와 분리해서 처리하는 별도 서버입니다.

사용 라이브러리:
🔥 PyTorch — 딥러닝 모델
📐 Scikit-learn — 기계학습 알고리즘
💬 Sentence-Transformers — 텍스트 임베딩
Redis를 통해 FastAPI와 비동기 통신
🐳
인프라: Docker
Docker란?
서비스들을 독립적인 컨테이너에 담아 실행하는 도구. 어디서나 같은 환경에서 실행 가능합니다.

5개 서비스:
서비스역할포트
MySQL데이터베이스3306
Redis캐시/메시지큐6379
FastAPIAPI 서버8000
AI-WorkerAI 추론-
Nginx리버스 프록시80
🔐
인증: JWT + bcrypt
JWT란?
로그인 후 발급받는 "디지털 신분증"입니다. 서버에 요청할 때마다 이 토큰을 보내서 "나는 누구입니다"를 증명합니다.

토큰 구성:
Access Token
유효시간: 60분. API 요청 시 사용
Refresh Token
유효시간: 14일. Access Token 갱신에 사용
🗄️
ORM: Tortoise ORM
ORM이란?
Python 코드로 데이터베이스를 조작하는 도구입니다. SQL을 직접 쓰는 대신 Python 클래스로 DB를 다룹니다.

User 테이블 구조:
id, email, hashed_password name, gender, birthday phone_number is_active, is_admin last_login created_at, updated_at
asyncmy로 MySQL과 비동기 통신

Evaluation Criteria

평가 기준 핵심 요약

이 프로젝트를 심사할 때 어떤 기준으로 평가하는지 정리했습니다

📊
평가에서 가장 중요한 것 (4가지)
  • 📈
    정량 근거
    "왜 이렇게 했나?" → 논문, 수치, 데이터로 답할 수 있어야 함
    예: "DPP 2002 논문에 따르면 주 150분 운동 시 58% 위험 감소"
  • 📝
    문서화
    기획서, 아키텍처 다이어그램, API 명세서, 테스트 결과
  • 🧪
    테스트 및 검증 결과
    단위 테스트, 통합 테스트, 모델 성능 지표 (AUC + 추가 지표)
  • 🚀
    배포 환경에서 안정 동작
    로컬뿐 아니라 AWS EC2에서 핵심 3기능이 정상 작동
⚡
기술 검증 기준 (개발 시 항상 체크)
  • ⏱️
    P95 latency 3초 이내
    100명이 요청했을 때 95명이 3초 안에 응답받아야 함
    예측 API, 대시보드 API, 챌린지 API 각각 측정
  • 📊
    모델 성능 지표 2개 이상
    AUC만으론 부족. F1, Precision, Recall 등 추가
    Phase 1 목표: AUC ≥ 0.80
  • ⚡
    비동기 처리 성능 기록
    AI 추론을 비동기로 분리한 전후 응답 시간 비교
  • 🔐
    인증/인가 구조 설명 가능
    JWT 발급 → 검증 → 권한 분리 흐름 다이어그램으로 설명

Dev Workflow

개발 시작 워크플로우

PC 켜고 → 코드 쓰기까지, 매번 이 순서대로 따라하면 됩니다

⚠️ Windows 필수 우회 작업
이 프로젝트 경로에 한글(레퍼런스, 마지막 웹프로젝트)이 포함되어 있어 Docker가 오류를 냅니다.
아래 Step 0을 반드시 먼저 실행해야 나머지 단계가 작동합니다.
Step 0 — 한글 경로 우회 (매일 1회)
PowerShell을 관리자 권한으로 열고 실행:
subst X: "C:\Users\mal03\Desktop\레퍼런스\마지막 웹프로젝트" Set-Location X:\
이후 모든 Docker 명령어는 X:\ 에서 실행합니다.
Step 1 — Docker 서비스 시작
X:\ 경로에서 핵심 4개 서비스를 실행:
docker compose -p ai-health-local up -d mysql redis fastapi nginx
확인 브라우저에서 http://localhost/api/docs 열리면 성공
Step 2 — 프론트엔드 개발 서버 시작
새 터미널 창을 열고 (Git Bash 또는 VSCode 터미널):
cd apps/web npm run dev
확인 http://localhost:3000 에서 Next.js 화면이 보이면 성공
Step 3 — Claude Code 시작
또 다른 터미널에서 프로젝트 루트로 이동 후:
claude
Claude Code가 실행되면 CLAUDE.md를 자동으로 읽고 프로젝트 규칙을 파악합니다.
Step 4 — 작업 컨텍스트 파악
Claude에게 지금 어디쯤 있는지 확인 요청:
"task_plan.md 와 progress.md 파일을 읽고 현재 상황을 요약해줘"
Claude가 이전 세션에서 어디까지 했는지, 다음에 뭘 해야 하는지 알려줍니다.
Step 5 — 작업 시작
상황에 따라 두 가지 방법 중 선택:
/next-task # 다음 할 일을 Claude가 제안해줄 때 /plan # 새 기능을 만들 계획을 먼저 세울 때

🌅 매일 시작할 때 (Step 0~2)

  • PowerShell에서 subst X: 실행
  • X:\ 로 이동
  • docker compose up 4개 서비스 실행
  • 새 터미널에서 npm run dev
  • 브라우저 2곳 확인 (localhost, localhost:3000)
컴퓨터를 껐다 켤 때마다 Step 0~2를 반복해야 합니다.
subst 매핑은 재부팅하면 초기화됩니다.

💬 Claude 세션 시작할 때 (Step 3~5)

  • 터미널에서 claude 입력
  • task_plan.md + progress.md 확인 요청
  • /next-task 또는 /plan 으로 시작
  • 작업 후 세션 종료 전 핸드오프 작성
Claude 세션은 컨텍스트(기억)가 유한합니다.
오래 사용하면 초기화가 필요 — /checkpoint 로 저장하세요.

12개 명령어 완전 가이드

명령어 언제 쓰나? 한 줄 설명 우선순위
/plan 새 기능 만들기 전 구현 계획을 먼저 세우고 사용자 승인을 받는다 필수
/next-task 세션 시작 시 / 할 일 모를 때 task_plan.md 기반으로 다음 작업을 제안 필수
/build-fix 에러가 났을 때 에러 메시지 분석 후 수정 방법 제안·실행 (3회 실패 시 중단) 필수
/evaluate 코드 수정 후 검증할 때 네이밍·보안·의료 가드레일 등 프로젝트 특화 품질 체크 필수
/commit-push-pr 작업 완료 후 저장·공유 build→보안 스캔→commit→push→PR 생성 전 파이프라인 자동화 필수
/checkpoint 세션 종료 직전 현재 진행 상황을 progress.md에 저장 + 핸드오프 작성 필수
/learn 에러 해결 후 / 새로 배운 것 있을 때 배운 내용을 findings.md에 분류·기록 (자동화 제안도 포함) 권장
/tdd 새 API나 핵심 로직 만들 때 테스트 먼저 작성 (RED) → 코드로 통과 (GREEN) → 개선 (REFACTOR) 권장
/explore 코드 구조를 모를 때 / 리팩토링 전 키워드 확장 탐색, 관련 파일·의존성 자동 분석 권장
/code-review 리팩토링 후 / 코드 정리 후 CRITICAL/HIGH/MEDIUM 3단계 보안+품질 리뷰 권장
/handoff-verify PR 머지 직전 fresh-context 7단계 검증 (빌드·타입·린트·테스트·코드리뷰·보안) PR 전
/security-review 인증/건강 데이터 API 수정 시 CWE Top 25 취약점 스캔 (SQL인젝션·XSS·하드코딩 시크릿 등) 보안 시

# 질문 왜 물어보나? 어디서 답을 찾나?
1 지금 어디에 있는가?
현재 코드 상태는?		 git status + 최근 변경 파악 git status, task_plan.md
2 어디로 가는가?
이번 세션 목표는?		 목표 없이 작업하면 방향을 잃음 task_plan.md 현재 Phase
3 지금 하는 작업의 목적은?
왜 이걸 만드나?		 큰 그림과 연결이 끊기지 않도록 CLAUDE.md, docs/planning/
4 뭘 알아냈는가?
이 세션에서 배운 것?		 배운 것을 기록하지 않으면 다음 세션에서 또 헤맴 progress.md 이번 세션 로그
5 실제로 뭘 했는가?
코드 변경 내역?		 말만 하고 실제 코드는 안 바꾼 게 없는지 확인 git diff, 변경된 파일 목록
