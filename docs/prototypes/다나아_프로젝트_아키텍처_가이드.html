📌 다나아가 뭔가요?

다나아(DA-NA-A)는 당뇨·고혈압 같은 만성질환을 가진 사람들이
자신의 생활습관을 기록하고, AI로 건강 위험도를 확인하고, 챌린지를 통해 건강해지도록 돕는 웹 서비스입니다.

🩸 당뇨 위험도 예측 📊 건강 추적 대시보드 🏃 생활습관 챌린지

🔧 어떤 기술로 만들었나요?

FastAPI 서버 (Python으로 만든 웹 API)
Next.js 14 화면 (React 기반 프론트엔드)
Docker 실행 환경 묶음
MySQL 데이터베이스
Redis AI 작업 대기열
Nginx 입구 서버

아키텍처 설계, 왜 필요할까요?
단순히 "기능을 만드는 것"을 넘어, 안정적이고 확장 가능한 구조를 설계해야 하는 이유

🏗️ 핵심 목표 한 줄 요약

"무거운 AI 작업을 안정적으로 처리하면서 확장 가능한 구조를 만드는 것"

AI 모델이 결과를 내려면 시간이 걸립니다. 그 사이에 서비스가 멈추면 안 되고, 사용자가 많아져도 버틸 수 있어야 합니다.

🛡️
시스템 안정성 (High Availability)
하나가 고장 나도 서비스는 계속 돌아가야 한다
▼
🌉 비유
한 강을 건너는 다리가 하나뿐이라면? 그 다리가 부서지면 강을 건널 수 없어요.
하지만 다리가 3개라면, 하나가 공사 중이어도 나머지 두 개로 건널 수 있죠!
서버도 똑같습니다. 여러 서버에 나눠 처리(로드밸런싱)하면 하나가 죽어도 서비스가 유지돼요.
실제 적용: Nginx가 들어오는 요청을 여러 FastAPI 서버에 나눠줍니다. 하나의 서버에 장애가 발생해도 다른 서버가 요청을 처리하기 때문에 사용자는 오류를 느끼지 못합니다.
📈
유연한 확장성 (Scalability)
사용자가 10배 늘어도 서버를 유연하게 키울 수 있다
▼
🧩 비유
레고 블록처럼 필요할 때 부품을 추가하면 됩니다.
AI 분석 요청이 폭발적으로 늘어나면? AI Worker(주방장)를 한 명 더 고용하면 돼요.
전체 서비스를 다시 만들 필요 없이, 필요한 부분만 늘리면 됩니다.
수직 확장 (Vertical Scaling): 서버 하나를 더 강한 것으로 교체 (컴퓨터 RAM, CPU 업그레이드)
수평 확장 (Horizontal Scaling): 같은 서버를 여러 개 늘리기 (AI Worker를 1개 → 10개로)
다나아 프로젝트는 AI Worker를 Docker로 독립 분리해서 수평 확장이 쉽습니다.
⚡
성능 최적화 (Performance)
무거운 작업은 뒤에서, 사용자에겐 빠른 응답을
▼
🛣️ 비유
고속도로 톨게이트가 하나뿐이라면 차들이 엄청 막힙니다.
하지만 주문 접수(FastAPI)와 요리(AI Worker)를 분리하면,
홀 매니저는 주문만 빠르게 받고, 주방에서 시간이 걸리는 요리를 따로 합니다.
비동기 처리의 핵심: AI 모델 학습/추론은 몇 초 ~ 몇 분이 걸릴 수 있습니다. FastAPI가 직접 처리하면 그 시간 동안 다른 요청을 처리할 수 없습니다. Redis Stream을 통해 AI 작업을 큐에 넣고, AI Worker가 따로 처리하면 FastAPI는 계속 다른 요청을 받을 수 있습니다.
🔧
유지보수 용이성 (Maintainability)
하나를 고쳐도 나머지가 망가지지 않는다
▼
🗄️ 비유
잘 정리된 서랍장을 생각해보세요.
양말 서랍, 셔츠 서랍이 따로 있으면 양말을 교체할 때 셔츠를 건드릴 필요가 없죠.
코드도 역할별로 폴더를 나누면, 인증 코드를 수정해도 AI 코드가 망가지지 않아요.
계층형 구조(Layered Architecture): app/ 폴더를 보면 apis/, services/, repositories/, models/ 등으로 완전히 분리되어 있습니다. 데이터베이스 로직(repositories/)을 바꿔도 API 라우터(apis/)에 영향이 없습니다.
🔒
보안 강화 (Security)
건강 데이터는 민감하다 — 여러 겹의 보안이 필요
▼
🏨 비유
호텔 카드키 시스템처럼요.
로비에서 체크인(Nginx) → 엘리베이터 카드키(FastAPI 인증) → 방 카드키(API 권한) —
각 단계마다 확인하니까 외부에서 직접 방으로 들어올 수 없어요.
실제 적용: Nginx가 외부에서 오는 요청을 먼저 받고, 내부 FastAPI의 실제 IP는 외부에 노출되지 않습니다. FastAPI는 JWT(JSON Web Token) 인증으로 로그인한 사람만 데이터에 접근할 수 있게 합니다. 혈당, 혈압 같은 민감한 건강 정보는 최소한만 저장하고 노출합니다.
💰
비용 효율성 (Cost Efficiency)
쓴 만큼만 내는 구조로 운영 비용을 최소화
▼
💡 비유
전기 요금처럼요. 24시간 에어컨을 켜놓는 것보다,
필요할 때만 켜는 게 당연히 더 저렴하죠!
AI Worker처럼 무거운 서비스는 실제로 필요할 때만 실행하고, 평소엔 끌 수 있어요.
실제 적용: 무거운 AI Worker 컨테이너는 평소에 꺼두고, 필요할 때만 실행합니다 (CLAUDE.md: "ai-worker는 별도, 무거워서 필요할 때만 실행"). AWS S3에 AI 모델 파일을 저장하면 서버 디스크 비용을 줄일 수 있습니다. Docker를 사용하면 클라우드에서 사용한 만큼만 비용이 나옵니다.

Nginx — 안내 데스크
Reverse Proxy · SSL 종단점 · SSE 버퍼링
인프라
▼
🍽️ 식당 비유
식당 입구의 안내 데스크예요.
손님(사용자)이 들어오면 "어디로 가시면 됩니다"라고 안내하죠.
안내 데스크는 실제 요리사(FastAPI)가 어디 있는지 손님에게 알려주지 않아요.
이게 바로 보안의 핵심입니다 — 내부 서버 주소가 외부에 노출되지 않아요.
✅ 실제로 하는 일
1. 리버스 프록시: 사용자 → Nginx → FastAPI로 요청을 중계
2. HTTPS 암호화: 인터넷으로 전송되는 데이터를 암호화 (도청 방지)
3. SSE 버퍼링: AI 분석 중 실시간으로 "처리 중..." 메시지를 전송할 때 연결을 유지
📂 관련 파일
nginx/default.conf— 현재 실행 중인 nginx 설정 (로컬 개발용)
nginx/prod_https.conf— 운영 배포용 HTTPS 설정 (Let's Encrypt 인증서 포함)
🧑‍💼
FastAPI — 홀 매니저
백엔드 API 서버 · 인증 · AI 작업 등록 · SSE
백엔드
▼
🍽️ 식당 비유
홀 매니저(서빙)와 같아요.
손님(사용자) 주문을 받고, 주방(AI Worker)에 전달하고, 결과가 나오면 손님에게 가져다줍니다.
하지만 직접 요리하지는 않아요! AI 분석이 필요하면 대기표(Redis)에 걸어두고
주방장이 완성하면 그때 손님에게 알려줍니다.
✅ 실제로 하는 일
1. 회원가입 / 로그인: POST /api/v1/auth/signup, POST /api/v1/auth/login
2. JWT 토큰 인증: 로그인한 사용자에게만 API 접근 허용
3. AI 작업 등록: 당뇨 예측 요청 → Redis Stream에 "작업표" 추가
4. 실시간 결과 전송 (SSE): AI가 완료되면 결과를 실시간으로 클라이언트에 push
📂 관련 파일
app/apis/v1/auth_routers.py— 회원가입·로그인 API
app/services/auth.py— 인증 비즈니스 로직
app/utils/jwt/— JWT 토큰 발급·검증
app/Dockerfile— FastAPI 컨테이너 이미지 설정
🎫
Redis — 주문 대기표 시스템
메시지 브로커 · 작업 큐 · Redis Stream
인프라
▼
🍽️ 식당 비유
손님이 주문하면 주방에 대기표(번호표)를 걸어두는 시스템이에요.
홀 매니저(FastAPI)가 주문을 받으면 "37번 대기표"를 걸고, 손님에게 "잠시만요" 하죠.
주방장(AI Worker)은 대기표를 순서대로 가져가 요리합니다.
장점: 식당이 잠깐 닫혀도(서버 재시작) 대기표는 사라지지 않아요!
✅ 실제로 하는 일
1. Redis Stream: 단순 큐가 아닌, 영속적인 메시지 스트림. 서버가 꺼져도 작업이 사라지지 않음
2. 디커플링(Decoupling): FastAPI와 AI Worker가 서로 몰라도 됨. Redis만 알면 됨
3. 상태 관리: XACK(완료), XCLAIM(재처리) 명령으로 작업이 성공했는지 실패했는지 추적
4. 확장성: AI Worker를 여러 개 띄워도 Redis가 중복 없이 작업을 분배
📂 관련 파일
docker-compose.yml → redis— redis:alpine 이미지로 실행
.env → REDIS_PORT— 포트 설정 (기본 6379)
👨‍🍳
AI Worker — 주방장
AI 모델 학습 · 추론 · Consumer Group
AI
▼
🍽️ 식당 비유
대기표를 보며 실제로 요리하는 주방장이에요.
홀 매니저(FastAPI)는 주방에 들어오지 않아요. 오로지 주방장만 요리합니다.
주방장이 여러 명이라면(Worker 여러 개) 동시에 여러 요리를 할 수 있어요!
✅ 실제로 하는 일
1. Redis Stream에서 "당뇨 예측 요청" 작업을 가져옴
2. 저장된 AI 모델(PyTorch, scikit-learn)을 로드해서 분석 실행
3. 사용자 입력 데이터로 당뇨 위험도 계산
4. 결과를 Database에 저장하고 Redis에 "완료!" 신호 전송

⚠️ 현재 다나아 프로젝트에서 AI Worker는 구조만 만들어진 상태(stub)입니다. 실제 AI 로직은 구현 예정!
📂 관련 파일
ai_worker/— AI Worker 폴더 (현재 skeleton 상태)
ai_worker/Dockerfile— PyTorch, scikit-learn 설치된 컨테이너
docker-compose.yml → ai-worker— 메모리 4GB 제한 설정
🗃️
MySQL — 냉장고
관계형 데이터베이스 · 사용자·건강 기록 저장
데이터
▼
🍽️ 식당 비유
신선한 재료를 보관하는 냉장고예요.
손님의 식이 알레르기 정보, 주문 이력 등을 냉장고(MySQL)에 저장해두고

사용자 정보 (이메일, 이름, 성별, 생년월일, 전화번호, 비밀번호 해시) 저장

1요청 접수 (Request)

사용자가 "당뇨 위험도 분석" 버튼을 누릅니다.
브라우저 → Nginx → FastAPI 순서로 요청이 전달됩니다.
FastAPI는 JWT 토큰으로 "이 사람이 로그인한 사람이 맞는지" 확인합니다.
확인 완료 → FastAPI가 응답: "요청 접수 완료! 처리 중입니다..."

2대기열 등록 (Enqueue)

FastAPI가 Redis Stream에 "이 사람의 데이터로 AI 분석해줘!" 라는 작업표를 등록합니다.
작업 내용은 Redis에 안전하게 저장되고, 사용자에게는 즉시 "처리 시작됨" 응답이 갑니다.
이 덕분에 사용자는 AI가 끝날 때까지 하얀 화면을 보며 기다리지 않아도 됩니다!

3AI 처리 (Processing)

항상 대기 중인 AI Worker가 Redis Stream을 감시하다가 새 작업표를 발견합니다.
작업표를 가져와 S3에 저장된 당뇨 예측 AI 모델을 로드합니다.
사용자가 입력한 건강 데이터(나이, 체중, 혈당 등)로 모델을 실행합니다.
이 과정은 몇 초~몇 분이 걸릴 수 있습니다.

4완료 확인 (Acknowledge)

AI Worker가 분석을 마치면 결과를 Database에 저장합니다.
그리고 Redis에 XACK 명령을 보내 "이 작업 완료했어요!" 라고 알립니다.
이 덕분에 시스템이 재시작되어도 이미 완료된 작업을 중복 처리하지 않아요.
FastAPI는 Redis의 Pub/Sub을 구독하고 있다가 완료 신호를 받습니다.

5결과 전송 (Notify via SSE)

FastAPI가 완료 신호를 받자마자 SSE(Server-Sent Events)를 통해 사용자 브라우저에 결과를 전송합니다.
사용자가 다른 페이지를 보거나 기다리는 동안에도 결과가 도착하면 자동으로 알림이 뜹니다.
"당뇨 위험도: 중간 수준 (42점) — 생활습관 개선이 필요합니다." 같은 결과가 화면에 표시됩니다.

📝 .env 파일에 있는 중요한 값들:

🐳 Docker로 서버 실행하기

MySQL, Redis, FastAPI, Nginx를 한 번에 실행합니다
Windows에서 경로에 한글이 있다면 반드시 이 우회법을 사용하세요!
Docker BuildKit이 한글 경로를 처리하지 못하는 버그가 있습니다.

① Windows PowerShell에서 (한글 경로 우회):

# X: 드라이브로 임시 경로 매핑 (한글 경로 우회) subst X: "C:\Users\사용자이름\Desktop\레퍼런스\마지막 웹프로젝트" # X: 드라이브로 이동 Set-Location X:\ # 핵심 4개 서비스만 실행 (ai-worker 제외) docker compose -p ai-health-local up -d mysql redis fastapi nginx

② Mac / Linux 또는 영어 경로라면:

docker compose up -d mysql redis fastapi nginx

③ 실행 상태 확인:

docker ps # fastapi, mysql, nginx, redis 4개가 보여야 합니다
처음 실행할 때는 Docker 이미지를 다운로드하므로 몇 분 걸릴 수 있어요. 기다려주세요!

🔧 문제 해결 — 포트 충돌

MySQL(3306) 또는 Redis(6379) 포트가 이미 사용 중이라면

Docker 컨테이너가 시작되지 않을 때 이미 실행 중인 프로세스를 찾아서 종료합니다.

Mac / Linux:

# 3306 포트를 사용 중인 프로세스 찾기 sudo lsof -i :3306 # PID(프로세스 번호)로 종료 kill -9 <PID번호>

Windows PowerShell:

# 3306 포트 사용 프로세스 확인 netstat -ano | findstr :3306 # PID로 종료 taskkill /PID <PID번호> /F
MySQL이나 Redis가 PC에 직접 설치되어 있다면, 서비스를 잠시 중지하고 Docker를 사용하세요.
