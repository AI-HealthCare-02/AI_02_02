# 앱 설명형 채팅 개발 안전 체크리스트

## 1) 요구사항 / 영향 범위 분석
- [x] 변경 목적과 성공 기준을 한 문장으로 정리했다.
- [x] 영향 범위(화면, 서버 액션, DB, 외부 연동)를 나열했다.
- [x] 현재 API / DB 계약과 충돌 여부를 확인했다.
- [x] 캐시, 권한, 성능, 동시성, 날짜/시간 처리 영향을 점검했다.

근거:
- 목적: 다나아 AI 채팅이 앱 기능 설명과 일부 현재 상태 질문에 대해 안전하게 답하도록 내부 `app_context`를 추가한다.
- 영향 범위: `backend/services/chat/*`, `backend/services/health_question.py`, `backend/services/challenge.py`, `backend/core/config.py`, `backend/core/sentry.py`, 관련 unit/integration tests.
- 비영향 범위: public API, 프론트, DB schema, migration.
- 캐시 주의: 동적 상태는 cached system prompt에 넣지 않고 request-scoped layer로만 붙인다.
- 날짜 기준: `datetime.now(tz=config.TIMEZONE).date()`로 오늘 기준 미기록 질문을 계산한다.

## 2) 데이터 안전 (중요)
- [x] 이번 변경의 SoT(Source of Truth)를 명시했다.
- [x] 상태 전이 규칙을 정의했다.
- [x] 멱등성 규칙을 정의했다.
- [x] 동시성 / race condition 위험 구간을 확인했다.
- [x] UTC / 사용자 로컬 날짜 처리 기준을 정했다.
- [x] 실패 시 복구 경로를 정리했다.

근거:
- SoT:
  - 리포트 상태: `HealthProfile`
  - 챌린지 상태: `ChallengeService.get_overview()`
  - 오늘 아직 안 적은 질문 수: `HealthQuestionService.get_daily_missing_summary()`
- 프론트 localStorage, `pendingHealthSummary`, `dashboard/init` 캐시 요약은 진실값으로 쓰지 않는다.
- 멱등성: `app_context`는 read-only request snapshot이다. 저장/갱신 부작용이 없다.
- 동시성: 동적 상태는 요청마다 새로 조립하고 cached prompt에 섞지 않아 stale cache 오염을 막는다.
- 실패 복구: `CHAT_APP_CONTEXT_MODE=off`로 즉시 롤백 가능.

- [x] 외부 API 한도와 실패 조건을 반영했다.
- [x] 동기 호출과 비동기 이벤트의 책임 경계를 정했다.
- [x] 재시도 / 중복 방지 / 로깅 전략을 정리했다.
- [x] 로컬 개발 환경과 운영 환경의 차이를 기록했다.

근거:

- `ChallengeService.get_overview()` 실패나 상태 조립 실패는 경고 로그 후 기존 채팅 fallback으로 흘린다.
- 로깅은 raw prompt/raw snapshot 대신 메타데이터만 남긴다.
- 로컬 테스트에서는 Docker/Postgres가 떠 있지 않으면 integration test가 막힐 수 있다.
