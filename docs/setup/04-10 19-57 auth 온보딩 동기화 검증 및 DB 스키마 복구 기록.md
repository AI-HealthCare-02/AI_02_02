# auth 온보딩 동기화 검증 및 DB 스키마 복구 기록

## 한 줄 요약

최신 `origin/main` 기준으로 이메일 인증 회원가입, 로그인, 온보딩 저장, 채팅 전송까지 실제 브라우저에서 확인했고, 그 과정에서 발견된 DB migration 누락과 프론트 채팅 인증 문제를 별도 검증 브랜치에서 수정했다.

## 왜 바로 kwanju 브랜치에 pull하지 않았나

현재 `kwanju` 브랜치는 기존 작업이 많이 남아 있는 dirty 상태다. 이 상태에서 바로 `main`을 pull하면 충돌이 커지거나, 기존 LangGraph/TTFT 작업과 auth/onboarding 변경이 섞일 수 있다.

그래서 최신 `origin/main`은 별도 clean worktree에서 먼저 확인했다.

- 검증 worktree: `마지막 웹프로젝트-main-check`
- 검증 브랜치: `fix/auth-onboarding-migration-format-check`
- 기준 커밋: `origin/main` 최신 merge commit

## 발견한 문제

### 1. Aerich migration old format

`aerich heads` 실행 시 old format migration 오류가 발생했다. migration 1, 2, 3에 현재 Aerich가 요구하는 `MODELS_STATE` 정보가 없어서 실제 DB migration 흐름이 끊겼다.

### 2. 이메일 인증 관련 schema migration 누락

현재 모델에는 아래 구조가 필요하다.

- `users.email_verified`
- `users.email_verified_at`
- `email_signup_sessions` 테이블

하지만 최신 main의 migration 흐름에는 이 schema를 생성하는 migration이 없었다.

### 3. 건강 리포트 저장 중 중복 update field 오류

`health_daily.py`에서 여러 입력 필드가 같은 source 컬럼을 업데이트할 수 있어, PostgreSQL에서 같은 컬럼을 한 UPDATE에 두 번 지정하는 오류가 발생할 수 있었다.

### 4. 프론트 채팅 요청에 로그인 token이 붙지 않음

로그인은 성공해도 채팅 페이지는 `NEXT_PUBLIC_AUTH_TOKEN`만 보고 있었다. 실제 로그인 token은 `localStorage`에 저장되므로, 브라우저에서 `/api/v1/chat/send`가 401이 날 수 있었다.

## 적용한 수정

### Migration 정리

- migration 1, 2, 3에 `MODELS_STATE`를 보강했다.
- 신규 migration 4를 추가했다.
- migration 4는 `email_signup_sessions` 테이블과 `users.email_verified`, `users.email_verified_at` 컬럼을 추가한다.
- 수동 `ALTER TABLE`이나 DB reset은 하지 않았다.

### Backend 수정

- 회원가입 성공 메시지를 기존 integration contract에 맞춰 한국어 응답으로 복구했다.
- `health_daily.py`의 `update_fields` 중복을 제거해 PostgreSQL 중복 컬럼 update 오류를 막았다.

### Frontend 수정

- 채팅 요청에서 `getToken()`으로 로그인 token을 읽어 Authorization header에 붙이도록 수정했다.
- cookie 기반 흐름과도 맞도록 `credentials: "include"`를 추가했다.

## 실제 DB 적용

로컬 `ai_health` DB는 backup 후 migration을 적용했다.

- 백업 위치: repo 밖 `Desktop/db-backups`
- 적용 방식: `uv run --group app aerich upgrade`
- 수동 SQL 적용 없음
- DB reset 없음

적용 후 확인한 항목:

- `aerich` history에 migration 0, 1, 2, 3, 4 존재
- `users` 테이블에 `provider`, `provider_user_id`, `onboarding_completed`, `onboarding_completed_at`, `email_verified`, `email_verified_at` 존재
- `email_signup_sessions` 테이블 존재
- `aerich heads` 결과: pending migration 없음

## 남은 주의점

채팅 기능 자체는 정상 동작했지만, 채팅 화면 상단 안내 문구가 온보딩 이전 문구처럼 남아 있었다. 실제 API 기준으로는 온보딩이 완료되었고 사이드바에도 사용자 그룹이 보였기 때문에, 이는 백엔드 저장 실패가 아니라 프론트의 localStorage 기반 온보딩 안내 상태 동기화 이슈로 보인다.

이 문제는 채팅 실패와는 별도 UX 정리 항목으로 분리하는 것이 안전하다.

## PR 전략

이번 수정은 `kwanju`의 기존 LangGraph/TTFT 작업과 섞으면 안 된다.

권장 순서:

1. 최신 main 기반 clean PR로 migration/auth/onboarding/chat 인증 수정만 올린다.
2. main에 merge된 뒤 `kwanju`에서 main을 pull/merge한다.
3. LangGraph off pivot, TTFT 기록, DB 복구 기록은 성격별로 별도 PR로 분리한다.

## 팀원에게 설명할 핵심 문장

최신 main을 그대로 브라우저에서 테스트해보니 기능 코드는 거의 맞았지만, 실제 DB migration 흐름과 프론트 채팅 인증 연결이 덜 맞아 있었다. 그래서 main을 `kwanju`에 바로 섞지 않고 clean worktree에서 schema migration과 채팅 token 문제를 먼저 고쳤고, 이메일 인증부터 실제 채팅 저장까지 브라우저에서 확인했다.
