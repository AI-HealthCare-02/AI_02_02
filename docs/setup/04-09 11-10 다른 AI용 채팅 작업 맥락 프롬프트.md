# 다른 AI용 채팅 작업 맥락 프롬프트

## 문서 목적
- 이 문서는 다른 AI가 다나아 채팅 구조를 빠르게 이해하고, 어제 작업과 오늘 작업의 연결 맥락까지 한 번에 이어받을 수 있게 만드는 인계 문서다.
- 특히 `왜 LangGraph를 전체 전환이 아니라 부분 적용했는지`, `적용 전후에 어떤 구조 차이가 생겼는지`, `다음 작업자가 어디를 먼저 봐야 하는지`를 설명하는 데 목적이 있다.

## 한 줄 요약
- 어제는 `reason_codes enum 전환 + normalize option C + chat 패키지 분리`로 내부 구조를 먼저 정리했다.
- 오늘은 그 위에 `LangGraph를 pre-LLM preparation 구간에만 부분 적용`해서, safety-critical path는 그대로 두고 `RAG / UserContext / prompt layer 조립`만 그래프로 분리했다.

## 다른 AI에게 바로 붙여넣을 프롬프트
```md
아래 맥락을 전제로 다나아 채팅 코드를 분석해줘.

[프로젝트 목표]
- 의료 안전이 중요한 채팅 서비스다.
- 이번 리팩터링의 핵심은 "설명 가능한 구조"와 "회귀를 통제한 구조 개선"이다.
- 전체 채팅 엔진을 LangGraph로 바꾸는 것이 아니라, prep 단계만 부분 적용하는 방향으로 구현했다.

[어제 완료된 작업]
1. content_filter의 reason code를 문자열에서 enum으로 바꿨다.
2. _normalize_text()에 option C를 반영했다.
   - zero-width 제거
   - NFKC 정규화
   - 숫자/범위/단위 보호
   - 우회용 구두점 정리
3. backend/services/chat.py 단일 파일 구조를 패키지 구조로 분리했다.
   - service.py
   - streaming.py
   - persistence.py
   - prompting.py
   - enrich.py
4. branch parity/SSE 안전망과 normalize golden set 문서를 정리했다.

[오늘 완료된 작업]
1. LangGraph를 backend/services/chat_graph/ 내부에 새로 추가했다.
2. graph boundary는 prep 단계에만 두었다.
   - graph 밖: validation, content_filter, crisis/block/consent early return, cooldown, DB write, SSE streaming
   - graph 안: decide_rag, run_rag, decide_user_context, build_user_context, assemble_prompt_layers
3. /chat/send의 실제 응답 경로는 유지하고, prep 단계에서만 adapter seam을 통해 graph를 호출한다.
4. rollout 모드는 off/shadow/partial 이다.
5. timeout, graph exception, invalid output 시 legacy fallback 하도록 만들었다.
6. parity/logging/mode-switch/prompt-layer 테스트와 bench 스크립트를 추가했다.

[중요 판단]
- LangGraph는 전체 전환이 아니라 "설명 가능한 LLM 준비 그래프"로 제한 적용했다.
- safety-critical path를 graph 안에 넣지 않은 이유는 의료 안전, 조기 종료 우선순위, DB write semantics, SSE contract를 유지하기 위해서다.

[당장 먼저 볼 파일]
- backend/services/chat/service.py
- backend/services/chat/prompting.py
- backend/services/chat/enrich.py
- backend/services/chat_graph/adapter.py
- backend/services/chat_graph/graph.py
- backend/services/chat_graph/nodes.py
- backend/services/chat_graph/state.py
- backend/tests/unit/test_chat_langgraph_parity.py
- backend/tests/unit/test_chat_langgraph_prompt_layers.py
- backend/tests/unit/test_chat_langgraph_mode_switch.py
- backend/tests/unit/test_chat_langgraph_logging.py
- docs/setup/04-08 14-19 채팅 3트랙 구현 기록.md
- docs/setup/04-09 10-50 LangGraph 부분 적용 결정 기록.md
- docs/setup/04-09 10-50 채팅 흐름 전후 비교.md

[원하는 분석]
- 현재 LangGraph boundary가 적절한지
- legacy prep와 graph prep 사이에 parity가 깨질 가능성이 있는지
- 성능/회귀 위험이 남아 있는 지점을 우선순위 순으로 정리해줘
- 다음 단계로 full rollout 전에 무엇을 더 잠가야 하는지 제안해줘

[주의]
- content_filter, crisis/block/consent, cooldown, DB write, _stream_openai, _sse_event는 graph 밖에 남아 있어야 한다.
- API surface는 유지되어야 한다.
- test surface 변경은 shim을 통해 최소화했다.
- 로컬에서는 unit 테스트는 통과했지만 integration의 일부는 postgres 테스트 호스트 부재로 미검증 상태다.
```

## 어제 작업 정리
### 1. content_filter 구조 정리
- `reason_codes`를 문자열 기반에서 enum 기반으로 바꿨다.
- 목적은 오타를 줄이고, 분기 membership 검사에서 타입 안정성을 높이는 것이다.
- `set` 전환은 하지 않았다.
- 현재 판단은 `list[FilterReasonCode]`가 유연성과 디버깅 측면에서 더 낫다는 쪽이다.

### 2. normalize option C 반영
- `_normalize_text()`를 강화했다.
- 핵심은 `무조건 공격적으로 정리`가 아니라 `의료 수치/범위/단위는 보호하고, 우회 표현만 정리`하는 것이다.
- golden snapshot과 characterization 테스트를 추가해서 normalize 변경을 회귀 없이 관리하도록 했다.

### 3. chat.py 패키지 전환
- 기존 `backend/services/chat.py` 단일 파일 구조를 역할별 패키지 구조로 나눴다.
- 분리 결과:
  - `backend/services/chat/service.py`
  - `backend/services/chat/streaming.py`
  - `backend/services/chat/persistence.py`
  - `backend/services/chat/prompting.py`
  - `backend/services/chat/enrich.py`
- 외부 surface는 유지했다.
  - `backend.services.chat.ChatService`
  - `backend.services.chat._sse_event`
  - `backend.services.chat.ROUTE_INSTRUCTIONS`
  - `backend.services.chat.EMOTIONAL_INSTRUCTION`

### 4. 어제 기준 핵심 의미
- 어제 작업의 본질은 `LangGraph를 넣기 위한 준비`가 아니라, `LangGraph를 넣지 않아도 구조를 설명할 수 있게 내부를 정리한 것`이다.
- 즉, safety-critical path를 먼저 정리하고 테스트를 잠근 뒤에야 LangGraph 부분 적용을 검토할 수 있는 상태가 됐다.

## 오늘 작업 정리
### 1. LangGraph 적용 범위
- 오늘은 전체 채팅 엔진을 그래프로 바꾸지 않았다.
- `/chat/send` 안에서 `pre-LLM preparation`만 그래프로 분리했다.
- graph 밖:
  - request validation
  - chat access / consent
  - `content_filter.check_message()`
  - `CRISIS_ESCALATE`, `BLOCK`, `ai_consent_required`
  - cooldown, `_last_crisis_at`
  - session/message DB write
  - `_stream_openai`, `_sse_event`, done payload
- graph 안:
  - `decide_rag`
  - `run_rag`
  - `decide_user_context`
  - `build_user_context`
  - `assemble_prompt_layers`

### 2. 구현 포인트
- 새 패키지 `backend/services/chat_graph/`를 추가했다.
- `service.py`는 그대로 오케스트레이터 역할을 유지한다.
- prep 단계에서만 `chat_graph.adapter.prepare_openai_messages(...)`를 호출한다.
- graph 결과가 깨지거나 느리거나 실패하면 legacy prep으로 즉시 fallback 하도록 만들었다.

### 3. rollout 모드
- `off`
  - legacy prep만 사용
- `shadow`
  - 사용자 응답은 legacy prep만 사용
  - graph는 parity/latency 확인용으로만 함께 돌린다
- `partial`
  - sticky bucket으로 일부 사용자만 graph prep 결과를 사용
  - 실패 시 요청 단위로 legacy fallback

### 4. 오늘 기준 핵심 의미
- 오늘 작업의 본질은 `LangGraph 도입` 자체보다 `LangGraph를 안전하게 실서비스 경로 안에 넣는 운영 구조`를 만든 것이다.
- 즉, graph보다 중요한 건 `boundary`, `rollout`, `fallback`, `parity logging`, `performance budget`이다.

## LangGraph 적용 전과 적용 후 비교
| 구분 | 적용 전 | 적용 후 |
| --- | --- | --- |
| prep 구조 | `service.py`가 legacy helper를 직접 호출 | `service.py`가 adapter seam을 통해 legacy/graph prep 선택 |
| enrichment 판단 | `enrich.py` helper가 service 흐름 안에서 직접 실행 | graph node가 projection 기반 helper를 사용해 실행 |
| prompt layer 조립 | `prompting.py`에서 legacy path만 조립 | 같은 `prompting.py`를 graph와 legacy가 공유 |
| 안전 경계 | 코드상으로는 유지되지만 문서/운영 관점이 약함 | graph 밖/안 경계를 명시적으로 분리 |
| rollout | 없음 | `off/shadow/partial` |
| fallback | helper 단위 예외 처리 위주 | graph timeout/exception/invalid output 시 legacy fallback |
| 운영 추적 | 기본 로그 중심 | parity 로그, latency 비교, sticky bucket |
| 포트폴리오 설명력 | "리팩터링했다" 수준 | "safety-critical path를 유지한 채 prep graph만 분리했다" 수준 |

## 적용 전후 흐름 비교
### 적용 전
1. `service.py`가 검증, 필터, early return, 저장, enrichment, prompt 조립, streaming까지 순차 실행
2. `enrich.py`와 `prompting.py` helper를 직접 호출
3. prep 결과는 legacy helper 경로 하나만 존재

### 적용 후
1. 검증, 필터, early return, cooldown, DB write, streaming은 기존 셸에 그대로 남음
2. prep 단계에서만 adapter가 작동
3. adapter가 모드에 따라 legacy 또는 LangGraph prep을 선택
4. shadow/partial 모드에서는 parity와 latency를 함께 기록
5. 이상 징후가 있으면 legacy path로 자동 fallback

## 다른 AI가 특히 주의해야 할 지점
- `content_filter`, crisis/block/consent, cooldown, DB write, SSE는 graph 안으로 넣으면 안 된다.
- LangGraph는 지금 `설명 가능한 prep graph`이지 `전체 chat state machine`이 아니다.
- `prompting.py`는 legacy와 graph가 공유하는 소스오브트루스다.
- `adapter.py`가 사실상 운영 정책의 핵심이다.
  - mode switch
  - sticky bucketing
  - timeout fallback
  - parity logging
  - shadow audit

## 현재 검증 상태
- unit 테스트는 통과했다.
- 새 LangGraph 관련 unit 테스트도 추가되어 있다.
- bench 스크립트도 추가되어 있다.
- 다만 integration의 일부는 로컬 `postgres:5432` 테스트 호스트 부재로 아직 완전 검증되지 않았다.

## 잔여 리스크
- partial rollout을 실제로 켜기 전에는 shadow parity 로그와 latency budget을 더 확인해야 한다.
- graph boundary가 넓어지면 safety-critical path까지 침범할 위험이 있다.
- adapter의 fallback/timeout 정책이 느슨해지면 사용자 체감 first-token latency가 악화될 수 있다.

## 최종 정리
- 어제 작업은 `구조 정리와 안정성 확보`
- 오늘 작업은 `LangGraph를 prep 구간에만 안전하게 끼워 넣는 구조 설계와 구현`
- 따라서 현재 다나아 채팅 구조는 `전면 그래프화`가 아니라 `명시적 셸 + prep graph` 모델로 이해하는 것이 가장 정확하다.
