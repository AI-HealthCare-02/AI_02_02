  - 성능: 오른쪽 패널 직접 입력은 짧은 debounce 필요
  - 날짜: `health/daily/{today}`와 `health/weekly`는 서버 기준 오늘 날짜 사용

## 2) 데이터 안전 (중요)
- [x] 이번 변경의 SoT(Source of Truth)를 명시했다.
- [x] 상태 전이 규칙을 정의했다.
- [x] 멱등성 규칙을 정의했다.
- [x] 동시성 / race condition 위험 구간을 확인했다.
- [x] UTC / 사용자 로컬 날짜 처리 기준을 정했다.
- [x] 실패 시 복구 경로를 정리했다.

근거:
- SoT:
  - 오늘 건강 기록: 서버 `DailyHealthLog`
  - 상세 리포트 열림/잠금 판단: `GET /api/v1/health/weekly`
  - 활성 채팅 세션: URL `session_id`
- 상태 전이:
  - 오른쪽 패널 편집 -> PATCH 요청 -> 서버 응답의 `daily_log`로 화면 재동기화
  - `skipped(already_answered)`면 사용자가 누른 값이 아니라 서버 값 유지
  - `새 대화` 클릭 시에만 `session_id` 제거
- 멱등성:
  - 같은 필드 재저장은 기존 `First Answer Wins` 규칙으로 서버가 거절 가능
  - 프론트는 매번 서버 응답으로 상태를 다시 맞춤
- race 구간:
  - 빠른 연속 클릭 시 PATCH 경쟁 가능 -> debounce + 최신 요청 기준 반영 필요
  - 세션 복원 중 다른 탭 이동 -> invalid session이면 빈 채팅으로 fail closed
- 날짜 처리:
  - 프론트는 `todayKey()`를 UI 캐시용으로만 사용
  - 실제 저장/판단은 서버 API의 오늘 날짜 기준
- 실패 복구:
  - 저장 실패 시 마지막 서버 성공값으로 롤백
  - guide/세션 오류 시 빈 화면으로 fail closed, 다른 세션으로 몰래 이동 금지

## 4) 외부 연동 / 운영 경계
- [x] 외부 API 한도와 실패 조건을 반영했다.
- [x] 동기 호출과 비동기 이벤트의 책임 경계를 정했다.
- [x] 재시도 / 중복 방지 / 로깅 전략을 정리했다.
- [x] 로컬 개발 환경과 운영 환경의 차이를 기록했다.

근거:
- 외부 API 없음
- 동기/비동기 경계:
  - 오른쪽 패널 저장은 동기 PATCH
  - 대화 스트리밍(SSE)은 기존 chat send 흐름 유지
  - 사이드바 세션 선택은 URL/query + custom event 유지
- 재시도:
  - 패치 실패 시 자동 재시도는 넣지 않고 사용자 재입력/재클릭에 맡김
  - 중복 저장은 서버 `First Answer Wins`로 방어
- 로깅:
  - raw app state/log 본문 노출 금지
  - coarse error only
- 환경 차이:
  - Docker 백엔드 + 로컬 Next 개발 서버 조합
  - 날짜와 auth는 서버 기준
