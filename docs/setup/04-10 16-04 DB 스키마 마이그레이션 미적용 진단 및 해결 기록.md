# DB 스키마 마이그레이션 미적용 진단 및 해결 기록

## 1. 한 줄 요약

브라우저 `/app/chat` 채팅 실패의 직접 원인은 서버 장애가 아니라, 실제 실행 DB인 `ai_health.user_settings`에 `chat_notification` 컬럼이 빠져 있던 DB 스키마 mismatch였다. DB 백업 후 Aerich migration 1을 적용했고, 브라우저에서 실제 AI 답변 출력과 DB 저장까지 확인했다.

## 2. 문제 상황

- 브라우저 `/app/chat`에서 메시지 전송 시 AI 답변이 실패 메시지로 표시됐다.
- 백엔드 로그에는 `chat_notification` 컬럼을 찾지 못하는 DB 오류가 남아 있었다.
- 테스트 DB에서는 문제가 드러나지 않았지만, 실제 브라우저가 연결한 `ai_health` DB에는 migration 1이 적용되어 있지 않았다.

## 3. 원인 분석

- `ai_health.user_settings` 테이블은 존재했다.
- `chat_notification` 컬럼은 없었다.
- `aerich` history에는 `0_20260403200603_init.py`만 있었다.
- repo에는 `1_20260407115500_add_chat_notification_to_user_settings.py` migration 파일이 이미 있었다.
- 결론: 코드 문제가 아니라, 로컬 실제 DB에 `aerich upgrade`가 적용되지 않은 상태였다.

## 4. 안전 조치

- DB reset은 하지 않았다.
- 수동 `ALTER TABLE`은 하지 않았다.
- 먼저 DB 백업을 만들었다.

백업 파일:

```text
C:\Users\mal03\Desktop\db-backups\ai_health_before_chat_notification.dump
```

백업 크기:

```text
75014 bytes
```

## 5. 진행 중 추가로 발견한 문제

처음 `uv run aerich upgrade`를 실행했을 때 아래 오류가 발생했다.

```text
RuntimeError: Old format of migration file detected, run aerich fix-migrations to upgrade format
```

`aerich fix-migrations`도 한 번에 해결하지 못했다.

- 첫 실행은 Windows 콘솔 `cp949` 인코딩 문제로 실패했다.
- `PYTHONIOENCODING=utf-8`을 설정한 뒤에는 실행 자체는 성공했다.
- 하지만 migration 1은 아직 DB history에 없는 미적용 migration이라 Aerich가 `No matching record ... Skipping`으로 건너뛰었다.

그래서 migration SQL은 그대로 두고, 최신 Aerich가 요구하는 `MODELS_STATE` 메타데이터만 migration 1 파일에 추가했다.

수정 파일:

```text
backend/db/migrations/models/1_20260407115500_add_chat_notification_to_user_settings.py
```

## 6. 해결

로컬 터미널에서 실제 DB 주소를 명시하고 migration을 적용했다.

```powershell
$env:DB_HOST='localhost'
$env:DB_NAME='ai_health'
$env:PYTHONIOENCODING='utf-8'
uv run aerich upgrade
```

결과:

```text
Success upgrading to 1_20260407115500_add_chat_notification_to_user_settings.py
```

## 8. 서버 실행 방식

기존 Docker `fastapi` 컨테이너는 현재 repo 구조와 맞지 않는 `app.main` 기준으로 떠 있어 `/api/docs` 확인에 실패했다.

그래서 브라우저 검증은 현재 repo 기준 로컬 uvicorn으로 진행했다.

```powershell
$env:DB_HOST='localhost'
$env:DB_NAME='ai_health'
$env:REDIS_HOST='localhost'
$env:PYTHONIOENCODING='utf-8'
uv run uvicorn backend.main:app --host 127.0.0.1 --port 8000 --reload
```

API docs 확인:

```text
GET http://127.0.0.1:8000/api/docs -> 200 OK
```

프론트는 토큰이 없는 상태에서는 `/api/v1/chat/send`가 `401 Unauthorized`로 실패했다. 이 실패는 DB 문제가 아니라 인증 토큰 문제였다.

## 13. 재발 방지

- migration 파일을 만든 뒤에는 테스트 DB뿐 아니라 실제 로컬 실행 DB에도 `aerich upgrade`를 적용했는지 확인한다.
- 브라우저 테스트 전에는 “백엔드가 어떤 DB를 보는지”를 `DB_HOST`, `DB_NAME` 기준으로 확인한다.
- Docker 내부 실행이면 `DB_HOST=postgres`, Windows 로컬 실행이면 `DB_HOST=localhost`를 써야 한다.
- 프론트 채팅 테스트는 인증 토큰이 없으면 401이 나므로, 실제 채팅 검증 전 토큰 주입 여부를 확인한다.

## 14. 포트폴리오 포인트

1. 모든 테스트가 통과해도 브라우저 실제 실행 DB에서는 schema mismatch가 날 수 있다는 점을 발견했다.
2. 에러 로그, ORM 모델, 실제 DB 컬럼, Aerich migration history 순서로 원인을 좁혔다.
3. 수동 SQL이나 DB reset 없이, 백업 후 migration 경로로 복구했다.
4. 마지막에는 브라우저, Network, Console, DB 저장까지 end-to-end로 검증했다.
