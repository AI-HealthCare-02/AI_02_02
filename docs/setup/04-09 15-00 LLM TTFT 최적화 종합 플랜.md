> - **Round 2**: 3개 검토 에이전트 피드백 반영 (보수 안전·성능·아키텍처 관점)

## Context (배경)

**왜 이 작업이 필요한가**
- 현재 다나아 채팅 서비스는 gpt-4o-mini + SSE 스트리밍으로 동작 중이지만, **첫 글자까지의 시간(TTFT)** 이 사용자 체감 속도의 핵심이며 이를 개선할 여지가 많다.
- LangGraph 부분 적용(Day 2) 이후 prep 구간이 구조적으로 분리됐지만, 실제 지연 병목은 prep 그래프 구조 바깥에도 다수 존재함(클라이언트 싱글톤, DB 순차, 프롬프트 구조, 프론트 SSE 파싱 등).
- 멘토링 회의에서 "기능보다 구조·설명 가능성"이 강조됐고, 이번 최적화는 "어디가 왜 느린가 → 어떻게 고쳤는가 → 얼마나 빨라졌는가"를 수치로 보여주는 포트폴리오 포인트가 될 수 있다.

**사용자 확정 요구사항 (AskUserQuestion 응답 기반)**
1. **범위**: 백엔드 prep + 프론트엔드 SSE 수신 (nginx 프록시/배포 영역은 제외)
2. **허용 수단**: **코드/알고리즘만**. 외부 서비스(Redis/Vector DB) 추가 금지, 모델 변경 금지, 로컬 LLM 도입 제외
3. **우선순위**: 속도 > 안정성 > 비용
4. **목표 지표**: **TTFT (첫 글자까지 시간) P95 단축**. 보조 지표로 prep 처리 시간, end-to-end, 토큰 사용량 추적

**제약**
- `backend/services/chat/*`, `backend/services/chat_graph/*`, `backend/services/rag.py`, `backend/services/user_context.py`, `frontend/app/app/chat/page.js`, `frontend/app/layout.js`, `next.config.mjs`, `pyproject.toml` 이 주요 변경 대상
- safety-critical path(content_filter, crisis/block/consent early return, DB write 계약, SSE contract)는 **의미론 변경 금지** — 순서·타이밍만 최적화 가능
- LangGraph parity 계약 유지 — 토폴로지 변경 시 `_GRAPH_VERSION` bump 필수

## 개선 플랜 (단계별)

### Phase 1 — Quick Wins (1~2일, 즉시 효과 큰 것부터)

**목표**: TTFT 150~500ms 단축 + 향후 측정을 위한 관측 기반 마련

**P1.1. AsyncOpenAI 싱글톤화** ⭐ 최우선
- 파일: `backend/services/chat/streaming.py`, `workers/ai/tasks/chat.py`
- 작업: 모듈 레벨 `_CLIENT` 싱글톤 생성, `httpx.AsyncClient(limits=Limits(max_connections=100, max_keepalive_connections=50, keepalive_expiry=30), timeout=httpx.Timeout(connect=2.0, read=30.0, write=5.0, pool=1.0), http2=True)`를 `AsyncOpenAI(http_client=...)`에 주입
- 공용 모듈 `backend/core/openai_client.py` 신설, FastAPI startup에서 warmup (빈 모델 목록 조회 1회)
- FastAPI shutdown에서 `aclose()` 처리
- 예상 단축: 100~300ms (cold) · keepalive로 지속 이득

**P1.2. next.config.mjs rewrites + 프런트 상대 경로 전환**
- 파일: `frontend/next.config.mjs`, `frontend/app/app/chat/page.js`
- 작업: `rewrites()` 추가해 `/api/*` → `http://localhost:8000/api/*` 프록시, `NEXT_PUBLIC_API_BASE` 의존 제거 → 상대 경로 `/api/v1/chat/send` 호출
- CORS preflight 제거 (동일 origin 간주) — 단, Next dev server는 `async rewrites` 후에도 streaming pass-through를 검증해야 함 (development/production 모두)
- 예상 단축: 20~200ms (preflight 1RTT 제거)

**P1.3. 폰트 로딩 non-blocking 전환**
- 파일: `frontend/app/layout.js`
- 작업: Pretendard → `next/font/local` (또는 `next/font/google` if available), `display: swap`, `preload: true`. FontAwesome → tree-shake 또는 chat 페이지에서 쓰는 아이콘만 SVG inline
- `<link rel="preconnect">`는 제거되고 fonts가 self-host됨
- 예상 단축: 50~300ms (첫 페인트 blocking 제거)

**P1.4. timeout + stream_options 명시**
- 파일: `backend/services/chat/streaming.py`
- 작업: `stream=True`에 `stream_options={"include_usage": True}` 추가, 클라이언트 생성 시 timeout 명시(P1.1에 포함), 마지막 chunk의 usage를 로깅해 prompt_tokens 추적
- 예상 단축: 0ms (측정 기반 마련) + p99 개선

**P1.5. history 10 → 6 turn + 방향 수정**
- 파일: `backend/services/chat/service.py:71` (상수), `service.py:166` (쿼리)
- 작업: `MAX_HISTORY_MESSAGES = 6`, `order_by("-created_at").limit(6)` 후 `reversed()` 적용 (최신 6건을 시간순으로)
- parity 영향 확인 필수 (LangGraph adapter의 history_snapshots 해시 변경)
- 예상 단축: 30~80ms (400 입력 토큰 감소)

**P1.6. 지연 import → 최상단 이동**
- 파일: `backend/services/chat/prompting.py:85-94`
- 작업: `from backend.services.content_filter_patterns import WARN_PROMPT_INSTRUCTION`을 모듈 최상단으로 이동. 순환 import 여부 확인 후 안전하면 적용
- 예상 단축: 1~5ms (warm), cold 요청 30ms

**P1.7. history 스냅샷 ORM → dict 전환**
- 파일: `backend/services/chat/service.py:166`, `chat_graph/adapter.py:81-84`
- 작업: `ChatMessage.filter(...).values("role", "content")` 로 dict list 받음, `_history_snapshots` 제거 또는 dict-to-dict로 경량화
- 예상 단축: 5~15ms

**Phase 1 소계**: TTFT 200~700ms 단축 예상, 모든 작업이 Low 난이도 (P1.3 제외), 1~2일 내 완료 가능

### Phase 2 — Parallelization (2~3일, 병렬화)

**목표**: prep 단계 순차 실행을 병렬화로 30~100ms 추가 단축

**P2.1. prep 단계 asyncio.gather 병렬화**
- 파일: `backend/services/chat/service.py:117-166`
- 작업:
  - **그룹 A (진입 검증)**: `_validate_request`의 일일 카운트 COUNT 쿼리 + `_validate_chat_access`의 HealthProfile 조회 → `asyncio.gather`
  - **그룹 B (prep 전 로딩)**: USER 메시지 저장 이후 `get_eligible_bundles` + history 조회 + system_prompt 생성(순수 함수라면 제외) → `asyncio.gather`
  - content_filter는 동기이므로 gather 밖 유지 (CPU 바운드)
- 예상 단축: 20~75ms

**P2.2. USER 메시지 fire-and-forget 저장**
- 파일: `backend/services/chat/service.py:156`
- 작업: `asyncio.create_task(ChatMessage.create(session=, role=USER, content=))` 로 저장을 스트림 시작 경로에서 분리, history는 DB에서 9건만 가져와 in-memory로 현재 메시지 append. task는 `_save_response`와 함께 `gather`로 수거하거나 background에서 자연 완료
- **에러 처리**: task 예외는 logger.exception 후 사용자에게 조용히 영향 없음 (조회 시점은 이후 요청이라 UX 영향 없음). shutdown 시 pending task 확인
- 예상 단축: 10~30ms

**P2.3. LangGraph run_rag + build_user_context 병렬화**
- 파일: `backend/services/chat_graph/graph.py`, `chat_graph/nodes.py`
- 작업:
  - `decide_rag` + `decide_user_context`는 순수 동기 판단 → 한 `decide_both` 노드로 통합
  - `run_rag` + `build_user_context`는 서로 독립 → LangGraph의 `add_node` + fan-out 엣지로 병렬 실행 (또는 한 노드 안에서 `asyncio.gather`)
  - `assemble_prompt_layers`는 두 결과를 받아 조립
- **parity 영향**: 노드 토폴로지 변경 → `_GRAPH_VERSION = "chat_prep_graph_v2"` bump + shadow parity 로그로 검증
- 예상 단축: 15~50ms (RAG 검색과 UserContext 빌드가 둘 다 활성일 때)

**Phase 2 소계**: TTFT 45~155ms 추가 단축, 의존성·parity 관리 필요 (Medium 난이도)

### Phase 3 — Algorithmic Optimization (3~5일)

**목표**: 알고리즘 개선으로 50~200ms 추가 단축 + 장기적 prompt caching 이득

**P3.1. RAG inverted index + asyncio.to_thread**
- 파일: `backend/services/rag.py`, `backend/services/rag_corpus.py`
- 작업:
  - startup 시 inverted index 구축: `token → [(snippet_idx, tf)]`, `df[token]`, `idf_cache`
  - `search(query)`는 쿼리 토큰별 posting list union만 스캔 → O(|posting|)
  - search 호출부를 `await asyncio.to_thread(rag_service.search, ...)` 로 래핑해 이벤트 루프 블로킹 방지
  - `_should_run_rag` 게이팅 공격적 강화 (짧은 대화·감정 우선 skip은 이미 있음, 추가로 쿼리 토큰 길이 기준)
- 예상 단축: 15~80ms + 이벤트 루프 stall 제거 (전체 요청 영향)

**P3.2. Prompt caching 구조화 — system prefix 고정**
- 파일: `backend/services/chat/prompting.py`, `chat_graph/nodes.py::assemble_prompt_layers`
- 작업:
  - 현재: base + user_context + route + rag + filter instruction을 **system 메시지 하나**로 합침
  - 변경: **system 메시지는 base_system_prompt + 고정 routing 표**만 포함(1024+ 토큰 달성 목표 — 필요 시 고정 예시 확장), **가변 부분(user_context, rag_context, filter_instruction)은 별도 user 메시지로 history 직전에 삽입**
  - OpenAI는 prefix 1024 토큰 이상이면 prompt caching 자동 활성 → 동일 system prefix는 prefill 시간 ~50% 단축
  - **A/B 필수**: 응답 품질 회귀 여부 shadow 비교 (parity 필드 확장)
- 예상 단축: 100~300ms (캐시 hit 시 반복 요청)
- 리스크: HIGH — 프롬프트 레이아웃 변경은 응답 품질 영향 가능

**P3.3. LangGraph audit 재실행 제거**
- 파일: `backend/services/chat_graph/adapter.py:281-290`
- 작업: `_audit` 함수에서 `_run_legacy_prep`를 "또" 호출하는 부분 제거. shadow/partial 모드에서 parity 비교는 이미 생성된 `legacy_output`/`graph_output`만 사용. 별도 legacy 재실행 없음
- 예상 단축: 50~150ms (샘플링된 요청) · 이벤트 루프 경쟁 완화로 TTFT 간접 개선
- 리스크: LOW — audit 로직만 간소화

**P3.4. 프롬프트 조립 dict 복사·해시 경량화**
- 파일: `backend/services/chat/prompting.py:141-167`, `chat_graph/adapter.py:65-67`
- 작업:
  - `openai_messages`를 처음부터 `list[dict]`로 생성 (tuple→dict 변환 제거)
  - `_sha256_messages`는 TTFT 경로에서 제거, parity audit는 background에서만 계산
  - `_sha256_text(final_system_prompt)`만 hot path에 유지
- 예상 단축: 3~10ms

**P3.5. 면책조항 suffix 지시 완화**
- 파일: `backend/services/chat/prompting.py:37`
- 작업: 현재 "건강 질문에 답할 때 맨 끝에 자연스럽게 ... 포함" 지시를 후처리(응답 스트림 종료 후 string append)로 이동. 시스템 프롬프트에서는 "필요 시 한 줄 면책" 수준으로 완화
- A/B 필요 (응답 톤/품질)
- 예상 단축: 20~80ms (편차 큼)

**Phase 3 소계**: TTFT 188~620ms 추가 단축 (prompt caching이 가장 큰 레버)

### Phase 4 — Frontend Optimization (3~5일, 병렬 가능)

**목표**: SSE 수신·렌더링 최적화로 첫 글자 paint 품질 향상 + 토큰 지터 제거

**P4.1. parseSSE 스트리밍 파서로 교체**
- 파일: `frontend/app/app/chat/page.js:80-97, 136-139`
- 작업: buffer에서 `indexOf('\n\n')` 단위로 꺼내 쓰는 incremental 파서. 이미 파싱한 이벤트는 재파싱하지 않음
- 예상 효과: 토큰당 파싱 비용 O(N²) → O(N), 지터 제거

**P4.2. 스트리밍 텍스트를 useRef + 별도 state로 분리**
- 파일: `frontend/app/app/chat/page.js:143-150`
- 작업:
  - `messages` state는 완료된 메시지만 보관
  - 스트리밍 중인 텍스트는 `streamingText` state 또는 `useRef` + `forceUpdate` 조합
  - `<MessageList>` 는 `messages` 변경 시에만 재렌더, `<StreamingMessage>` 는 `streamingText` 변경 시에만
- 예상 효과: 토큰당 렌더링 비용 O(N) → O(1)

**P4.3. 컴포넌트 분할 + React.memo**
- 파일: `frontend/app/app/chat/page.js` 전체 리팩토링
- 작업: `ChatPage` → `Sidebar` + `RightPanel` + `MessageList` + `MessageBubble` + `StreamingMessage` + `Input` 컴포넌트로 분리. 각각 `React.memo`, props 안정화
- 예상 효과: 토큰마다 935줄 전체 리렌더 → 실제 변경된 영역만 리렌더

**P4.4. scroll throttle + smooth 제거**
- 파일: `frontend/app/app/chat/page.js:58-60`
- 작업: `scrollIntoView({behavior: 'auto'})` + throttle 100ms. 사용자가 위로 스크롤한 상태면 auto-scroll 비활성
- 예상 효과: 토큰 paint 지터 5~10ms 제거

**P4.5. 첫 토큰 flushSync + 이후 rAF flush**
- 파일: `frontend/app/app/chat/page.js`
- 작업: 첫 토큰은 `ReactDOM.flushSync(() => setStreamingText(token))` 로 즉시 페인트 보장. 이후 토큰들은 `requestAnimationFrame` 단위로 batched flush
- 예상 효과: 첫 글자 paint 타이밍 예측 가능성 확보

**Phase 4 소계**: 체감 품질 개선 + 토큰당 지터 제거 (TTFT 자체 영향은 작지만 UX 핵심)
