2 작업 타임라인 (이틀치)

Day 1에서 구조를 먼저 정리한 뒤, Day 2에서 그 위에 LangGraph를 얹는 순서로 진행했습니다.

📅 Day 1 (2026-04-08) — "집 구조 정리"
LangGraph를 넣기 전에 내부 경계부터 깨끗하게 정리
reason_codes를 문자열 → enum으로 전환
필터 사유 코드를 타입 안정성 있는 enum으로 교체해 오타·분기 실수 차단.
🧠 왜? 분기 membership 검사의 타입 안정성 확보
normalize() 함수 option C 반영
zero-width 제거, NFKC 정규화, 의료 수치·단위 보호, 우회용 구두점만 정리. golden snapshot 테스트로 회귀 방지.
🧠 왜? "공격적 정리"가 아니라 "의료 수치는 보호하면서 우회만 차단"
chat.py → 5개 파일로 패키지 분리
backend/services/chat/ 아래 service.py / streaming.py / persistence.py / prompting.py / enrich.py 로 역할 분리. 외부 API surface(ChatService 등)는 그대로 유지.
🧠 왜? 이후 LangGraph를 "어디에 얹을지" 명확히 정의하기 위한 전제
📅 Day 2 (2026-04-09) — "LangGraph 부분 적용"
정리된 구조 위에 준비 구간만 그래프로 분리
backend/services/chat_graph/ 패키지 신규 생성
state.py (상태 타입), nodes.py (5개 노드), graph.py (컴파일된 그래프), adapter.py (legacy/graph 경로 선택) 4개 파일.
🧠 왜? 기존 chat/ 패키지와 명확히 분리해 "새로 추가된 그래프 경로"를 한눈에 볼 수 있게
ChatPreparationGraph 5개 노드 설계
decide_rag → run_rag → decide_user_context → build_user_context → assemble_prompt_layers 순서. 각 노드는 상태를 받고 결과를 반환하는 단순 함수.
🧠 왜? RAG·UserContext·프롬프트 조립이 가장 복잡한 구간이므로 여기부터 그래프로 시각화
adapter.py 로 legacy / graph 경로 분기 구현
feature flag 기반으로 두 경로를 선택. 기존 ChatService는 adapter만 호출하면 되도록 접점 최소화.
🧠 왜? graph 코드 도입으로 기존 경로가 깨지지 않게 격리
off / shadow / partial 3단계 롤아웃 모드
off = 미사용, shadow = 기존 경로로 답변하되 그래프 함께 돌려 결과 비교, partial = 일부 사용자만 그래프 적용(1%→5%→...→100%).
🧠 왜? 의료 서비스는 한 번에 바꾸면 안 되므로 점진 전환 + 관찰
타임아웃·예외 시 legacy fallback 안전망
graph 경로가 250ms 이내 응답 못 하거나 예외 발생 시 즉시 기존 경로로 자동 복구.
🧠 왜? 사용자 체감 first-token latency(첫 글자까지의 시간)가 악화되지 않도록 방어
unit 테스트 8개 + 벤치마크 스크립트 추가
parity / mode_switch / logging / prompt_layers 4개 영역에서 LangGraph 관련 테스트 추가. 모두 PASS.
🧠 왜? 레거시 결과와 그래프 결과가 100% 일치하는지 자동 검증

3 채팅 흐름 상세 순서도 (Before / After)

가장 중요한 핵심 변화. 사용자가 메시지를 보내는 순간부터 AI 응답이 돌아올 때까지의 흐름을, 각 단계 설명과 조건 분기까지 포함해 그렸습니다.

BEFORE 📉 기존 흐름 — prep 구간이 하나의 덩어리
기존 구조는 모든 처리가 service.py 안에서 순차적으로 일어납니다. 특히 ⑥번 준비 구간은 RAG 검색, 사용자 맥락 생성, 프롬프트 조립이 한 덩어리로 뒤섞여 있어서 "어디가 복잡한지", "어디만 바꾸면 되는지"를 말로 설명하기 어려웠습니다.
1
요청 검증 _validate_request
user_id 존재 여부, OpenAI 키 설정, 오늘 대화 횟수(기본 20건 제한) 확인. 실패 시 즉시 error 이벤트 후 종료.
2
접근·동의 확인 _validate_chat_access
HealthProfile 조회 + AI 채팅 동의 여부 확인. 동의가 없으면 ai_consent_required 에러 반환.
3
Content Filter 검사 content_filter.check_message
메시지에 위기 표현·차단 표현·의료 단정 표현이 있는지 검사. 결과에 따라 3가지로 분기됩니다.
분기 A CRISIS (자해/위기) → 위기 전용 고정 응답(상담 번호 포함) + 24시간 쿨다운 설정 후 흐름 종료
분기 B BLOCK (부적절 표현) → content_blocked 에러 이벤트 전송 후 흐름 종료
분기 C ALLOW (통과) → 아래 4번 단계로 계속 진행
4
세션 준비 + 사용자 메시지 저장 _prepare_session
대화 세션(ChatSession)을 찾거나 새로 만들고, 사용자가 보낸 메시지를 DB에 저장합니다.
5
건강 질문 대상 계산 + 최근 대화 불러오기
오늘 사용자에게 물어볼 건강 질문 묶음(eligible_bundles)을 계산하고, 최근 대화 10건을 불러옵니다.
6
⚠️ 프롬프트 준비 — 단일 덩어리 _build_openai_messages
RAG 검색 여부 판단 → 자료 검색 → 사용자 맥락 판단 → 맥락 생성 → 프롬프트 레이어 조립이 한 함수 안에서 뒤섞여 실행됨. 어디가 복잡한지 말로 설명하기 어려운 구간.
7
OpenAI 스트리밍 호출 _stream_openai
gpt-4o-mini 모델에 메시지를 전송하고, 응답을 한 토큰씩 받아오면서 클라이언트에 SSE로 흘려보냅니다.
8
응답 저장 + done 이벤트 _save_response
GPT가 만든 전체 응답을 ChatMessage로 저장하고, 건강 질문 데이터와 함께 done 이벤트를 마지막으로 전송.
AFTER 📈 부분 적용 후 흐름 — prep만 그래프로 분리
1~5번과 7~8번은 그대로 유지합니다(의료 안전이 걸린 경로니까). 오직 6번 prep 구간만 LangGraph로 분리해서, 어떤 판단이 어떤 순서로 일어나는지를 5개 노드로 명확히 나눴습니다. 실제 응답 경로는 adapter가 feature flag를 보고 legacy/graph 중 하나를 고릅니다.
1
요청 검증 변경 없음
기존과 동일. 그래프 밖에 남김.
2
접근·동의 확인 변경 없음
기존과 동일. 그래프 밖에 남김.
3
Content Filter 검사 enum 기반으로 강화
reason_codes를 enum으로 교체, normalize 함수 option C 반영. 분기 로직은 그대로지만 안정성이 개선됨.
분기 A CRISIS → 위기 전용 고정 응답 (그래프 밖에서 처리 · 의료 안전)
분기 B BLOCK → 차단 에러 이벤트 (그래프 밖에서 처리)
분기 C ALLOW → 4번 단계 계속 진행
4
세션 준비 + 사용자 메시지 저장 변경 없음
DB 쓰기 경로는 운영 계약이 걸려 있어 그래프 밖에 유지.
5
건강 질문 대상 + 최근 대화 불러오기 변경 없음
기존과 동일.
6
🧩 프롬프트 준비 — LangGraph로 분리 chat_graph.adapter
adapter가 feature flag(off/shadow/partial)를 확인해 legacy 경로 또는 graph 경로를 선택. 실패하면 즉시 legacy로 자동 복구. 아래 ZOOM IN 박스에서 그래프 내부를 확인하세요.
7
OpenAI 스트리밍 호출 변경 없음
SSE 스트리밍 계약 유지. 그래프 밖에 남김.
8
응답 저장 + done 이벤트 변경 없음
기존과 동일.
🧩 ChatPreparationGraph 내부 — 6번 단계 확대
5개 노드가 START → END 까지 순차 실행됩니다
1
decide_rag
이 질문에 RAG(참고 자료 검색)가 필요한지 판단. 라이프스타일 질문이나 사실 확인 질문이면 true.
입력: 메시지, 라우팅 정보 / 출력: should_run_rag (bool), topic_hint
2
run_rag
should_run_rag=true 인 경우에만 실제로 지식 베이스에서 참고 자료를 검색. false면 skip.
입력: 메시지, topic_hint / 출력: rag_context_text, hit_count
3
decide_user_context
사용자 프로필(목표, 운동 빈도, 수면 등)을 프롬프트에 넣을지 판단. 감정 우선 메시지면 skip.
입력: 라우팅, 감정 우선 플래그 / 출력: should_build_user_context (bool)
4
build_user_context
should_build=true 인 경우에만 사용자 프로필을 요약 문장으로 변환. 예: "체중관리 목표, 수면 6-7시간 유지".
입력: HealthProfile, topic_hint / 출력: user_context_text
5
assemble_prompt_layers
base 시스템 프롬프트 + RAG 레이어 + 사용자 맥락 레이어 + 필터 지시 레이어를 순서대로 쌓아서 최종 messages 배열을 완성.
입력: 위 모든 결과 / 출력: openai_messages (최종 프롬프트)

1, 2, 3, 4, 5, 7, 8번 (총 7단계)

4 왜 전체 전환이 아니라 부분 적용인가?

이 결정이 이번 작업의 포트폴리오 핵심 포인트입니다.

핵심 원칙: "안전이 중요한 곳은 그대로, 복잡성이 몰린 곳만 그래프로." 의료 서비스 특성상 회귀 위험을 0으로 관리해야 하는 경로가 있고, 그 경로를 그래프 안으로 넣으면 설명력은 늘어나지만 운영 리스크가 커집니다.
🛡️ 그래프 밖 (legacy 유지)
  • content_filter (위험 표현 차단)
  • crisis / block / consent early return
  • DB write (세션·메시지 저장)
  • SSE 스트리밍 (응답 전달)
  • cooldown (위기 쿨다운)
의료 안전 — 회귀 위험 0으로 관리, 운영 계약 안정성
🧩 그래프 안 (새로 분리)
  • decide_rag (RAG 사용 여부 판단)
  • run_rag (참고 자료 검색)
  • decide_user_context (사용자 맥락 빌드 여부)
  • build_user_context (사용자 프로필 요약)
  • assemble_prompt_layers (프롬프트 레이어 조립)
실제 복잡성이 몰린 곳 — 그래프 시각화가 설명력을 높임

5 용어 해설 (코딩 초심자용)

회의록과 문서에 자주 나오는 기술 용어를 일상 비유로 풀어 설명합니다.

LangGraph
AI 흐름을 노드(단계)와 엣지(연결선)로 관리하는 도구. 상태가 한 노드에서 다음 노드로 넘어가며 처리됩니다.
요리 레시피를 단계별 카드로 정리하는 것. 각 카드가 노드, 카드 순서가 엣지.
ChatPreparationGraph
다나아에서 만든 LangGraph 기반 "준비 전용" 그래프. GPT에게 메시지 보내기 직전까지의 5개 단계만 담당.
주방장에게 주문 넘기기 전에 재료 준비 팀장이 5단계로 준비하는 것.
RAG Retrieval-Augmented Generation
질문에 맞는 참고 자료를 미리 찾아서 GPT 프롬프트에 끼워 넣는 기법. 지식의 정확성을 높입니다.
시험 볼 때 교과서를 오픈북으로 쓰는 것과 같습니다.
SSE Server-Sent Events
서버가 클라이언트에게 글자를 한 조각씩 실시간으로 흘려보내는 방식. ChatGPT의 "타이핑 효과"가 SSE입니다.
전화기로 상대가 말하는 걸 실시간으로 듣는 것. 편지(전체 응답 대기)가 아닙니다.
Shadow 모드
사용자는 기존 경로로 답변을 받지만, 백그라운드로 새 경로(그래프)도 함께 돌려서 두 결과를 조용히 비교합니다.
신입 직원에게 같은 업무를 시키고 선배 결과와 몰래 비교해 실력 검증하는 것.
Partial 모드
일부 사용자에게만 새 경로를 적용. 1% → 5% → 10% → 50% → 100% 순서로 점진 전환합니다.
새 메뉴를 전체 매장에 한 번에 도입하지 않고 1호점부터 천천히 확장하는 것.
Parity 동등성
두 경로의 결과가 완전히 똑같은지 비교하는 것. 프롬프트 해시·메시지 개수·RAG 히트 수 등 30여 가지 항목을 검증합니다.
사본과 원본이 모든 페이지에서 동일한지 한 장씩 대조하는 것.
Fallback 자동 복구
새 경로(그래프)가 느리거나 실패하면 즉시 기존 경로(legacy)로 자동 복구. 사용자는 전혀 눈치채지 못합니다.
엘리베이터 고장 시 자동으로 계단 안내로 전환되는 것.
