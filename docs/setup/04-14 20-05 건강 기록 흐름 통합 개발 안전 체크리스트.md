- 날짜/시간: `missing_summary`는 오늘 날짜 응답에만 포함, 상세 리포트는 최근 7일 기준

## 2) 데이터 안전 (중요)
- [x] 이번 변경의 SoT(Source of Truth)를 명시했다.
- [x] 상태 전이 규칙을 정의했다.
- [x] 멱등성 규칙을 정의했다.
- [x] 동시성 / race condition 위험 구간을 확인했다.
- [x] UTC / 사용자 로컬 날짜 처리 기준을 정했다.
- [x] 실패 시 복구 경로를 정리했다.

근거:
- 오늘 건강 기록 SoT: `DailyHealthLog`
- 챌린지 상태 SoT: `ChallengeService.get_overview()`
- 채팅 세션 SoT: URL의 `session_id`, 없으면 서버 최신 세션
- 멱등성: `PATCH /health/daily/{date}`의 `First Answer Wins` 유지
- 동시성 위험: 오른쪽 패널 저장과 인라인 카드 저장이 같은 날 로그를 동시에 건드릴 수 있으므로 저장 후 항상 서버 응답으로 재동기화
- 날짜 기준: 오늘 로그만 `missing_summary` 포함, 과거 날짜는 `null`
- 실패 복구: 저장 실패 시 서버 재조회값으로 UI 동기화, 없는 `session_id`는 빈 채팅으로 fail closed

## 3) DB / API 변경 (중요)
- [x] DB 변경 시 마이그레이션 전략이 있다.
- [x] DB 변경 시 롤백 전략이 있다.
- [x] 기존 데이터 호환성 영향을 정리했다.
- [x] API 변경 시 하위 호환성 영향과 클라이언트 영향이 정리되어 있다.
- [x] 파괴적 작업은 명시적 승인 범위 안에 있다.

근거:
- DB 스키마 변경 없음 → migration / rollback 불필요
- API는 additive change만 적용
  - `GET/PATCH /api/v1/health/daily/{log_date}` 응답에 `missing_summary`만 추가
- 기존 클라이언트는 새 필드를 무시해도 동작 가능
- 파괴적 작업 없음

- [x] 외부 API 한도와 실패 조건을 반영했다.
- [x] 동기 호출과 비동기 이벤트의 책임 경계를 정했다.
- [x] 재시도 / 중복 방지 / 로깅 전략을 정리했다.

- 로컬/운영 차이: 로컬 Windows 환경에서는 `postgres` 호스트명 대신 `127.0.0.1`이 필요할 수 있음
