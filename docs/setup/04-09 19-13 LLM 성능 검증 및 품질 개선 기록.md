# LLM 성능 검증 및 품질 개선 기록

## 문서 목적
- 이 문서는 다나아 프로젝트에서 진행한 `LLM 응답속도 개선`, `실험`, `재측정`, `반증`, `수정`, `검증` 과정을 한곳에 모아 두는 기준 기록이다.
- 참여기업 주제 안내의 아래 항목에 대응하는 근거 문서로 사용한다.
  - `지속적인 모델 성능 검증 및 결과 분석을 통해 품질 개선을 위해 노력을 하였는가?`
- 좋은 결과만 남기지 않고, `잘못된 가설`, `방법론 오류`, `재검증으로 뒤집힌 결론`까지 함께 기록한다.

## 운영 원칙
- 이 문서는 `속도/품질 검증 관련 기준 문서`로 사용한다.
- 이후 관련 작업이 생기면 새 문서를 따로 만들더라도, 핵심 결과와 결론은 이 문서에도 누적 갱신한다.
- 특히 아래 4가지는 항상 남긴다.
  - 무엇을 바꿨는지
  - 어떻게 측정했는지
  - 무엇이 반증되었는지
  - 다음 의사결정이 무엇인지

## 한눈에 보는 결론
- 이번 속도 개선 작업은 단순 구현이 아니라 `가설 수립 -> 구현 -> 실험 -> 반증 -> 재측정 -> 결론 수정`의 흐름으로 진행됐다.
- 초기에 `shared client`, `warmup`, `SSE receiver 개선`이 큰 개선으로 보이는 결과가 있었지만, 후속 재검증 결과 일부는 `측정 방법론 오류`로 판명되었다.
- 현재 기준 가장 신뢰할 수 있는 결론은 다음과 같다.
  - `warmup/shared client`는 `첫 cold 사용자`에게는 도움이 될 수 있다.
  - 하지만 `steady-state 전체 서비스 경로`에서 TTFT와 done 시간을 유의미하게 개선했다고 단정할 수는 없다.
  - 따라서 현재 작업은 실패가 아니라, `실험 결과를 반증하고 방향을 수정한 검증 기록`으로 보는 것이 맞다.

## 배경
- 프로젝트 목표:
  - 사용자가 채팅을 보냈을 때 `첫 글자까지의 시간(TTFT)`을 줄이고, 가능하면 `done까지의 총 완료시간`도 줄이는 것
- 절대 건드리지 않기로 한 계약:
  - `validation -> content_filter -> crisis/block/consent early return -> cooldown -> session/user save -> prep -> stream -> save -> done`
  - `ChatService`
  - SSE event type: `token / error / done`
  - `done` payload shape
  - branch precedence
  - DB write semantics
  - LangGraph 범위는 `prep-only`

## 현재 기준 주요 실측 결과
### A. 내부 prep 격리 bench
출처:
- `scripts/bench_chat_prep_langgraph.py`

결과:
- legacy prep median: `0.01ms`
- graph prep median: `5.05ms`
- delta: `+5.04ms`

해석:
- Python 레벨 prep 조립 자체는 병목이 아니다.
- LangGraph prep-only 구조의 내부 오버헤드는 존재하지만, 절대값은 매우 작다.

### B. 외부 OpenAI first-token 단독 비교
출처:
- 초기 transport 비교 기록

결과:
- 기존(per-request client) median: `0.60912초`
- 개선(shared client) median: `0.56447초`
- 차이: `-0.04465초`

해석:
- shared client는 외부 first-token 단독 비교에서는 소폭 개선 신호가 있었다.
- 다만 이 수치만으로 전체 서비스 체감 개선을 단정할 수는 없다.

### C. 서비스 전체 경로 1차 측정
출처:
- `ChatService.send_message_stream()` 기준 직접 측정

결과:
- 기존 TTFT median: `0.76487초`
- 개선 후 TTFT median: `0.75431초`
- 기존 done median: `1.30798초`
- 개선 후 done median: `1.51558초`

해석:
- TTFT는 아주 소폭 개선된 것처럼 보였지만,
- done은 오히려 느려졌다.
- 이 단계부터 `transport 미세 최적화만으로는 충분치 않다`는 신호가 보이기 시작했다.

### D. 서비스 전체 경로 재검증 최종값
출처:
- `docs/setup/04-09 17-00 LLM 속도 최적화 작업 정리 및 실측 재검증 전달용.md`
- `backend/tests/integration/test_bench_service_ttft.py`

결과:

| 항목 | baseline | candidate | 차이 |
|------|----------|-----------|------|
| TTFT median | 1165.82 ms | 1306.14 ms | +140 ms |
| done median | 2298.74 ms | 2756.84 ms | +458 ms |
| iter 1 TTFT max | 2963.32 ms | 1910.62 ms | -1053 ms |

해석:
- `steady-state 전체` 기준으로는 유의미한 개선을 증명하지 못했다.
- 하지만 `첫 cold 사용자`는 크게 개선됐다.
- 따라서 warmup의 효과는 `전체 평균 개선`이라기보다 `첫 사용자 tail latency 완화` 쪽에 가깝다.

## 무엇이 반증되었는가
### 반증 1. "77% 개선" 같은 강한 표현
- 초기 격리 bench 결과는 과대평가였다.
- 이유:
  - 매 iteration마다 warmup을 다시 호출
  - 실제 steady-state 서비스 경로가 아님

### 반증 2. "shared client + warmup이면 전체 서비스도 빨라진다"
- 현재 기준으로는 성립하지 않는다.
- 첫 사용자에는 효과가 있지만, 일반 warm 요청에는 차이가 작거나 불안정하다.

### 반증 3. "문제는 transport가 전부다"
- 실제로는 transport 외에도 아래 요인이 남아 있다.
  - 입력 토큰 수
  - 출력 길이
  - stream tail
  - route별 응답 패턴
  - hot path에서의 부가 로직

## 품질 개선 관점에서 이번 작업이 남긴 의미
- 이번 작업의 가치는 `속도를 무조건 개선했다`가 아니다.
- 더 중요한 가치는 아래 5가지다.
  - 성능을 느낌이 아니라 수치로 검증하려고 했다.
  - 초기 결과를 그대로 믿지 않고 다시 반증했다.
  - 잘못된 방법론을 문서로 남겼다.
  - 실제 서비스 경로 bench를 따로 만들었다.
  - 향후 최적화 방향을 더 정확하게 수정할 수 있게 됐다.

## 현재 판단
- `warmup/shared client`는 유지할 가치가 있다.
  - 이유: 첫 cold 사용자 경험 개선에는 분명한 신호가 있다.
- 하지만 이것을 `LLM 응답속도 개선의 핵심 성공 사례`로 단독 포장하면 안 된다.
- 현재 더 유망한 다음 축은 아래다.
  - 입력 토큰 수 감사
  - 출력 길이 제어
  - route별 응답 정책 분리
  - hot path에서 꼭 필요하지 않은 부가 계산 분리
