### 3.3 해석

#### 의미 있는 단일 사실
- **iter 1 (첫 요청) TTFT**: 2963ms → 1910ms (**−1053ms, −35.5%**)
- **iter 1 (첫 요청) done**: 4133ms → 3388ms (**−745ms, −18%**)
- 이는 **warmup이 첫 사용자 cold latency의 일부를 startup으로 흡수**했음을 뒷받침

#### 하지만 iter 2~10 (warm 상태)는 오히려 악화
- median +140ms, p90 +313ms
- 이건 warmup이 iter 2~10에는 **영향 없음** (shared client는 루프 밖에서 이미 warm)
- 악화 원인 후보:
  1. **OpenAI 서버 부하 변동** (baseline 09:41 → candidate 09:43, 2분 시간차)
  2. **샘플 n=10은 통계적으로 약함** (분산 ±600ms 범위 안)
  3. **Interleave 없이 blocks로 측정** (순차 baseline 10개 → candidate 10개)

#### 통계적 판정
- **candidate가 baseline보다 명확히 느려졌다고 할 수 없음** (noise 범위 안)
- **candidate가 baseline보다 명확히 빨라졌다고 할 수도 없음** (noise 범위 안)
- **단, iter 1(첫 요청)은 유의미하게 개선됨** (−1053ms는 개별 샘플 기준이지만 큼)

#### 사용자 관점 재해석
- **"첫 사용자 한 명"**은 warmup 덕에 약 1초 빨라짐 (iter 1 개선)
- **"두 번째 이후 사용자"**는 차이 없음 (shared client 이미 warm)
- 즉 warmup 효과는 "tail latency (가장 느린 케이스) 감소"에 국한됨

### 3.4 이전 격리 bench(`bench_ttft_ab.py`)가 왜 틀렸는가

격리 bench의 치명적 함정:
```
# 격리 bench (잘못된 방법)
for i in range(10):
    init_shared_openai_client()
    warmup_shared_openai_client()    ← 매 iteration 호출
    measure_first_chunk()
    close_shared_openai_client()
```

- 매 iteration이 **"fresh start + warmup"** 사이클
- warmup 효과가 **10회 모두에 반영**됨
- 결과: median 834 → 540 (−294ms) "개선"

실제 프로덕션 (서비스 경로):
```
# 실제 프로덕션 (test_bench_service_ttft.py)
init_shared_openai_client()
warmup_shared_openai_client()    ← 1번만 호출
for i in range(10):
    service.send_message_stream()   ← 10회 요청, 모두 같은 warm client
```

- warmup은 **iter 1에만** 영향
- iter 2~10은 이미 warm 상태
- 결과: median 1166 → 1306 (+140ms) noise 범위

**결론**: 격리 bench는 프로덕션 경로를 재현하지 못했고, "77% 단축" 주장은 방법론 오류였다.

### 4.2 `backend/services/chat/openai_client.py::warmup_shared_openai_client()` 현재 상태

```python
async def warmup_shared_openai_client() -> bool:
    try:
        await init_shared_openai_client()
        if _shared_openai_client is None:
            return False
        # 실제 OpenAI 네트워크 호출로 TCP/TLS/DNS/OpenAI 내부 라우팅 캐시 warming.
        # 1 토큰 dummy 요청으로 첫 사용자 cold latency를 startup 시점에 흡수.
        # 실패는 non-fatal: 앱 부팅은 계속되고 첫 사용자가 cold를 그대로 겪을 뿐.
        await asyncio.wait_for(
            _shared_openai_client.chat.completions.create(
                model=config.OPENAI_MODEL,
                messages=[{"role": "user", "content": "."}],
                max_tokens=1,
                stream=False,
                user="warmup",
            ),
            timeout=10.0,
        )
        return True
    except Exception:
        logger.warning("chat_openai_client_warmup_failed")
        return False
```

## 6. 이번 세션에서 배운 것 (정직한 교훈)

### 6.1 격리 bench는 프로덕션 경로 측정이 아니다
- `bench_ttft_ab.py`는 `_stream_openai()`만 격리 호출하고 매 iteration마다 warmup 사이클 → 프로덕션 재현 실패
- 실제 프로덕션은 "shared client 초기화 1번, 수많은 요청 재사용" 구조
- **교훈**: 성능 최적화 측정은 반드시 실제 서비스 경로(service.py 이상)에서 해야 한다.

### 6.2 추정치와 실측의 차이를 혼동하면 안 된다
- 이전 세션들에서 "100~300ms 단축 예상", "1.5초 누적 단축" 같은 수치를 플랜에 박았지만 전부 추정치였음
- 실제 실측은 격리 bench에서 과장, 서비스 경로에서 미미
- **교훈**: 플랜 문서에 숫자를 쓸 때는 "이건 추정"인지 "이건 실측"인지 명시. 추정치를 실측인 것처럼 취급하지 말 것.

### 6.3 샘플 10개는 OpenAI 서버 변동을 이기지 못한다
- 서비스 경로 bench 10회 결과의 분산이 매우 컴 (±600ms)
- Interleave 없이 순차 측정(baseline 10개 → candidate 10개)은 시간대 편향에 취약
- **교훈**: p50은 10회로 대략 신뢰 가능, p95는 최소 30~50회 + interleave 필요.

### 6.4 "첫 사용자 경험"과 "평균 사용자 경험"은 다르다
- warmup 수정은 iter 1(첫 사용자)에만 효과
- 이를 median/mean으로 판단하면 "효과 없음"
- **교훈**: 성능 지표는 목적에 따라 선택. "첫 인상"이 중요하면 p99/max, "평균 경험"이 중요하면 p50/mean.

### 6.5 사용자 지적이 없었다면 잘못된 결론으로 마무리될 뻔했다
- 중간 보고에서 "warmup 수정 77% 단축" 주장
- 사용자가 "도커 띄워서 실측한 거야?" 지적 후에야 방법론 오류 발견
- **교훈**: 개발자 검증 없이 격리 bench 결과만 가지고 프로덕션 영향을 판단하지 말 것. **"end-to-end 실측 먼저"**는 성능 최적화의 기본이다.

## 7. 현재 상태에서 할 수 있는 다음 단계 선택지

### 옵션 A: Interleave 재측정
- `test_bench_service_ttft.py`에 interleave 모드 추가

### 옵션 B: warmup 함수 원복
- 서비스 경로 실측에서 median/p90 개선 없음 확인
- startup +2.5초 비용만 남음
- **실측 증거 기반으로 원복 정당화 가능**
- 단, iter 1 (첫 사용자 cold) 개선 효과는 포기
- **예상 소요**: 5분 (Edit 1번 + unit test)

### 옵션 C: warmup 유지 + 다른 병목 탐색

- 진짜 병목은 다른 곳 (예: `chat_stream_tail_ms`이 항상 1~1.5초, `eligible_bundles` 조회 30~40ms 등)
- prep/stream 구간 breakdown을 더 깊게 분석
- **예상 소요**: 30~60분 조사 후 결정

### 옵션 D: 완전히 방향 전환 — 응답 길이 제한

- 단 응답 품질 영향 A/B 검증 필요
