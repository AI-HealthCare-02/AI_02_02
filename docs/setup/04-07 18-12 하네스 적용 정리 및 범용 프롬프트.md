## 3. 과거에 이 프로젝트에 적용됐던 하네스 기법

이 프로젝트에는 예전 `real-main` 시점에 문서 규칙과 실제 도구 차단 장치가 함께 있었다.

### 3-1. `CLAUDE.md`가 맡았던 역할

`real-main`의 `CLAUDE.md`는 아래 철학과 운영 규칙을 담고 있었다.

- 질문/논의 모드와 실행 모드 분리
- `"실행해줘"` 전에는 조사, 설명, 제안, 플랜만 수행
- 구현 후 자동 자기검증 5단계 수행
- `/evaluate`까지 이어지는 검증 체인
- 큰 파일은 `Write` 전체 교체 금지, `Edit` 부분 수정 우선
- `archive/`, `docs/planning/`, `docs/prototypes/` 수정 금지
- 플랜 모드에서는 병렬 에이전트 수를 정량적으로 요구
- 변경 시 Grep 기반 누락 검증 요구

즉 `CLAUDE.md`는 이 프로젝트에서 하네스의 `철학/운영 규칙 문서` 역할을 했다.

### 3-2. `.claude/settings.json`이 맡았던 역할

예전 `real-main`의 `.claude/settings.json`은 PreToolUse 훅을 연결하고 있었다.

- `Write` → `bash .claude/hooks/write-guard.sh`
- `Edit` → `bash .claude/hooks/edit-guard.sh`
- `Bash` → `bash .claude/hooks/commit-guard.sh`

즉 이 파일은 하네스의 `배선도` 역할이었다.

### 3-3. hook 3종이 맡았던 역할

#### `write-guard.sh`

- 50줄 이상 파일을 `Write`로 통째로 덮어쓰려 할 때 경고
- `"Edit로 부분 수정하는 것이 안전합니다"`라고 사용자 확인 유도

#### `edit-guard.sh`

- 아래 경로 수정 시 차단
  - `archive/`
  - `docs/planning/`
  - `docs/prototypes/`
- 즉 보호 경로 편집을 도구 수준에서 막음

#### `commit-guard.sh`

- Bash에서 `git commit` 실행 시 사용자 확인 요청
- 커밋 메시지 컨벤션 확인 유도

즉 hook 3종은 하네스의 `실제 자동 차단 장치`였다.

### 3-4. `.claude/rules/*`가 맡았던 역할

예전 규칙 파일은 하네스의 세부 정책을 문서화했다.

- `skill-usage.md`
  - 어떤 스킬을 허용/금지할지
  - UI 작업과 보안 검증 타이밍
- `doc-sync.md`
  - 설계 문서 동기화 순서
  - 원천 문서 → DB 명세 → API 명세 → 코드 전파
- `design-context.md`, `medical-domain.md`, `eval-criteria.md`
  - 프로젝트 성격, 의료 안전, 품질 기준 보강

즉 `.claude/rules/*`는 하네스의 `세부 운영 매뉴얼` 역할이었다.

## 5. 다른 폴더나 새 프로젝트에서 범용적으로 쓸 수 있는 핵심

아래는 이 프로젝트 전용이 아니라 다른 개발 폴더에서도 재사용 가능한 하네스 패턴이다.

### 5-1. 작업 모드 분리

- 질문/분석 모드와 실행 모드를 분리한다.
- 명시적 실행 지시 전에는 설명, 조사, 플랜만 수행한다.
- 이 규칙 하나만 있어도 무단 수정 사고가 크게 줄어든다.

### 5-2. 보호 경로 정책

- 수정하면 안 되는 폴더를 미리 정한다.
- 예:
  - `archive/`
  - `planning/`
  - `generated/`
  - `legal/`
- 문서에만 적지 말고, 가능하면 hook이나 guard로 차단한다.

### 5-3. 큰 파일 전체 교체 방지

- 일정 줄 수 이상 파일은 `Write` 전체 교체보다 `Edit` 부분 수정을 기본으로 둔다.
- 이 패턴은 대형 파일 손상, 누락, 포맷 붕괴를 줄이는 데 유용하다.

### 5-4. 커밋 전 사용자 확인

- `git commit` 같은 중요한 상태 변화는 사용자 확인을 받게 한다.
- 특히 메시지 규칙, 커밋 범위, 민감 파일 포함 여부 확인이 중요하다.

### 5-5. 설계 문서 동기화 순서

- 문서/코드가 섞여 있는 프로젝트는 아래 순서가 범용적으로 안전하다.
  1. 원천 요구사항
  2. DB 명세
  3. API 명세
  4. 실제 코드
  5. 프론트 전달 메모
- 이 순서를 고정하면 문서가 코드보다 앞서가는 사고를 줄일 수 있다.

### 5-6. 구현 후 자기검증

- `git diff`
- 영향 import 검색
- 패턴 일관성 확인
- 도메인/보안 규칙 점검
- 실행 검증

이 5개는 대부분 프로젝트에서 재사용 가능하다.

## 6. 새 프로젝트용 범용 하네스 프롬프트

아래 프롬프트는 폴더명만 바꿔서 다른 프로젝트에서도 쓸 수 있다.

```text
이 프로젝트에서는 아래 작업 하네스를 따른다.

1. 질문/논의 모드와 실행 모드를 분리한다.
- 사용자가 명시적으로 실행을 요청하기 전에는 설명, 조사, 플랜만 수행한다.
- 임의로 코드 수정, 파일 생성, 대규모 실행, 커밋을 하지 않는다.

2. 실행 전에는 먼저 현재 구조를 조사한다.
- 현재 폴더 구조
- 핵심 엔트리포인트
- 데이터 흐름
- API/DB 계약
- 보호해야 하는 디렉터리

3. 보호 경로는 수정하지 않는다.
- 보호 경로: <PROTECTED_DIRS>
- 이 경로는 참고만 하고 수정이 필요하면 먼저 사용자에게 알린다.

4. 큰 파일은 전체 교체보다 부분 수정(Edit)을 우선한다.
- 일정 규모 이상 파일은 전체 덮어쓰기를 피한다.
- 전체 교체가 필요하면 먼저 이유를 설명한다.

5. 구현 후 자기검증을 반드시 한다.
- git diff 검토
- 영향받는 import/호출부 검색
- 패턴 일관성 확인
- 도메인/보안 규칙 확인
- 실행 검증: <VALIDATION_COMMANDS>

6. 문서와 코드가 함께 바뀌는 프로젝트면 아래 순서로 동기화한다.
- 요구사항 원본
- DB 명세
- API 명세
- 실제 코드
- 프론트/소비자 전달 메모

7. 아래 항목이 바뀌면 먼저 사용자에게 알린다.
- API 경로
- 요청/응답 형식
- enum 값
- DB 스키마
- migration 상태

8. 답변 형식
- 먼저 짧은 요약
- 그 다음 근거
- 실행했다면 검증 결과와 잔여 리스크 포함
```

## 7. 이 저장소용 현재 구조 맞춤 프롬프트

아래 프롬프트는 이 저장소 `마지막 웹프로젝트`에 바로 맞춘 버전이다.

현재 작업폴더는 `C:\\Users\\mal03\\Desktop\\레퍼런스\\마지막 웹프로젝트`다.

이 저장소에서는 아래 기준을 따른다.

1. 현재 구조 기준

- 코드 수정, 파일 생성, 커밋, 푸시는 실행 요청 후에만 한다.

3. 보호 경로

- `docs/planning/`

- 위 경로는 먼저 사용자 확인 없이 수정하지 않는다.

4. 현재 하네스 상태 해석

5. 역사적 하네스 근거

- 그 시점의 하네스는 Write/Edit/Bash 도구 가드와 문서 동기화 규칙을 함께 사용했다.

6. 구현 후 자기검증
- `git diff`

- API/DTO/모델 영향 확인
- 문서 동기화 필요 여부 확인
- 가능한 테스트 또는 최소 실행 검증 수행

7. 먼저 알려야 하는 변경
- API 경로
- 요청/응답 형식
- enum 값
- DB 스키마
- migration 상태

8. 답변 시 항상 먼저 정리
- 현재 기준 브랜치
- 현재 구조 기준 경로
- 관련 공식 문서
- 로컬 상태와 tracked 상태가 같은지 다른지

## 8. 만약 이 저장소에서 하네스를 다시 정식 활성화하고 싶다면

지금 문서 기준으로는 참고만 가능하고 자동 하네스는 꺼져 있다.

정식 활성화하려면 최소 아래 결정을 먼저 해야 한다.

1. `CLAUDE.md`를 다시 Git 추적 대상으로 둘 것인지
- 현재 `.gitignore` 정책과 충돌

2. `.claude/`를 다시 프로젝트 내부에서 관리할 것인지
- 현재는 디렉터리만 남아 있고 실제 파일이 없음

3. 현재 구조 기준으로 규칙을 최신화할 것인지
- 예전 `doc-sync.md`에는 `app/*`, `apps/web/*` 기준이 남아 있음
- 현재 구조에 맞추면 `backend/*`, `frontend/*`, `workers/ai/*`로 다시 써야 함

즉 바로 필요한 건 “복원”보다 “정식 운영 정책 결정”이다.

## 9. 이 문서의 사용 목적

이 문서는 아래 용도로 쓴다.

- 현재 로컬 `CLAUDE.md`가 어떤 상태인지 설명할 때
- 과거 하네스가 실제로 어떻게 동작했는지 설명할 때
- 다른 폴더/새 프로젝트에 적용할 범용 작업 하네스를 만들 때
- 현재 저장소에서 다시 하네스를 정식 운영할지 판단할 때

이 문서는 참고 문서이며, 자동으로 Git tracked 상태를 바꾸거나 hook를 복원하지 않는다.
