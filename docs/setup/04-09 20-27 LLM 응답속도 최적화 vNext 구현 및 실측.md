# LLM 응답속도 최적화 vNext 구현 및 실측

## 1. 목적
- 모델은 `gpt-4o-mini`로 고정한다.
- 이번 라운드의 목표는 `실제 서비스 경로 기준 TTFT/wire/paint/done`을 줄일 수 있는 기반을 만드는 것이었다.
- 핵심 방향은 `transport 미세튜닝`보다 아래 4가지였다.
  - 측정 정합성 강화
  - prompt/output budget 실측 기반 최적화
  - prompt caching 친화 구조 정리
  - 프론트 wire/paint 경로 단축

## 6. 현재 해석
- 이번 라운드는 `steady-state TTFT`를 크게 개선했다고 말할 수 있는 단계는 아니다.
- 대신 아래는 확실히 확보했다.
  - 실제 서비스 경로에서 `prompt_tokens`, `TTFT`, `done`, `output length`를 함께 보는 측정 인프라
  - 프론트 `wire/paint`를 재기 위한 계측 지점
  - same-origin rewrite와 `next/font` 적용 기반
  - prompt budget을 줄일 수 있는 구조 정리
- 현재 warm-path prompt 크기는 중앙값 기준 약 `372 tokens` 수준이다.
  - 즉 이 라운드의 핵심 병목은 Python prep이 아니라 외부 네트워크 + 모델 생성 시간일 가능성이 높다.

## 7. 남은 리스크와 후속 우선순위

### 7-1. 아직 남은 리스크
- 브라우저 실측은 코드 계측만 넣었고, 실제 `wire_ttft_ms / paint_ttft_ms` 값은 별도 수동/자동 브라우저 실행으로 확인해야 한다.
- 이번 벤치 샘플에서는 `route=None`, `emotional_priority=false`가 많아 고난도 branch coverage가 낮다.
- `bench_ttft_ab.py`는 제품 steady-state 근거로 사용하면 안 된다.

### 7-2. 다음 우선순위
1. 브라우저 기준 `wire_ttft_ms`, `paint_ttft_ms` 실제 샘플 확보
2. output length와 `done` 상관관계 분석
3. `RAG/user_context`가 실제 붙는 케이스만 따로 측정
4. 샘플의 `30% 이상`이 `1024 tokens`를 넘는지 먼저 확인한 뒤 prompt caching track 여부 결정
5. `openai[aiohttp]`는 20+ 동시 스트림이나 transport contention 증거가 있을 때만 별도 실험

## 8. 제출/공유용 한 문장
- 이번 라운드에서는 `gpt-4o-mini` 고정 상태에서 속도 개선을 위해 무작정 transport를 더 만지는 대신, 실제 서비스 경로의 `prompt budget / TTFT / done / wire / paint`를 측정 가능한 구조로 바꾸고, prompt prefix 안정화와 프론트 체감속도 경로 단축 기반을 먼저 만들었다.
