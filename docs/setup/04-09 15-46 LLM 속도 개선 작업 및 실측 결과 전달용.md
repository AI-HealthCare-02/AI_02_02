# LLM 속도 개선 작업 및 실측 결과 전달용

## 문서 목적

이 문서는 **Claude Code 같은 다른 AI가 현재 LLM 속도 개선 작업의 맥락을 빠르게 이해하고**,  
이어지는 분석이나 후속 최적화를 할 수 있도록 정리한 전달용 문서다.

이번 문서의 초점은 아래 3가지다.

1. 이번 라운드에서 실제로 무엇을 바꿨는가
2. 어떤 테스트를 어떤 환경에서 실행했는가
3. `기존 대비 현재` 성능을 **초 단위로** 어떻게 해석해야 하는가

---

## 한 줄 요약

이번 작업은 **모델 품질이나 safety shell을 건드리지 않고**,  
`TTFT(first token 체감 속도)`를 줄이기 위해

- backend prep 경로 낭비 제거
- shared OpenAI client 도입
- LangGraph `off` fast path 보장
- frontend SSE receiver 정합성 개선

을 진행한 작업이다.

현재까지의 실측 결론은 다음과 같다.

- **도커 Postgres를 붙인 서비스 전체 경로 기준으로는 TTFT가 소폭 개선**
- **총 완료 시간(done)까지 빨라졌다고 단정할 정도의 결과는 아직 아님**
- **외부 OpenAI transport 단독 비교에서는 shared client가 유의미하게 빠름**

---

## 먼저 알아야 할 중요한 전제

이번 비교는 **과거 커밋 vs 현재 커밋** 비교가 아니다.

이번에 실제로 비교한 것은 아래 2가지 조건이다.

- `baseline`
  - 현재 코드 기준
  - `CHAT_OPENAI_SHARED_CLIENT_ENABLED=False`
  - `CHAT_LANGGRAPH_MODE=off`
- `improved`
  - 현재 코드 기준
  - `CHAT_OPENAI_SHARED_CLIENT_ENABLED=True`
  - `CHAT_LANGGRAPH_MODE=off`

즉, 이번 비교는

- “기존 구조 전체”와
- “최적화 적용 후 전체”

를 git 단위로 되돌려 비교한 게 아니라,

**현재 코드에서 shared client 유무를 바꿔 같은 조건으로 비교한 것**이다.

그래서 이 문서는 아래처럼 해석해야 한다.

- `shared client 도입 효과`는 직접 비교 가능
- `TTFT 최적화 라운드 전체가 완전히 성공했다`고 단정하는 문서는 아님

## 절대 안 건드린 부분

이번 라운드에서 의도적으로 유지한 것:

- `validation -> content_filter -> crisis/block/consent early return -> cooldown -> session/user save -> prep -> stream -> save -> done` shell
- `ChatService`
- SSE event type: `token / error / done`
- `done` payload shape
- branch precedence
- DB write semantics
- LangGraph 범위는 여전히 `prep-only`

즉, 이번 최적화는 **기능 재설계가 아니라 성능 낭비 제거**에 가깝다.

## 현재 결론

### 확실히 말할 수 있는 것

- unit, frontend build, integration은 현재 기준 통과했다.
- shared client는 transport 단독 측정에서 개선 효과가 있다.
- 전체 서비스 경로 TTFT도 소폭 개선됐다.

### 아직 확실히 말하면 안 되는 것

- “전체 응답 시간이 빨라졌다”
- “TTFT 최적화가 이미 충분히 끝났다”
- “shared client만으로 체감 개선이 크게 났다”

### 가장 정직한 표현

이번 라운드는

- **성능을 나쁘게 만들지 않으면서**
- **TTFT를 아주 조금 개선하는 방향은 확인했고**
- **총 완료 시간 최적화는 아직 추가 작업이 필요하다**

로 정리하는 게 맞다.
