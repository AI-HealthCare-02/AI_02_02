위험 신호

왜 지금 바로 한 폴더로 섞으면 위험할까요?

폴더를 하나로 만든다고 자동으로 잘 섞이는 게 아니에요. 특히 로그인, 데이터, 서버 책임이 다르면 먼저 선을 그어야 해요.

1. 로그인 방식이 달라요

다나아는 자체 JWT 중심이고, Just Do It은 Supabase Auth 중심이라 바로 합치면 사용자 계정이 꼬일 수 있어요.

2. 데이터 성격이 달라요

건강 데이터는 더 민감해요. 생산성 앱 데이터와 처음부터 같은 통로로 다루면 위험이 커져요.

3. 서버 역할이 달라요

다나아는 API 서버가 핵심이고, Just Do It은 Next 앱 안에서 많은 로직을 처리해요. 바로 한 구조로 맞추기 어려워요.

4. 나중 변경 비용이 커져요

지금 억지로 섞으면 나중에 다시 분리하거나 고칠 때 더 오래 걸릴 수 있어요.

추천 로드맵

가장 안전한 순서는 “따로 개발 → 하나의 입구 → 필요할 때 통합”이에요.

중학생 버전으로 말하면, 두 과목 공책을 먼저 따로 잘 쓰고, 나중에 같은 바인더에 넣는 순서예요.

0단계

각자 따로 잘 만들기

지금은 다나아와 Just Do It을 각각 자기 방식대로 개발해요.

  • 다나아는 건강 기능과 API를 계속 다듬기
  • Just Do It은 생산성 화면과 흐름을 완성하기
  • DB와 로그인은 아직 서로 건드리지 않기
1단계

하나의 포털 화면 만들기

사용자는 한 앱처럼 느끼지만, 안쪽은 아직 따로일 수 있어요.

  • 공통 홈에서 “건강 코칭 / 생산성 관리” 선택
  • 상단 메뉴, 공통 브랜딩, 진입 화면 통일
  • 이 단계에서는 화면 경험이 핵심
2단계

계정 체계 정리

같은 사람을 두 앱에서 어떻게 같은 사용자로 볼지 규칙을 만들어요.

  • 이메일만으로 같은 사람 판단하지 않기
  • 공통 사용자 ID 또는 계정 연결표 설계
  • 권한과 역할 이름도 함께 맞추기
3단계

코드 통합은 그 다음

이때도 “한 폴더에 몽땅”이 아니라, 역할을 나눈 큰 저장소 구조가 좋아요.

  • 공통 UI 패키지 만들기
  • 포털, 건강 웹, Just Do It 웹을 나눠 보관
  • 필요할 때만 하나의 모노레포로 묶기
미래 구조

나중에 합친다면 이런 식의 “큰 서랍장” 구조가 안전해요.

핵심은 “한 저장소”와 “한 앱 폴더”를 구분하는 거예요. 하나의 집 안에서도 방은 나눠야 하듯이, 저장소도 역할별 구역이 필요해요. 

workspace/
├─ apps/
│  ├─ portal/         # 다나아 + Just Do It 통합 입구
│  ├─ health-web/     # 다나아 웹 화면
│  └─ justdoit-web/   # Just Do It 웹 화면
├─ services/
│  ├─ danaa-api/      # 다나아 FastAPI 서버
│  └─ integration-api/# 계정/연결 어댑터
├─ packages/
│  ├─ ui/             # 공통 버튼, 카드, 화면 조각
│  ├─ auth-contract/  # 로그인/계정 규칙
│  └─ shared-types/   # 공통 타입
└─ docs/
핵심 메시지

한 저장소는 가능

하지만 그 안에서도 앱 / 서비스 / 공통 패키지를 나누는 게 맞아요. 방이 있는 집처럼 생각하면 쉬워요.

FAQ

정말 자주 헷갈리는 질문

짧게 보면 더 쉽게 정리돼요.

Q. 결국 나중에 하나의 폴더에 다 넣어야 하나요?
아니요. 하나의 저장소로 모을 수는 있어도, 그 안에서 역할별 폴더를 나누는 게 보통 더 좋습니다. 지금 당장은 아예 따로 두고 개발해도 됩니다.
Q. 지금부터 따로따로 만들어도 괜찮나요?
네. 오히려 지금은 그게 더 안전합니다. 다만 나중 통합을 위해 이름, 메뉴, 사용자 흐름은 미리 생각해두면 좋아요.
Q. 왜 DB는 바로 합치면 안 되나요?
건강 데이터는 더 민감하고, Just Do It은 Supabase 인증과 RLS 규칙에 많이 묶여 있어요. 둘을 너무 일찍 합치면 로그인과 권한이 꼬일 수 있어요.
Q. 그럼 지금 제일 먼저 해야 할 건 뭔가요?
제일 먼저는 공통 포털 화면의 모습과 공통 메뉴 흐름을 정하는 거예요. 코드 통합보다 먼저 사용자 경험을 맞추는 게 더 중요합니다.
한 줄 결론: 다나아와 Just Do It은 지금은 따로 잘 만들고, 나중에 하나의 바인더(포털)로 연결한 뒤, 정말 필요할 때만 저장소 통합을 검토하는 것이 가장 안전합니다.
