## 2) 데이터 안전 (중요)
- [x] 이번 변경의 SoT(Source of Truth)를 명시했다.
- [x] 상태 전이 규칙을 정의했다.
- [x] 멱등성 규칙을 정의했다.
- [x] 동시성 / race condition 위험 구간을 확인했다.
- [x] UTC / 사용자 로컬 날짜 처리 기준을 정했다.
- [x] 실패 시 복구 경로를 정리했다.

근거:
- SoT(진짜 기준 데이터):
  - 테마 설정의 최종 기준은 서버 `user_settings.theme_preference`
  - `localStorage`는 첫 화면 깜빡임을 줄이는 캐시만 담당
- 상태 전이:
  - `dark -> light`
  - `light -> dark`
  - UI는 즉시 반영하고, 저장 실패 시 직전 값으로 롤백
- 멱등성:
  - 같은 테마를 다시 저장해도 최종 상태는 동일
- 동시성 위험:
  - 설정 화면에서 토글을 연속 클릭하면 마지막 요청이 최종 상태가 되어야 함
  - `ThemeContext`는 취소 플래그와 낙관적 업데이트 후 실패 롤백 구조로 정리
- 날짜/시간:
  - 이번 변경은 날짜 계산 규칙을 직접 바꾸지 않음
  - 기존 채팅/리포트의 날짜 처리 계약 유지
- 복구 경로:
  - 테마 저장 실패 시 UI 즉시 원복
  - DB 마이그레이션 실패 시 컬럼 롤백 가능

## 3) DB / API 변경 (중요)
- [x] DB 변경 시 마이그레이션 전략이 있다.
- [x] DB 변경 시 롤백 전략이 있다.
- [x] 기존 데이터 호환성 영향을 정리했다.
- [x] API 변경 시 하위 호환성 영향과 클라이언트 영향이 정리되어 있다.
- [x] 파괴적 작업은 명시적 승인 범위 안에 있다.

근거:
- DB 변경:
  - `user_settings.theme_preference VARCHAR(10) NOT NULL DEFAULT 'dark'`
- 마이그레이션 전략:
  - 새 컬럼 추가 + 기본값 `dark`
  - 기존 사용자 데이터는 자동으로 `dark` 채워짐
- 롤백 전략:
  - 컬럼 제거 마이그레이션으로 되돌릴 수 있음
- 기존 데이터 호환성:
  - 기존 클라이언트는 `theme_preference`를 몰라도 동작 가능
  - 새 프론트는 값이 없으면 `dark`로 취급
- API 영향:
  - `/api/v1/settings` 응답/패치에 `theme_preference` 추가
  - 기존 필드(`chat_notification`, `challenge_reminder`, `weekly_report`)는 유지
- 파괴적 작업:
  - 기존 데이터 삭제나 스키마 파괴 없음

## 4) 외부 연동 / 운영 경계
- [x] 외부 API 한도와 실패 조건을 반영했다.
- [x] 동기 호출과 비동기 이벤트의 책임 경계를 정했다.
- [x] 재시도 / 중복 방지 / 로깅 전략을 정리했다.
- [x] 로컬 개발 환경과 운영 환경의 차이를 기록했다.

근거:
- 외부 연동:
  - 로그인/설정/채팅은 `localhost:3000` 프론트 + `localhost:8000` 백엔드 기준으로 확인
- 동기/비동기:
  - 테마 저장은 동기 PATCH
  - 채팅 스트리밍/백엔드 작업은 기존 구조 유지
- 재시도/중복 방지:
  - 테마 저장 실패는 사용자 입력 직전 상태로 복구
  - 중복 클릭은 마지막 상태 기준으로 정리
- 환경 차이:
  - 로컬 pytest는 기본값으로 `postgres` 호스트를 찾다가 실패할 수 있음
  - 이번 검증은 `DB_HOST=localhost`, `DB_PORT=5432`, `PGHOST=localhost`, `PGPORT=5432` 보정으로 통과
  - 도커 환경에서는 `postgres`, `redis` 서비스명이 유효
