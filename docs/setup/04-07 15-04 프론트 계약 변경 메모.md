# 프론트 계약 변경 메모

- 작성일: 2026-04-07
- 기준 브랜치: `kwanju`
- 기준 코드 경로: `backend/`, `frontend/`, `workers/ai/`
- 목적: 프론트가 반드시 알아야 하는 DB/API 변경만 따로 묶어서 전달

## 1. 이번 변경 요약

- 신규 API
  - `PATCH /api/v1/auth/consent`
  - `GET /api/v1/health/weekly`
  - `GET /api/v1/risk/history`
  - `GET /api/v1/settings`
  - `PATCH /api/v1/settings`
  - `POST /api/v1/settings/export`
- 응답 필드 추가
  - 온보딩 상태 응답: `gender`, `age_range`, `bmi`
  - 챌린지 overview item: `is_fixed`
  - 챌린지 overview stats: `max_active_count`, `remaining_active_slots`
- 행동 규칙 변경
  - 건강 질문 노출은 이제 `user_settings` 값을 반영함
- DB 변경
  - `user_settings.chat_notification` 컬럼 추가

## 2. 프론트가 바로 반영해야 하는 응답 변화

### 2-1. `GET /api/v1/onboarding/status`

기존 주요 응답:

```json
{
  "is_completed": true,
  "completed_at": "2026-04-02T10:05:00+09:00",
  "user_group": "B"
}
```

현재 응답:

```json
{
  "is_completed": true,
  "completed_at": "2026-04-02T10:05:00+09:00",
  "user_group": "B",
  "gender": "MALE",
  "age_range": "45_54",
  "bmi": 26.1
}
```

프론트 액션:

- 기존 화면은 추가 필드가 있어도 깨지지 않아야 함
- 온보딩 완료 카드나 리포트 요약에서 추가 필드를 사용하려면 null-safe 처리 필요

### 2-2. `GET /api/v1/challenges/overview`

현재 추가 필드:

- item: `is_fixed`
- stats: `max_active_count`, `remaining_active_slots`

프론트 액션:

- 기존 카드 렌더링은 additive change로 처리
- “고정 추천 챌린지” 표시는 `is_fixed`를 기준으로만 판단
- 활성 슬롯 UI가 있다면 `remaining_active_slots`를 우선 사용

## 3. 행동 규칙 변경

### 건강 질문 노출 조건

이제 건강 질문은 아래 설정값에 따라 노출 여부가 달라집니다.

- `chat_notification`
- `max_bundles_per_day`
- `preferred_times`

프론트 액션:

- “항상 질문이 뜬다”는 전제로 UI를 짜면 안 됨
- 질문이 안 뜨는 경우를 정상 상태로 처리해야 함
- 설정 화면이 있다면 값 저장 전에는 실제 노출이 바뀌지 않는다고 안내할 것

## 4. DB / 운영 영향

- DB migration 필요
  - 추가 컬럼: `user_settings.chat_notification`
- 머지 후 실행 명령:

```bash
docker compose exec fastapi uv run python -m aerich upgrade
```

프론트 액션:

- API 연결 전, 병준 팀장 또는 백엔드 머지 완료 브랜치에 migration 적용 여부 확인

## 5. 이번 범위에서 아직 미구현인 것

아래는 문서 초안에 언급되지만 현재 구현 응답에는 없는 항목입니다.

- `risk_badge`
- `overall_grade`
- `coach_summary`

프론트 액션:

- 이 값들을 현재 실데이터로 가정하지 말 것
- 필요하면 mock UI로만 두고, 백엔드 합의 후 후속 구현으로 분리할 것
