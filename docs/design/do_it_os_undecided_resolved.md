## 2. 다크 메모 색상 5색 다크 변형 (항목 4 상세)

### 2-1. 설계 원칙

1. **다나아 톤 유지**: 차분한 크림·스톤 계열. 네온·고채도·강한 채도 금지(`danaa_design_system.md §1-2`).
2. **대비 보장**: 본문 텍스트 `#ECECEC` (다크 `--color-text`) 기준 WCAG AA 4.5:1 이상.
3. **라이트 팔레트와 색상 hue 매칭**: 사용자가 "크림" 메모를 라이트에서 다크로 바꿔도 동일한 심리적 구분감 유지.
4. **surface 대비**: 다크 surface `#2A2A2A` 위에서 **살짝 구분되는 음영**. 너무 어두우면 bg와 동색화.

### 2-2. 확정 팔레트 (버전 B · 색상 변경안, 2026-04-21 사용자 확정)

| 역할 | 라이트 Hex | **다크 Hex** | 특성 | WCAG 대비 (vs `#ECECEC`) |
|---|---|---|---|---|
| 크림 (황토) | `#FFF5E6` | `#544327` | 웜 황토·진한 크림 | ~ 8.0 : 1 ✅ AAA |
| 스톤 (샌드) | `#F0EDE8` | `#4D4538` | 웜 베이지 샌드 | ~ 7.9 : 1 ✅ AAA |
| 민트 (세이지) | `#EBF4EF` | `#374F40` | 세이지 그린 | ~ 7.5 : 1 ✅ AAA |
| 라벤더 (더스티) | `#EDE8F4` | `#423C54` | 더스티 퍼플 | ~ 8.8 : 1 ✅ AAA |
| 블러시 (로즈) | `#F7E9E9` | `#544040` | 로즈 브라운 | ~ 8.0 : 1 ✅ AAA |

> 대비비는 메모 카드 내부 텍스트 `#ECECEC` 기준. 각 색의 hue 특성을 살려 5색 구분감 ↑.
> 사용자가 v1(명도만·칙칙)/A(명도+10%p·구분 약함) 대비 **B(색상 조정)를 선택** (`Do it OS/04-21 15-50 Do it OS 다크 팔레트 명도vs색상 비교.html` 참조).

### 2-3. CSS 변수 네이밍 (구현 시 권장)

```css
/* globals.css 내 .doit-* 섹션에 추가 */
html[data-theme="light"] {
  --doit-memo-cream:   #FFF5E6;
  --doit-memo-stone:   #F0EDE8;
  --doit-memo-mint:    #EBF4EF;
  --doit-memo-lavender:#EDE8F4;
  --doit-memo-blush:   #F7E9E9;
}
html[data-theme="dark"] {
  --doit-memo-cream:   #544327;  /* 황토 크림 */
  --doit-memo-stone:   #4D4538;  /* 샌드 스톤 */
  --doit-memo-mint:    #374F40;  /* 세이지 민트 */
  --doit-memo-lavender:#423C54;  /* 더스티 라벤더 */
  --doit-memo-blush:   #544040;  /* 로즈 블러시 */
}
```

- 메모 카드는 **반드시** 위 변수만 사용. 하드코딩 금지 (`danaa_design_system.md §9`).
- 구현 시 `ThoughtCanvas.js` 의 색상 팔레트 배열은 CSS 변수 이름만 참조 (`var(--doit-memo-cream)` 형태).

### 2-4. 보완 메모

- 다크 카드 보더는 `rgba(255,255,255,0.04)` 정도로 **아주 얕게**. `--color-border` (`#3A3A3A`) 를 그대로 쓰면 너무 진함.
- 메모 내부 서브 텍스트(시간·삭제 버튼)는 `--color-text-muted` (`#888888`). 다크 팔레트 위에서 가독성 OK.

## 3. i18n 정책 (항목 6 상세)

### 3-1. 확정: 한글 고정 (MVP)

- 모든 페이지·메뉴·placeholder·토스트·빈 상태 메시지를 **한글 하드코딩**.
- `next-intl`·`i18next` 같은 i18n 프레임워크 **도입 안 함**.
- 이유:
  1. MVP 범위 최소화(다나아 전체 i18n 체계가 아직 미구축).
  2. Do it OS 는 "다나아 위에 붙는 레이어"라 다나아 본체보다 먼저 i18n 을 도입하면 정책 불일치 발생.
  3. 사용자 타겟: 한국어 만성질환 사용자.

### 3-2. Phase 2 대비 가이드

- Do it OS 문자열은 **모두 한 상수 파일** (`frontend/components/doit/doit_strings.js`) 에 모아두기 **권장**(선택 사항).
- 나중에 i18n 도입 시 이 파일만 추출해서 변환 가능.
- 샘플 구조:
  ```js
  export const DOIT_STRINGS = {
    thinking_placeholder: "머릿속에 떠오른 것을 적어보세요. Enter 누르면 위로 쏟아져요",
    thinking_empty:       "아직 아무것도 없어요. 떠오르는 것부터 적어보세요",
    go_to_classify:       "정리하러 가기",
    mobile_coming_soon:   "모바일은 곧 만나볼 수 있어요",
    /* ... */
  };
  ```
- 이 상수 파일은 **의무 아님**. 페이지 안에 직접 한글 리터럴로 써도 OK. 구현 속도 우선.

## 6. 권장 후속 액션

1. 사용자 확인:
   - 항목 4 다크 팔레트 5색 시안을 그대로 채택할지, 또는 실제 다크 화면 렌더링 시 1~2색 조정 필요한지.
   - 항목 6 i18n 한글 고정 정책 동의 여부.
2. 동의하면 바로 옵션 3(MVP 구현) 진입 가능. 진입 시 다음 순서로 실행:
   - `main` 에서 `feat/do-it-os-mvp` 브랜치 생성
   - Stage B1 (Sidebar nested 확장) → B2 (대시보드) → B3 (ThoughtCanvas 핵심) → B4 (준비중 4페이지) → B5 (레이아웃 점검) → B6 (E2E)
   - 각 Stage 후 `npm run build` + 시각 확인
   - 완료 후 3팀 리뷰 + fresh-context evaluate
