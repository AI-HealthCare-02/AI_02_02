## 1. 이 문서는 무엇인가

이 문서는 DANAA 서비스에서 어떤 데이터를 어디에 저장하는지 쉽게 설명한 최종 DB 문서입니다.

쉽게 말하면 아래 3가지를 빠르게 이해하기 위한 문서입니다.

- 사용자가 처음 가입할 때 입력한 정보는 어디에 저장되는가
- 매일 답하는 건강 질문은 어떤 구조로 쌓이는가
- 리포트, 챌린지, 설정 데이터는 어떤 테이블로 나뉘는가

실제 ORM, migration, enum, JSON schema, 제약조건 상세는 구현 문서인 [danaa_db_spec_final_2026-04-02.md](/C:/PycharmProjects/DANAA_project/docs/danaa_db_spec_final_2026-04-02.md)를 기준으로 합니다.

## 2. 먼저 알아두면 좋은 말

- `table`: 같은 종류의 데이터를 모아두는 저장 공간
- `column`: 테이블 안의 한 항목
- `PK`: 한 행을 구분하는 고유 키
- `FK`: 다른 테이블과 연결하는 키
- `정본(source of truth)`: 실제 원본 데이터
- `cache`: 화면을 빠르게 보여주기 위해 저장한 요약값
- `null`: 아직 답하지 않았거나 해당 없음

| 테이블명 | 역할 | 핵심 설명 |

| `health_profiles` | 온보딩 프로필 | 가입 직후 받는 기본 건강 정보 |

| `periodic_measurements` | 측정값 기록 | 체중, 허리둘레, 혈압, HbA1c, 공복혈당 |
| `risk_assessments` | 위험도 계산 결과 | 주간/월간 리포트 계산 결과 |

| `challenge_templates` | 챌린지 마스터 | 챌린지 종류와 목표 정의 |

> 주요 컬럼: `id`, `email`, `hashed_password`, `name`, `gender`(`MALE`/`FEMALE`), `birthday`, `phone_number`, `is_active`, `is_admin`, `last_login`, `created_at`, `updated_at`

## 4. 테이블별 설명

## 4-1. `user_consents`

회원의 최신 동의 상태를 저장합니다.

주요 컬럼:

- `user_id`
- `terms_of_service`
- `privacy_policy`
- `health_data_consent`
- `disclaimer_consent`
- `marketing_consent`
- `consented_at`

핵심 규칙:

- MVP에서는 최신 상태만 저장합니다.
- 별도의 동의 이력 테이블은 두지 않습니다.
- 사용자 1명당 1행입니다.

## 4-2. `health_profiles`

온보딩 설문 결과를 저장합니다.

여기에 들어가는 정보:

- 사용자 그룹 `A/B/C`
- 성별, 연령대, 키, 몸무게
- 가족력, 동반 상태, 치료 방식
- 운동 습관, 식습관, 수면 습관
- 목표
- 초기 FINDRISC 점수와 초기 위험도

주요 컬럼:

- `user_group`
- `relation`
- `gender`
- `age_range`
- `height_cm`
- `weight_kg`
- `bmi`
- `family_history`
- `conditions`
- `has_hypertension`
- `has_high_glucose_history`
- `treatments`
- `hba1c_range`
- `fasting_glucose_range`
- `exercise_frequency`
- `diet_habits`
- `sleep_duration_bucket`
- `alcohol_frequency`
- `smoking_status`
- `goals`
- `ai_consent`
- `initial_findrisc_score`
- `initial_risk_level`

파생값:

- `bmi`는 키와 몸무게로 계산합니다.
- `has_hypertension`은 `conditions`에 `hypertension`이 있으면 `true`입니다. DB 필드로 저장됩니다.
- `has_high_glucose_history`도 같은 방식입니다. DB 필드로 저장됩니다.

중요:

- `relation`은 사용자가 직접 고른 원본 응답입니다.
- `user_group`은 `relation`을 바탕으로 서버가 판단하는 그룹값입니다.
- 즉, 원본은 `relation`, 분류 결과는 `user_group`이라고 이해하면 됩니다.

## 4-3. `daily_health_logs`

가장 중요한 테이블입니다.
사용자가 매일 답한 건강 질문이 날짜별로 저장됩니다.

예를 들면:

- 오늘 잠은 어땠는지
- 아침/점심/저녁을 먹었는지
- 운동했는지
- 물을 몇 컵 마셨는지
- 야식, 기분, 복약, 음주 여부

주요 컬럼:

- `log_date`
- `sleep_quality`
- `sleep_duration_bucket`
- `breakfast_status`
- `lunch_status`
- `dinner_status`
- `vegetable_intake_level`
- `meal_balance_level`
- `sweetdrink_level`
- `exercise_done`
- `exercise_type`
- `exercise_minutes`
- `walk_done`
- `water_cups`
- `nightsnack_level`
- `took_medication`
- `mood_level`
- `alcohol_today`
- `alcohol_amount_level`

source 컬럼:

같은 답변이라도 어디서 들어왔는지 추적해야 해서 source 컬럼을 둡니다 (총 14개).

- `sleep_quality_source`
- `breakfast_status_source`
- `lunch_status_source`
- `dinner_status_source`
- `vegetable_intake_level_source`
- `meal_balance_level_source`
- `sweetdrink_level_source`
- `exercise_done_source`
- `walk_done_source`
- `water_cups_source`
- `nightsnack_level_source`
- `took_medication_source`
- `mood_level_source`
- `alcohol_today_source`

허용값:

- `chat`
- `direct`
- `backfill`

중요 규칙:

- 사용자 1명당 하루 1행만 저장합니다.
- `unique(user_id, log_date)`를 가집니다.
- `exercise_done=false`면 `exercise_type`, `exercise_minutes`는 `null`이어야 합니다.
- `alcohol_today=false`면 `alcohol_amount_level`은 `null`이어야 합니다.
- 답하지 않은 항목은 억지로 `0`이나 `false`로 바꾸지 않고 `null`로 둡니다.

대표 enum:

- `sleep_quality`: `very_good`, `good`, `normal`, `bad`, `very_bad`
- `*_status`: `hearty`, `simple`, `skipped`
- `vegetable_intake_level`: `enough`, `little`, `none`
- `meal_balance_level`: `balanced`, `carb_heavy`, `protein_veg_heavy`
- `sweetdrink_level`: `none`, `one`, `two_plus`
- `nightsnack_level`: `none`, `light`, `heavy`
- `mood_level`: `very_good`, `good`, `normal`, `stressed`, `very_stressed`
- `alcohol_amount_level`: `light`, `moderate`, `heavy`

## 4-4. `periodic_measurements`

매일 질문이 아니라, 별도로 입력하는 수치형 건강 데이터 테이블입니다.

예:

- 체중
- 허리둘레
- 혈압
- HbA1c
- 공복혈당

주요 컬럼:

- `measurement_type`
- `measured_at`
- `numeric_value`
- `numeric_value_2`
- `unit`
- `source`

중요 규칙:

- `blood_pressure`는 숫자 2개가 필요하므로
  - 수축기 -> `numeric_value`
  - 이완기 -> `numeric_value_2`
- `source`는 `manual`, `import`, `medical_checkup` 중 하나입니다.

MVP 측정 타입:

- `weight`
- `waist`
- `blood_pressure`
- `hba1c`
- `fasting_glucose`

## 4-5. `risk_assessments`

위험도 계산 결과와 리포트용 핵심 점수를 저장합니다.

주요 컬럼:

- `period_type` — 리포트 기간 단위 (`weekly`, `monthly`, `quarterly`)
- `period_start` — 리포트 시작일
- `period_end` — 리포트 종료일
- `findrisc_score` — FINDRISC 총점
- `risk_level` — 위험도 단계 (`low`, `slight`, `moderate`, `high`, `very_high`)
- `sleep_score` — 수면 점수 (0-100)
- `diet_score` — 식단 점수 (0-100)
- `exercise_score` — 운동 점수 (0-100)
- `lifestyle_score` — 생활습관 종합 점수 (0-100)
- `score_age` — FINDRISC 나이 점수
- `score_bmi` — FINDRISC BMI 점수
- `score_waist` — FINDRISC 허리 점수
- `score_activity` — FINDRISC 운동 점수
- `score_vegetable` — FINDRISC 채소 점수
- `score_hypertension` — FINDRISC 고혈압 점수
- `score_glucose_history` — FINDRISC 고혈당 점수
- `score_family` — FINDRISC 가족력 점수
- `top_positive_factors` — 상위 긍정 요인 (JSON)
- `top_risk_factors` — 상위 위험 요인 (JSON)
- `assessed_at` — 평가 시점

DB에 저장하지 않는 것:

- `risk_badge`
- `overall_grade`
- `coach_summary`

이 값들은 화면 표현이나 API 계산 결과로 다루는 것이 더 안전합니다.

## 4-6. `user_engagements`

사용자의 응답 참여 상태를 운영용으로 관리합니다.
사용자 1명당 1행입니다.

주요 컬럼:

- `state` — 참여 상태 (`ACTIVE`, `MODERATE`, `LOW`, `DORMANT`, `HIBERNATING`)
- `seven_day_response_rate` — 최근 7일 응답률 (0.000~1.000)
- `consecutive_missed_days` — 연속 미응답 일수
- `state_since` — 현재 상태 시작 시점
- `total_responses` — 누적 응답 수
- `today_bundle_count` — 오늘 질문 묶음 표시 횟수
- `cooldown_until` — 다음 질문 가능 시점 (90분 쿨다운)
- `last_bundle_key` — 마지막으로 표시한 묶음 키
- `last_response_at` — 마지막 응답 시점

이 테이블은 사용자에게 직접 크게 보이는 데이터라기보다, 질문 운영 로직에 중요합니다.

## 7. 변경 이력 (V1 → V2)

V2는 FINDRISC 점수 계산 로직 구현 과정에서 코드와 정확히 일치하도록 수정한 내용을 반영합니다.

### 7-1. `health_profiles` — 파생 필드 2개 추가

| 필드 | 설명 | V1 상태 |
|------|------|---------|
| `has_hypertension` | `conditions`에 `hypertension` 포함 시 `true` | V1에 없음 |
| `has_high_glucose_history` | `conditions`에 `high_glucose` 포함 시 `true` | V1에 없음 |

FINDRISC 8변수 중 `score_hypertension`, `score_glucose_history` 계산에 필수.

### 7-2. `risk_assessments` — FINDRISC 개별 점수 8개 필드 추가

V1에서는 "여기에 저장되는 것"으로 설명만 있었으나, V2에서 구체적 컬럼명 나열:

| 필드 | 점수 범위 | FINDRISC 변수 |
|------|-----------|---------------|
| `score_age` | 0-4 | 나이 |
| `score_bmi` | 0-3 | 체질량지수 |
| `score_waist` | 0-4 | 허리둘레 |
| `score_activity` | 0-2 | 운동 |
| `score_vegetable` | 0-1 | 채소 |
| `score_hypertension` | 0-2 | 고혈압 |
| `score_glucose_history` | 0-5 | 고혈당 이력 |
| `score_family` | 0-5 | 가족력 |

### 7-3. `user_engagements` — 구체적 필드명 확정

V1에서는 "예시" 수준이었으나, V2에서 코드 기준 확정:

| 필드 | 타입 | 설명 |
|------|------|------|
| `state` | enum | `ACTIVE` / `MODERATE` / `LOW` / `DORMANT` / `HIBERNATING` |
| `seven_day_response_rate` | decimal(4,3) | 0.000~1.000 |
| `consecutive_missed_days` | smallint | 연속 미응답 일수 |
| `state_since` | datetime | 현재 상태 시작 시점 |
| `total_responses` | int | 누적 응답 수 |
| `today_bundle_count` | smallint | 오늘 묶음 표시 횟수 |
| `cooldown_until` | datetime | 90분 쿨다운 시점 |
| `last_bundle_key` | varchar(50) | 마지막 표시 묶음 키 |
| `last_response_at` | datetime | 마지막 응답 시점 |

### 7-4. `challenge_templates` — `is_active` 필드 명시

V1에서 누락되어 있던 `is_active` (soft delete용) 필드를 V2에서 명시.

### 7-5. 온보딩 enum 값 변경

DB에 저장되는 enum 값이 FINDRISC 로직에 맞게 변경:

| 필드 | V1 값 | V2 값 (코드 기준) |
|------|-------|-------------------|
| `family_history` | `parent_or_sibling` | `parents` / `siblings` / `both` / `none` / `unknown` |
