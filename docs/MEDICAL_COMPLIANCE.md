# 의료 데이터 컴플라이언스 가이드

> 다나아 서비스가 건강 데이터를 안전하게 다루기 위한 규칙을 정리한 문서입니다.
> 코딩을 처음 접하는 팀원도 이해할 수 있도록 쉽게 작성했습니다.

---

## 1. 데이터 보관 기간 (Data Retention)

> "수집한 건강 데이터를 얼마나 오래 보관하고, 언제 삭제하는가?"

- **활성 사용자**: 마지막 로그인으로부터 최대 **1년** 동안 건강 기록 보관
- **휴면 사용자** (HIBERNATING 상태): 180일 경과 후 건강 기록 **익명화 처리** (user_id 제거)
- **회원 탈퇴 시**: 요청일로부터 **5영업일 이내** 모든 개인 데이터 완전 삭제
- 삭제 로그는 별도 감사 테이블에 **3년** 보관 (삭제한 사실만 기록, 데이터 내용 없음)

---

## 2. 암호화 기준 (Encryption)

> "건강 데이터가 해커에게 털려도 읽을 수 없도록 암호화하는 방법"

- **전송 중 (In Transit)**: 모든 API 통신은 **TLS 1.2 이상** 사용 (nginx 설정으로 강제)
- **저장 시 (At Rest)**: PostgreSQL 디스크 레벨 암호화 적용 권장. 특히 혈압(`numeric_value`), 혈당(`fasting_glucose_range`), 당화혈색소(`hba1c_range`) 같은 의료 수치 필드가 대상
- **키 관리**: 암호화 키는 환경변수(`.env`)로만 관리. 코드에 절대 하드코딩 금지

---

## 3. 접근 제어 (Access Control)

> "누가 어떤 데이터에 접근할 수 있는가?"

- **인증(Authentication)**: JWT 토큰 기반. `app/dependencies/security.py`의 `get_request_user()`가 모든 보호 엔드포인트에 적용
- **인가(Authorization)**: 사용자는 **자신의 데이터만** 조회/수정 가능. 모든 쿼리에 `user_id=user.id` 조건 필수
- **관리자 접근**: 별도 어드민 API는 현재 미구현. 구현 시 역할(Role) 기반 접근 제어 필요
- DB 직접 접근은 프로덕션 환경에서 **VPN + SSH 터널** 경유만 허용

---

## 4. 사용자 동의 감사 (Consent Audit)

> "사용자가 언제 어떤 동의를 했는지 추적"

- **온보딩 동의 기록**: `POST /api/v1/onboarding/consent` 호출 시 UserConsent 테이블에 타임스탬프와 함께 저장
- **AI 데이터 활용 동의**: `HealthProfile.ai_consent` 필드 (agreed/declined). 거부 시 AI 채팅에서 건강질문 삽입 안 함
- **동의 변경 이력**: 동의를 철회하거나 재동의할 때마다 별도 행(row)으로 기록하여 변경 이력 보존

---

## 5. 삭제 요청권 -- 개인정보 보호법 제36조

> "사용자가 '내 데이터 다 지워줘'라고 하면 어떻게 처리하는가?"

- **삭제 요청 접수**: 설정 페이지의 "회원 탈퇴" 또는 고객센터 이메일로 접수
- **삭제 범위**: users, health_profiles, daily_health_logs, periodic_measurements, chat_sessions, chat_messages, user_engagements, user_challenges -- 해당 user_id의 모든 행
- **삭제 기한**: 접수일로부터 **5영업일 이내** 완료
- Redis 캐시에 남은 세션/쿨다운 키도 함께 삭제

---

## 6. 침해 사고 대응 (Breach Notification)

> "만약 건강 데이터가 유출되면 어떻게 대응하는가?"

- **탐지**: 에러 모니터링 + 비정상 DB 접근 로그 감시
- **보고 기한**: 인지 후 **72시간 이내** 한국인터넷진흥원(KISA)에 신고 + 이용자 통지
- **통지 내용**: 유출된 정보 항목, 유출 시점, 대응 조치, 피해 최소화 방법, 담당자 연락처
- 사고 대응 후 **재발 방지 보고서** 작성

---

## 7. 제3자 데이터 공유 -- OpenAI

> "AI 채팅 기능에서 사용자 데이터가 OpenAI로 전송되는 부분의 규칙"

- **전송 범위**: `app/services/chat.py`에서 OpenAI API로 전송되는 것은 대화 메시지와 시스템 프롬프트(user_group 정보 포함)
- **전송되지 않는 것**: 혈압/혈당 수치, 복약 기록, 음주량 등 건강 원시 데이터는 직접 전송하지 않음
- **OpenAI 정책**: API 요청 데이터는 모델 학습에 사용되지 않음 (OpenAI API Data Usage Policy)
- **사용자 고지**: 이용약관에 "AI 응답 생성을 위해 대화 내용이 OpenAI 서버로 전송될 수 있습니다" 명시 필요
- `ai_consent = "declined"` 사용자는 AI 채팅을 비활성화하여 OpenAI 전송 차단

---

## 8. 의료 표현 금지 규칙

> "다나아는 의료 서비스가 아닙니다"

### 절대 사용 금지

- "~해야 합니다" (진단/처방 느낌)
- "치료", "처방", "진단" 같은 의료 용어

### 대신 사용

- "~을 권장합니다", "전문가 상담을 추천합니다"
- "참고 정보", "건강 관리 도우미"

### AI 응답 면책조항 (항상 포함)

- "이 정보는 교육 목적으로 제공되며, 전문 의료 상담을 대체하지 않습니다"
- "건강 관련 결정은 반드시 의료 전문가와 상담해주세요"

---

## 9. 로그에 민감정보 제외

- 로그 파일에 혈압, 혈당, 복약 기록 등 건강 수치를 **절대 기록하지 않음**
- 에러 로그에 사용자 건강 데이터가 포함되지 않도록 필터링 필수
- user_id까지만 기록, 건강 데이터 값은 "[Filtered]" 처리
